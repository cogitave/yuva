## 1. Why this milestone, and why these choices

M31 put the first real model traffic on the M30 channel — but the model is Anthropic's. M32 is plan B1: the first model the machine RUNS itself. Three places a naive "vendor llama.cpp and call it" milestone would rot, each fixed by construction:

1. **The debt must be machine-checked, not prose-remembered.** The plan (sovereignty-plan §M32) requires the witness to carry the four debt tokens verbatim, and principle 1 makes vendored llama.cpp TEMPORARY debt with a grep-enforced closure plan. §7 makes the token↔dependency coupling *bidirectional and fail-closed* — dropping the token while the C remains, claiming closure while the C remains, and carrying the token after the C is gone are all CI failures. The B3 closure test the plan requires to pre-exist DOES: `b3-pure-rust-engine.md` §5 (five gates) + §6 (the grep-enforced flip); §12 freezes its constants into this DoD.
2. **The C must never share an address space with anything that can mint evidence or spend money.** The GGUF parser is the plan-named untrusted-input C attack surface, and it has the CVE record to prove it: five heap overflows in `gguf_init_from_file`/`gguf_fread_str` (CVE-2024-25664..25668, Databricks), the vocab/token_to_piece overflow CVE-2025-49847 (8.8), the `gguf_init_from_file_impl` integer overflow CVE-2025-53630, the tensor-parsing overflow GHSA-96jg-mvhq-q7q7 — and on 2026-05-15 six MORE parser flaws (V-01..V-06, oss-security) with **no CVEs assigned at all**, so scanner-driven patching is structurally blind to this dependency. The M30 host-custodied K, the M31 MAC path, and (on the real-infer runner class) the stage-C `ANTHROPIC_API_KEY` therefore live ONLY in the safe-Rust daemon; the worker that holds llama.cpp holds **no key, no env, no network** (§2/§5).
3. **The determinism claim must be earned per run inside a named envelope, never assumed from temp=0 — and the per-run evidence must be stub-resistant.** Greedy decoding does NOT rescue reproducibility when logits aren't bit-identical (arXiv:2506.09501: BF16 greedy diverges across batch/hardware; arXiv:2408.04667: temp-0 endpoints measurably nondeterministic), the mechanical cause is FP non-associativity under varying reduction order (arXiv:2408.05148), and llama.cpp upstream explicitly REFUSES to guarantee batch-size invariance while stating the envelope that IS bit-exact: *"disable prompt caching and use only a single slot"* (PR #16016, JohannesGaessler). §4 adopts that envelope verbatim, adds the cross-runner pins upstream doesn't need (GitHub's fleet mixes AVX-512 and non-AVX-512 machines — actions/runner-images#3389), and earns `repro=BIT-IDENTICAL-2RUN` per run by a fresh-process 2-run compare, a **frozen-at-pin-time second-prompt negative control**, a **per-run nonce-freshness leg**, and **repo-pinned golden digests adjudicated at the workflow level** — the M36 reproducibility-ladder pattern, applied early, hardened against the stub-daemon forgery the review named (§17.1).

**And one thing M32 deliberately is:** *fit for unattended runs* — zero network at run time (the only network is a hash-pinned workflow SETUP step, the toolchain-install class, §5.5), zero secrets (the inverse of M31 stage C's unfitness note). The whole lane is autonomously buildable on a stock GPU-less `ubuntu-24.04` runner.

## 2. The shape — two processes, the keyless engine worker, the dep-clean shared core

**`tools/infer-daemon/` — an independent nested workspace** (the exact tb-vmm + xport-harness pattern: an empty `[workspace]` table auto-excludes it from the root `no_std` workspace, own `Cargo.lock`, kbuild/`-Zbuild-std` untouched). Two binaries:

- **The daemon (safe Rust, key-holding, witness-emitting).** Path-deps `tb-encode` (one codec, never a re-implementation — the M30 §3d rule) and a NEW dep-clean crate **`tools/xport-core/`** — the `FrameAccum` serve loop + `verify_infer_req`/MAC plumbing extracted from `tools/xport-harness/src/main.rs:203-327`, consumed by BOTH the xport-harness bin and the daemon. **Why a new crate and not a `[lib]` target on the harness package (the review's §17.3 must-fix):** since stage C (PR #40) the xport-harness PACKAGE manifest carries unconditional `ureq` + `serde_json`, and Cargo records package-level deps — including optional and feature-gated ones — in every consumer's lockfile; ANY path-dep on that package would put `ureq`/`rustls` into `tools/infer-daemon/Cargo.lock` and trip DEBT-LOCKSTEP Rule 4 on day one. Feature-gating does not fix a lockfile grep; a dep-clean crate does. `xport-core` carries `tb-encode` only — zero network deps by construction, asserted by its own manifest deny-grep. On the M32 lane the daemon IS the chardev peer: it serves `ECHO_REQ` and the M31 mock `INFER_REQ` exchange byte-identically (those lanes stay green) and adds the third backend. **Peer-identity discipline (stated, per §17.15):** the daemon stamps `peer_id=0x02` on the M30 echo and M31 mock legs — the landed `transport=QEMU-CHARDEV-HARNESS` cross-pin and the byte-identical-mock requirement depend on it — and `0x03` ONLY on M32 frames; and because the M30 §5.6 cross-process guard and the M31 positive-require grep are PREFIX-load-bearing, the daemon also emits the legacy `xport-harness:`/`xport-harness-infer:` witness lines byte-identically on those legs, alongside its own `infer-daemon:` lines. One process wearing two MAC-bound identities is workable precisely because it is written down — and it underlines that `engine=` is daemon-asserted identity, never engine evidence; the engine evidence is §4. Activation is **double-gated** like `--anthropic`/`XPORT_ANTHROPIC=1` (`live.rs:12-16`): `--local-llama` flag AND `XPORT_LOCAL_LLAMA=1` env, with model-dir/hashes from pinned env; opted-in with artifacts missing/mismatched = **LOUD exit-3 refusal** (the `NO_KEY_REFUSAL` precedent), never a mock fallback wearing the local token.
- **The engine worker (the ONLY process containing C).** Links vendored llama.cpp via a `-sys` crate (plan §M32 + LANGUAGE-AND-STANDARDS §0 row 20-21 both pre-sanction this shape; whether it is our own vendored `-sys` or the `llama-cpp-2` ecosystem crate is pinned at implementation, with DEBT-LOCKSTEP Rule 1's grep patterns updated in lockstep — §7). Spawned by the daemon with `env_clear()` (no `ANTHROPIC_API_KEY`, no XKEY-class material reachable via `/proc/self/environ`), no argv secrets, a pipe pair + a read-only model fd only. Pipe protocol: trivial length-prefixed prompt-in/response-out + ONE closed status byte — a process-internal seam, not a second wire codec.

**Why the third backend is third at the LANE level, not in the manifest:** `xport-harness` is built on EVERY required CI lane; its manifest gaining `llama-cpp-sys` would compile llama.cpp's C on the required lanes — violating the plan's kernel-firewall intent (the single most likely implementation error in the "third backend" framing, named so it cannot happen). Dependency direction is daemon→xport-core and harness→xport-core, never reverse, never daemon→harness. The backend token set is closed and FINAL: `backend ∈ {MOCK-DETERMINISTIC, ANTHROPIC-LIVE, LOCAL-ENGINE}`.

**Runner-up (documented, not chosen): supervised `llama-server` over a UNIX socket** (`--host /path.sock`; native `/completion`). Also §0-sanctioned and zero `-sys` churn — but it puts upstream's HTTP/template C++ surface in the loop against upstream's own SECURITY.md ("Do not use the RPC backend, rpc-server and llama-server functionality"), and it defeats pre-parse self-sandboxing (we don't control its `main()`). Choose it only if the `-sys` worker proves flaky; the B3 swap cost is identical either way. **Rejected: `llama-cli` stdin/stdout pipes** — not a machine protocol (banners/perf text), and raw model bytes interleaved with logs fight the §6 hex-encode discipline.

## 3. The wire — nothing new on the wire; one const in the leaf; the kernel path is staged

M32 serves the SAME `INFER_REQ`/`INFER_RESP`/`INFER_PENDING`/`ERR` framing M31 built, under the SAME `YUVA-M31-INFER-V1` MAC domain and the same host-custodied K. The tb-encode delta is **one const**: `peer::INFER_DAEMON = 0x03` beside `TB_VMM_HOST=0x01`/`QEMU_CHARDEV_HARNESS=0x02` (`inferwire.rs:194-199`).

**Anti-mint (the M30 §3c discipline, applied to the engine token):** the engine identity is bound by `peer_id` — already INSIDE every `infer_tag` MAC input (`inferwire.rs:1056ff`). **Stated precisely, per the review (§17.9):** the landed `kani_inferwire_peer_label_bound` harness covers `echo_tag` ONLY, on the concrete `0x01`/`0x02` pair (`crates/tb-encode/src/proofs.rs:4242-4265`) — it is NOT a generic every-MAC peer proof; the actual evidence for the new peers is the §9 closed-set negative extension flipping `0x03`/`0x04` under `infer_tag` in the `YUVA-M31-INFER-V1` domain, landed with stage B. The kernel renders `engine=VENDORED-C-LLAMACPP` from peer `0x03` via a **closed in-kernel map** and never echoes daemon bytes — a loopback, the guest itself, or llama.cpp's own output cannot claim an engine. `peer::INFER_DAEMON_PURE = 0x04` is **reserved at proposal time and wired ONLY by the B3 closure landing** (§12) — the retirement path is pre-registered down to the byte.

**The stage-B kernel scope flag (named NOW, not discovered mid-landing):** the landed kernel selftest pins the wire exchange BIT-EXACT to `mock_infer` (`tb-hal/src/mem/selftests.rs` stages `0x21`/`0x25`/`0x26`: a-priori window sizing, exactly one PENDING, bit-exact equality — the `live.rs:39-56` doc-comment states this is why no live byte reaches the guest). So **no local-engine byte can reach the guest today**, and an M32 marker claiming guest-visible local inference at stage A would be exactly the hollow-pass class the guards exist for. Resolution, staged:

- **Stage A — answer-first, host-adjudicated** (the M31-stage-C precedent): the guest exchange stays the pinned mock shape on every lane (stages 0x21/0x25/0x26 UNTOUCHED); the daemon runs the engine beside it and the workflow adjudicates the daemon witness. Token: `guestpath=MOCK-PINNED-HOST-ADJUDICATED` — the limit is on the witness, not in a footnote.
- **Stage B — a PARALLEL M32 receive path**, a separate reviewed landing: the kernel sends a second `INFER_REQ` (after the untouched M31 mock exchange) and receives REAL engine bytes under relaxed-shape rules — digest-committed, `total_len`-sized window, MAC-verified per chunk, PENDING budget in `[1, cap]` reusing the §2f reset semantics (model load/decode latency is liveness plumbing, never completion). **This is a genuinely NEW tb-hal driver seam, named here per the review (§17.18):** the landed exchange is ONE exact-size `chan_send_recv` call with `expect_rx` computed a-priori (`selftests.rs:1707-1712`); a digest-committed, `total_len`-sized, PENDING-budget-reset receive of unknown frame count is a **variable-length bounded-receive primitive** in the virtio driver — driver work, not just guard/witness work, and it appears in the §11 fan-out. The M32 lane's `QEMU_TIMEOUT` is budgeted for in-boot engine latency under TCG at stage B (the x86 lane is 40 s today; model load + 64 tokens ride inside the boot — the boot-hang class, measured before the pin). Guest witness `linfer:` + the marker move to guest serial; the §8.6 cross-process leg activates. The M31 selftest's exact-shape pin is never relaxed — the M32 path is new, beside it.

## 4. The determinism envelope (NAMED section — the claim is the envelope, never the model)

**The lane contract = the upstream-stated bit-exact envelope + the cross-runner pins.** Every pin is a witness field; the digest binds the whole tuple on one line (partial re-pins structurally impossible — same-line corroboration):

| Pin | Value | Why (cited) |
|---|---|---|
| engine source | vendored, ONE pinned b-tag + commit; **built from source, never prebuilt releases** | release binaries do runtime CPU-feature dispatch; the hosted fleet mixes AVX-512/non-AVX-512 machines (actions/runner-images#3389) → different SIMD paths → different reduction orders → different bits |
| build flags | `GGML_NATIVE=OFF`, `GGML_CPU_ALL_VARIANTS=OFF`, **`GGML_OPENMP=OFF`** (BLAS=OFF alone does not remove ggml's OpenMP pool; `OMP_NUM_THREADS=1` is runtime-only mitigation — §17.20), ISA pinned **x86-64-v3** (AVX2+FMA+F16C; AVX-512 OFF), `GGML_BLAS=OFF`; RPC backend + server tools + libcurl **compiled OUT** (§5) | the dispatch/variant machinery (llama.cpp build docs); BLAS spawns its own OpenMP pool and defeats `--threads 1` (issue #22956) |
| threading | `n_threads=1`, `OMP_NUM_THREADS=1` set after `env_clear` (belt-and-braces over the compile-time OFF) | #22956; fixed reduction order is the determinism primitive (arXiv:2601.17768) |
| serving | single context, single sequence, NO prompt cache, no continuous batching, `n_batch=n_ubatch` pinned | the PR #16016 envelope verbatim; multi-slot nondeterminism even at temp 0 (issue #7052); logits differ across batch shapes (Thinking Machines 2025) |
| sampling | greedy `top_k=1`, `temp=0`; **`seed=MOOT-GREEDY`** — the seed is pinned but never claimed load-bearing | greedy ≠ reproducible by itself (arXiv:2506.09501); the claim comes from bit-identical logits inside the envelope |
| prompt | raw pinned bytes, NO chat template (also a security rule, §5), fixed `n_predict ≤ 64` | templates drift across b-tags; the CVE-2024-34359 class (§5.4) |
| substrate | **the runner itself, pinned by label `ubuntu-24.04` (never `ubuntu-latest`), with measured witness fields `runner-image=`/`glibc=`** — the daemon+worker run ON the runner on BOTH arches (§10); the previous "pinned container image" row was fiction under both existing lane topologies (x86 runs bare on the runner; the aarch64 boot container is a FLOATING trixie tag, which after this amendment hosts ONLY QEMU — §17.7/§17.19); glibc/libm is in the envelope (softmax/exp go through libm); CPU-only (no GPU exists on the runners, and GPU is nondeterministic anyway — issue #10197) | a runner-image version roll that moves a golden is a loud finding re-pinned by reviewed commit, not a flake |

**Earned per run — stub-resistant by construction (the §17.1 must-fix, applied additively):** the review showed the original 2-run + perturbed-byte design passes under a ~30-line canned-response stub, with every probe self-reported by the suspect process. The amended evidence has four legs:

1. **Repo-pinned golden digests, pulled forward from stage C to stage A.** For each (model × pinned prompt × tuple × lane-substrate) a golden `resp-digest` is **measured at landing and committed in-repo**; the WORKFLOW adjudication string-compares the witness digest against the repo constant — an independent second stream (repo constant vs daemon stdout), not daemon self-report. A hash-stub or broken engine cannot produce the golden. Mismatch = loud-red finding; re-pinned only by a reviewed commit binding engine-commit + gguf hash + runner-image + golden in one ledger edit. Token: `golden=REPO-PINNED-MATCHED`. The residual microarch-conditional divergence risk inside the pinned ISA is accepted as a loud finding, not a flake (the M24 refuse-not-overclaim posture); if the fleet proves heterogeneous-within-pin, the named fallback is keying goldens per detected microarch class — a reviewed amendment, never a silent relax.
2. **The fresh-process 2-run compare** (the daemon spawns the worker **twice** — in-process caching structurally cannot satisfy the compare) on the pinned prompt; `resp-digest == resp-digest-run2` must string-equal. Stated plainly per §17.10: this leg (and leg 3) is single-stream and therefore **daemon-honesty-conditional under `host=RESIDUAL-TCB`**; leg 1 is the independent leg.
3. **The frozen second-prompt negative control (replacing the per-run perturbed byte — the §17.2 must-fix).** A single perturbed byte does NOT guarantee a different greedy continuation at `n_predict ≤ 64` on a 260K/15M-param model — attention can wash it out, and the original design would misreport that legitimate outcome as "vacuous compare" and red a push-required lane. Instead: a SECOND fully pinned prompt (`NEG-PROMPT`), with its OWN repo-pinned golden, whose digest-distinctness from the primary is **measured at pin time and re-measured on every pin/model bump** (the M24 freeze-then-measure discipline; the two goldens being distinct is asserted in-repo at landing). The run must match each golden respectively — a stub or cache returning the same bytes for both prompts fails one golden; a false red is structurally excluded because distinctness was measured, never assumed. Tokens: `neg-prompt=FROZEN-PINNED neg-digest=… neg-golden=REPO-PINNED-MATCHED`.
4. **The per-run nonce-freshness leg (the canned-table killer).** The workflow samples a 16-byte nonce per run; the daemon appends its raw bytes to a pinned prefix and runs the fresh-process 2-run compare on THAT prompt (`nonce-digest == nonce-digest-run2`). No golden can exist for it by construction, so its claim is exactly freshness + envelope-determinism — but a canned-response table cannot cover unseen prompts. **The determinism-vs-freshness tension, named (the review required this, §17.1):** NO instruction-following challenge transform is possible here — stories260K/15M cannot follow instructions, and even the live Anthropic lane had to downgrade HEX-REVERSE → HEX-GROUP-REVERSE for capability realism (`live.rs`, d25e190) — so the transform-class anti-canning of M31 stage C is EXCLUDED by model capability, deliberately, not silently. The stub-resistance budget is therefore: golden (kills hash-stubs) + nonce-2run (kills canned tables) + DEBT-LOCKSTEP Rule 3 (proves the C is in the build graph). The residual forgery — a daemon that deliberately prints the repo-pinned golden without running the engine — is the same honest-host class M30/M31 already concede under `host=RESIDUAL-TCB`, conceded again in §15.

Only when all four legs hold are `repro=BIT-IDENTICAL-2RUN determinism=BUILD-PINNED-SINGLE-THREAD-GREEDY` emitted. The worker asserts its own `llama_print_system_info()` feature string against the pinned expectation in-process (`sysinfo-pinned=0x1` — the tripwire against an upstream bump silently re-enabling dispatch/AVX-512). The digest claim is **`digest=TUPLE-CONDITIONAL`** — stable under THIS pinned tuple, never "model-canonical."

**The honesty ladder:** stage A lands with goldens active and `xmachine=NOT-CLAIMED xarch=NOT-CLAIMED` (golden matches accumulate cross-machine evidence within the pinned image, but the cross-machine TOKEN is promoted only by stage C after an observation window — `xmachine=GOLDEN-DIGEST-MATCHED`, falsifiable and loud-red; un-promotion is a reviewed act). Goldens are keyed per lane-substrate; note both CI lanes' host peer (including the aarch64 guest lane) is the same x86_64 host binary on the same runner class, so cross-ARCH equality never arises in CI; `xarch=` guards future native-aarch64 hosts (NEON reduces differently by construction).

**NEVER claimed (each would be false):** batch-size invariance (upstream refuses to maintain it — PR #16016; we pin the batch instead), thread-count invariance (unexercised at threads=1), determinism-from-quantization (k-quants are int8-dot + **FP32 scale/accumulate + FP norms/softmax** — only integer-ONLY pipelines à la I-BERT/I-LLM are exact by construction, and llama.cpp is not one), cross-engine output equivalence (B3 §5.3 rejects it as dishonest), SGLang/batch-invariant-kernel-class concurrent determinism (the lane is stop-and-wait single-`req_id` by construction; if the daemon ever serves concurrent requests, the token must be re-scoped BEFORE that traffic exists).

## 5. The security shape (NAMED section — hardening, never an isolation claim)

1. **Worker self-sandbox installed BEFORE GGUF parsing** (the parser IS the attack surface — inverting llamafile's post-load order): (1) open the model fd; (2) `prctl(PR_SET_NO_NEW_PRIVS)`; (3) seccomp-BPF allowlist via the `seccompiler` crate (the Firecracker lineage) with `TSYNC` before llama.cpp spawns threads — `socket`/`connect`/`execve`/`ptrace` denied outright (network egress and C2 structurally impossible from the worker); (4) Landlock ruleset via `rust-landlock`: read-only on the model path, nothing else (the pinned `ubuntu-24.04` runner, kernel ~6.8 = ABI v4 — and the daemon+worker run ON the runner on both arches, §10, precisely so this row is satisfiable). **NOT built on user namespaces/bwrap/containers** — `apparmor_restrict_unprivileged_userns=1` is the 24.04 default and the mechanism is porous anyway (Qualys 2025 bypasses).
2. **The sandbox is itself anti-hollow AND its own pre-flight:** after installing filters the worker runs in-process negative probes — `socket(AF_INET)` must fail, `open("/etc/passwd")` must be denied — and reports closed probe tokens over the pipe; the daemon folds them into the witness (`sandbox=SECCOMP-LANDLOCK-PRE-PARSE net-denied=0x1 fs-denied=0x1 envclear=0x1`) and the guards positively require them. These probes double as the per-run **sandbox pre-flight** (§17.7): a runner image whose kernel/LSM stack stops providing the pinned seccomp/Landlock ABI is a loud named FAIL (`sandbox-preflight`), never a skip. A silently disabled sandbox (the llamafile-0.10.0 regression pattern — they DROPPED their pledge/SECCOMP sandbox in a rebuild) becomes a red lane, not a memory.
3. **Build-surface minimization:** the vendored tree compiles OUT the RPC backend (CVE-2024-42479, write-what-where, 9.8 + SECURITY.md "do not use"), the server/HTTP tools, and libcurl (no network code in the binary at all); CPU backends only. A run-script grep asserts the rpc/server objects are absent from the build (the M29/M30 guard idiom on the build surface).
4. **GGUF-embedded chat templates are NEVER rendered:** raw-completion only, the daemon builds the prompt. The class precedent is CVE-2024-34359 — **accurately cited per the review (§17.16): a Jinja2 SSTI in llama-cpp-python (the Python BINDINGS, 9.7), not llama.cpp's own template path** — model-metadata-as-template is the demonstrated attack class, and the rule kills it regardless of which layer renders. Tokenizer/vocab/metadata strings from the GGUF are untrusted bytes (CVE-2025-49847 proves vocab handling is exploitable) and cross to any log hex-encoded only (§6).
5. **Artifact custody (the Probllama inversion — CVE-2024-37032's anti-pattern, inverted; amended per §17.5 to be unattended-fit):** SHA256 pins in-repo. **The 1.19 MB `stories260K.gguf` is committed IN-REPO** (below any LFS concern — §17.17) — the smoke leg is fully network-independent. The 26.7 MB Q8_0 artifact rides `actions/cache` keyed by hash, and **on cache miss a dedicated workflow SETUP step re-fetches it hash-pinned** (network confined to workflow setup, the same class as toolchain installs — GitHub evicts caches routinely at 7-day-unused/10 GB pressure, and a push-required lane that hard-fails on eviction until an operator re-seeds would be a standing unattended-run breaker, contradicting §1). **Hard-FAIL is reserved for hash MISMATCH**; the SAFE-RUST daemon re-verifies the sha256 after every restore/fetch and BEFORE the worker fd handoff — the C parser never touches an unverified byte — and the DAEMON itself never downloads anything (missing/mismatched at run time = the exit-3 refusal; Rule 4 makes daemon-side zero-network structural). The pin guarantees identity, not safety: `weights=UNTRUSTED-INPUT-NAMED` stays even with the pin (the hub-poisoning base rate is measured — MalHug, arXiv:2409.09368). sigstore model-transparency/OMS signatures are a named successor, not stage A.
6. **Resource caps + liveness:** `RLIMIT_AS` on the worker (the V-02-class memory-exhaustion flaws); daemon-side wall-clock kill; worker death/hang after first contact = hard FAIL (the M30 present-then-silent rule), never a skip. Worker faults map to ONE new closed code `ERR code=ENGINE-FAULT` extending the M31 §2e enum — C library text never crosses anywhere.
7. **What is NOT claimed:** the sandbox is hardening inside `host=RESIDUAL-TCB`, **not an isolation boundary** — the worker shares the host kernel with the runner, the daemon, and tb-vmm; a worker escape is host compromise, named in assumptions.md §3c (§11). The vocabulary `secure|sandboxed|isolated|trusted|hardened` is banned near M32 lines except as the declared closed tokens (§8.9) — the plan's "must not overstate isolation" amendment exists because "separate driver daemon" overstated once already.
