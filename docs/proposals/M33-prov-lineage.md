## 1. Why this milestone, and why these choices

M29 made the prov head a real keyed cryptographic MAC — tamper-evident, collision-resistant under the assumed primitive. But a MAC is **symmetric**: the verification secret *is* the minting secret. Anyone who can read K can forge a head that verifies. The sovereignty plan calls this PARTICIPATION; M33 is the milestone that closes it to EXCLUSIVITY. Four places a naive "just add Ed25519 and sign the head" milestone would rot, each fixed by construction:

1. **Curve crypto is not Kani-provable, and that is dispositive — not a tuning problem.** Ed25519 verify is scalar multiplication over `GF(2^255-19)`: hundreds of 256-bit modular multiplications, each a bignum carry chain — and `m29-crypto-mac-literature.md §5` names carry chains on symbolic adds as the CBMC SAT-explosion cause; field mul is that squared. The literature is unanimous and load-bearing: **every** published curve25519/HBS formal verification uses *deductive/synthesis* tools — fiat-crypto synthesizes field arithmetic with machine-checked Coq (correct-by-construction), HACL\*/EverCrypt proves in F\*, Bedrock2/Rupicola generates verified C, eprint 2026/134 ("Completing the Chain") links a Jasmin XMSS to an EasyCrypt *security* proof (Jan 2026) — **none uses bounded model checking, because it does not scale to field arithmetic.** An in-house curve25519 proven by Kani is **infeasible**; P-256 is strictly worse; both also forfeit post-quantum resistance. **DISQUALIFIED** (§ the prompt's "weigh this heavily" — weighed, and it decides the family). The forced choice is a hash-based signature: a pure hash chain, implementable+verifiable in-house the same way M29 implemented BLAKE2s, consuming the landed leaf and adding no field math.

2. **No real-parameter HBS verify fits the sharded Kani lane — this is a structural wall, not a budget headache.** Each LM-OTS / Merkle chain-step input is a single sub-64-byte block = exactly **one** SHA-256 compression, so `verify_compressions == verify_hash_calls`. Worst-case verify compression counts (re-MEASURED before any `EXPECTED_HARNESSES` bump, per the #49/khash discipline; using the RFC 8554 Table-1 chain counts `p∈{265,133,67,34}` for W1/2/4/8): LMS-**`W4` p=67 ≈ 67×15 chain-steps + ~10 Merkle ≈ 1062 compressions** (the headline set); `W2` p=133 ≈ hundreds; `W1` p=265 ≈ many hundreds (the draft's "~275/~399/~1015" figures are superseded — they conflated p=34 with W4; the corrected `p=67` set is in the References and the co-landed survey, §16); `W8` p=34 ≈ 8670; XMSS ≈ 130–150; SLH-DSA-128s ≈ thousands. **CONCLUSION (parameter-error-corrected, conclusion unchanged): no full-parameter HBS verify lands in the lane.** So M33 (a) does the VERIFY/SIGN SPLIT (§2), and (b) re-scopes the Kani obligation (§9) to a **small concrete toy instance at `w=1`** (`p≈2, h≈1`, ≈ 9–14 compressions — *only `w=1` hits the khash 2–8 budget regime; a `W4` toy at `p=4,h=2` is ≈ 70 compressions and does NOT fit*, §9, §16), pinning full-parameter correctness via host-test + the in-boot RFC 8554 KAT. If a reviewer insists on a full-parameter symbolic verify proof, **no hash-based scheme is feasible and M33 cannot land as specified** — this is the single make-or-break, pre-negotiated here before any code.

3. **A CI-held signing key gives exclusivity against WHOM — and the honest answer is brutal.** The plan tokens `key=SIMULATED-ENROLLED`; M33 makes it sharper: `key=SIMULATED-ENROLLED-CI-CUSTODIED exclusivity=OFF-PLATFORM-ONLY`. A signing key held in the autonomous CI runner gives exclusivity against **anyone NOT holding it** — i.e. anyone *off-platform*: a third party, a future auditor, a downstream consumer of the attestation. It gives **ZERO** new protection against a CI/host compromise, a repo-write maintainer, a malicious workflow dependency, or a fork with secrets-inheritance misconfigured — the holder of the signing key in CI **is the CI**, not the operator (the M28/M31 `host=RESIDUAL-TCB` residual, carried verbatim). The gain over M29 is exactly: *public-verifiability + exclusivity against parties outside the signing host* — strictly more than a MAC (which must *share* the verification secret with every verifier). That asymmetry — **public verify, private sign** — is the entire point and the only thing M33 should claim over M29. The named upgrade ladder that restores cross-platform exclusivity only when custody leaves CI is §7; no rung's token overclaims the one below it.

4. **A signature proves authenticity-of-extension; it does NOT prove the chain is honest, complete, or fork-free — and the attestation must measure only what it can.** The signature binds *who extended the head*; it does not detect a *split-view fork* (a single signer signing two divergent heads is undetectable without an external witness/gossip quorum — RFC 6962/Rekor; tokened `splitview=UNDETECTED-NO-WITNESS-QUORUM`). The SLSA/in-toto attestation codec carries only truthfully-measurable subjects — the `crates/brand` build-id and the kernel-image digest via `khash::uhash` — under `measure=SELF-NO-HW-ROOT selfmeasure=UNATTESTED-LOADER`: **the QEMU image is loaded directly, with no measured boot, no chain of trust, no RTM, and the digest is computed by the very image being attested** (`assumptions.md` A4/A6); claiming SLSA L3 non-falsifiability or "measured boot" would be the exact overclaim the tokens exist to prevent (§5). M33 reaches toward **SLSA L1/L2** (an artifact exists / is signed); L3 is M36.

**And one thing M33 is, deliberately:** *autonomously buildable, fit for unattended runs* — the verify leaf needs only the 32-byte public root in the image; the toy-instance Kani proofs + the in-boot KAT run with zero network, zero secrets. The **signing** side (the only key-custody surface) is the named honesty problem, not a network/secret dependency: the CI-held simulated key is compiled-in exactly like M28's, so the stage-A lane is green unattended on both arches.

## 2. The shape — the VERIFY/SIGN SPLIT (the load-bearing decision)

The kernel TCB and the Kani lane see **only verification**; signing and all signature *state* live host/operator-side. This is what makes a stateful scheme safe to adopt: the catastrophic-state hazard (§3) never enters the verified kernel.

- **The kernel leaf — `crates/tb-encode/src/lmsig.rs` (NEW; VERIFY ONLY).** `#![no_std]`, `forbid(unsafe_code)`, zero-dep, the M29/M22 leaf discipline. One public entry point:

  ```rust
  /// Verify an LMS signature `sig` over `msg` against the 32-byte public root
  /// `root` (+ the 4-byte identifier `I`). TOTAL: fail-closed to `false` on ANY
  /// malformed byte (bad type word, out-of-range w-coefficient, wrong sig length,
  /// short buffer) — never panics, never allocates (the `prov::verify_inclusion`
  /// canon discipline: bad input → false, exactly as a bad buffer → 0/None).
  pub fn lms_verify(root: &[u8; 32], i_id: &[u8; 4], msg: &[u8], sig: &[u8]) -> bool
  ```

  Internally: LM-OTS verify walks each of `p` Winternitz chains from the signature coefficient up to `2^w-1` (input `I ‖ u32str(q) ‖ u16str(i) ‖ u8str(j) ‖ tmp = 23+n = 55 bytes = exactly 1 compression`, RFC 8554 §4.6 Alg 4b), recomputes the LM-OTS public key, then walks the `h`-node LMS Merkle auth path to a candidate root, and accepts iff it equals `root`. **The Merkle leg is structurally `prov::verify_inclusion`** — a fold over `(leaf, siblings)` accepted iff it lands on the committed value — so #75's future balanced-batch-Merkle upgrade swaps only the *path-walk shape*, not the verify contract (§ composes-with-#75, stated in the one-line). The hash is **SHA-256** (the second verified leaf, §4), not khash — the RFC pins it and the conformance claim depends on it.

- **The host/operator signer — `tools/prov-signer/` (an independent nested workspace; the tb-vmm / xport-core precedent).** Holds the LMS private key (the simulated enrolled key, compiled-in like M28's), the **never-reuse OTS leaf-index counter** (§3), signs the 32-byte prov head + the attestation PAE bytes, and emits the witness. `key=SIMULATED-ENROLLED-CI-CUSTODIED host=RESIDUAL-TCB`. The kernel never holds the secret key, never holds signature state, and is **stateless-by-construction** — it only ever sees `(root, I, msg, sig)`.

**Why verify-in-kernel and sign-host-side is the right cut (not relative hash cost).** For LMS, *sign* reveals chain elements (a cheaper hash count than full verify) — so the split is **not** justified by "verify is cheaper." It is justified by **key custody + state**: the secret key and the never-reuse counter are exactly the two things that must not enter the verified, stateless kernel TCB; the public root and the verify routine are exactly the two things that can. That is the entire architecture.

## 3. The statefulness hazard — contained host-side, named not solved (the stage-A key lifecycle stated concretely)

LMS/XMSS secret keys are **stateful**: reusing any one-time leaf index destroys all security of that key (RFC 8391/8554; SP 800-208 §1.1 calls state reuse via counter-reset / restore-from-backup / failover **"catastrophic and unrecoverable"** — requiring key-pair revocation; the BSI laser-fault-injection study shows it is *practically* inducible, not theoretical). The prov head **advances on every memory mutation**, so a naive "sign the head on every append" would reuse the OTS key catastrophically and reintroduce exactly the weakness M33 exists to close.

**Resolution (the honest reason statefulness is acceptable here): it never enters the verified kernel — AND the stage-A CI lane reuses indices deliberately, on a key with no security value, said loudly on its own token.**

- **Sign only at checkpoint/epoch boundaries**, never per-append: one OTS leaf consumed per *signed* head, the leaf index strictly monotonic. `H10 = 1024` signatures per key covers the prov-head checkpoint rate; **exhaustion = rotate key + re-anchor root** (a named limit, `state=HOST-CUSTODIED-NEVER-REUSE`, exhaustion tokened at rotation).
- **The leaf-index counter is host-custodied, crash-durable, never-decrement.** In stage A it lives in the signer process. The honest persistent custodian, when the M35 monitor exists, is the **M20 superblock's monotonic two-phase-commit `gen` counter** (`blkfmt`), with a never-decrement invariant — but M35 is downstream, so stage A states the counter durability/atomicity across CI re-runs / restored scratch disks / matrix jobs as a **host obligation outside the verified TCB** (`assumptions.md` §, alongside the M28 `oldkey-zeroized` residual).
- **The relocation hazard is named, not waved away (§16 must-fix).** A two-level "fresh per-checkpoint sub-key whose root is signed under the long-term root" does NOT make stage A reuse-free: the long-term/top-tree root still consumes one OTS leaf per sub-tree, so the top index must itself be durable+monotone — the hazard moves up one level, it is not closed. **So the stage-A CI lane is honest about the two distinct cases:**
  - **The compiled-in simulated key (default stage-A CI lane):** a fixed long-term public root is baked in the image. Every CI run re-signs sub-tree #0 under top-leaf #0 — i.e. it **DELIBERATELY REUSES leaf indices across runs**. This is acceptable *only because the simulated key carries no security value whatsoever*; the lane therefore emits **`state=SIMULATED-REUSE-OK-NO-SECURITY`** (NOT `HOST-CUSTODIED-NEVER-REUSE`, which would be a false claim for that lane). The stage-A signatures are **run-ephemeral**: they MUST NOT be treated as cross-run-valid, and the persisted-head record (§6) is rewritten fresh each invocation against the fresh-per-invocation scratch disk — the two-boot DoD (§6) lives **within a single script invocation against one disk file** (the M20 disk is created fresh per run, `run-aarch64.sh:50`), so no cross-run persisted signature ever outlives its run.
  - **A persistent key (rung 1+, §7, M35):** the durable, never-decrement TOP-LEVEL counter obligation becomes real and is owned by the M35 monitor (`leafidx=DEFERRED-TO-M35-MONITOR`); only then does `state=HOST-CUSTODIED-NEVER-REUSE` honestly apply.
- **`key_evolve` does NOT and MUST NOT touch the LMS leaf-index advance (§16 should_fix).** M28's symmetric MAC keys run the FssAgg PRF-chain `key_{i+1}=khash(key_i, EVOLVE_DOMAIN)` (`opframe_rx.rs:380`); the M33 asymmetric LMS signing key and its monotone OTS leaf-index are a **separate custody domain with a separate state machine**. An implementer must NEVER "evolve" the LMS signing seed via `key_evolve` — that is PRF-chain key evolution, not OTS-leaf advancement, and conflating them silently breaks the OTS structure. The two state machines are disjoint by construction; stated here and in §7 so it cannot be misread.
- **The crash-ordering window is closed by construction at stage A (§16 must-fix).** Sign-at-checkpoint + persist-to-M20 has a write-ordering hazard: a crash between (a) consuming/advancing the leaf-index, (b) persisting the index, and (c) persisting the ~2516-byte multi-sector signed record could leave a consumed index with no durable record, or a durable record whose index did not advance → reuse on replay. Stage A closes this by **single-signature-per-invocation-by-construction**: the run-ephemeral lane signs exactly one head per boot-1 against a fresh disk and the persisted record is committed index-durable-before-signature-emitted as one atomic multi-sector write (§6); the *persistent* never-decrement custody (where the window is real across reboots) is the named M35 obligation, not a stage-A claim. SPHINCS+ would avoid statefulness entirely but at prohibitive kernel-verify cost (§1.2) — a genuine tradeoff, named, not a free lunch.

## 4. The hash conformance fork — RESOLVED to D2 (a second verified SHA-256 leaf)

RFC 8554/8391 **pin** the hash to SHA-256 (SP 800-208 approves only SHA-256/SHA-256-192/SHAKE256; BLAKE2 is outside the approved set). Two honest options were weighed:

- **(D1) Substitute house-BLAKE2s into LMS** ("LMS-shaped, non-conformant"): reuses `khash` verbatim, **zero** new hash code / Kani cost — BUT cannot claim RFC 8554 / SP 800-208 conformance, the official KAT vectors **do not apply** (they are SHA-256), so we must mint our own vectors (weaker pedigree). Token would be `sig=LMS-BLAKE2S-NONSTANDARD conformance=NONE`.
- **(D2) Implement SHA-256 as a SECOND zero-dep verified leaf** — `crates/tb-encode/src/sha256.rs`, the M29-from-the-RFC discipline (FIPS 180-4 / RFC 6234, ~200–300 LOC, its own Kani totality/determinism/KAT harnesses + a second `.rodata` constant table). Gains RFC 8554 conformance + **official KAT pedigree** (FIPS 180-4 / RFC 6234 vectors; the Appel TOPLAS 2015 SHA-256-correctness lineage). Token: `sig=LMS-SHA256-W4-H10 conformance=RFC8554`.

**DECISION: D2.** The M33 value proposition is **exclusivity via a STANDARD scheme**; a non-standard hash would undercut exactly the pedigree that makes the signed root worth more than a MAC. The cost is one extra primitive and its harnesses (named in §9's `EXPECTED_HARNESSES` plan, measured before pinning). **But the SHA-256-under-CBMC seconds-per-compression rate is UNMEASURED and is the second make-or-break number (§9, §16):** the M29-measured ≈9s figure is calibrated on the *BLAKE2s ARX compression* (10 rounds over a fixed sigma permutation); SHA-256's 64-round structure with its 16→64-word message-schedule expansion (sigma0/sigma1) has a *different symbolic-SAT shape* and is plausibly 1.2–2× more expensive per compression in CBMC. The §9 budget table therefore treats SHA-256/compression as **UNMEASURED until benchmarked on a one-block harness**, and the SHA-256 leaf is flagged as the **likely binding budget constraint** — possibly forcing `kani_sha256_kat` host-side *from the start*, not as a fallback rung. The witness **never implies RFC conformance under any path that did not run SHA-256** — `conformance=RFC8554` is emitted *only* on the SHA-256 build, earned by its KAT, never compiled-in. (D1 remains the named fallback **only** if a future budget crisis makes the second leaf's harnesses unaffordable — a reviewed amendment, with the token loudly flipping to `conformance=NONE`; it must never silently happen.)

**The in-boot KAT (the `khash::kat_ok()` pattern, mandatory) — SMALL-PARAMETER per boot, full-parameter host-side (§16 should_fix).** A fail-closed boot self-test RECOMPUTES an RFC 8554 test-vector roundtrip's verify side over real SHA-256 compressions and emits `kat=RFC8554-PASS` **only** on `true` — never a compiled-in constant compared to itself (the exact `khash.rs:379` discipline). **Because a full W4/H10 verify is ~1062 SHA-256 compressions and the aarch64 boot ceiling is 90s with a cold-start that has historically raced it (`run-aarch64.sh:34,43`), the per-boot KAT is bounded to a SMALL-parameter roundtrip** (the same `w=1`, small-`p`,`h` instance the Kani toy uses, plus a one-byte negative control — cheap, deterministic, never races the ceiling); **full W4/H10 official-vector correctness is a host-side `cargo test` KAT**, not an every-boot cost. The boot cost of the small KAT is measured and budgeted before pinning. Plus a **non-vacuity negative control** in the boot path: a one-byte-perturbed signature must verify=`false` in the same KAT — and, per §16 must-fix #1, **two independent regional controls** (one flip in the OTS-signature region, one flip in the Merkle-auth-path region, §8) so a half-verifier turns the boot red. The SHA-256 leaf gets its own `sha256-kat=FIPS180-4-PASS` over the FIPS 180-4 vectors (a distinct token KEY, disjoint from the bare `kat=` key M29 owns — §8, §16).

## 5. The attestation codec — DSSE-PAE-shaped, truthfully-measurable only, self-reported digest named (NAMED section)

The plan's third M33 deliverable is the **SLSA-style attestation codec leaf** (`crates/tb-encode/src/attest.rs`; consumed by M34/M36; doubles as the machine-readable sovereignty-ledger carrier). It is a separate Kani-proven tb-encode codec — the `prov.rs` / `opframe_rx.rs` canon/decode pattern, **zero new hash math** — NOT a JSON producer.

- **The signed bytes are DSSE PAE.** `PAE = "DSSEv1" SP LEN(type) SP type SP LEN(body) SP body` — an ASCII length-prefixed injective encoding, trivially `no_std`/zero-dep, the house length-prefixed-injective canon (the `opframe_rx` "MAC-bytes-excluding-tag" split reused byte-for-byte: the signer signs the PAE bytes, the verifier re-derives them). Token `attest=DSSE-PAE-SHAPED` — we model the **byte-level PAE injectivity + roundtrip + fail-closed decode**, and we **do NOT** claim JSON-canonicalization security (the known footgun; eprint 2026/192 "Verification Theatre" is actively policing exactly this overclaim — over-scoping a JSON-semantics proof manufactures a green harness asserting less than it appears to).
- **The carried claims are only what `measure=SELF-NO-HW-ROOT selfmeasure=UNATTESTED-LOADER` permits, and the digest is SELF-REPORTED (§16 should_fix).** The in-toto Statement v1 SUBSET the codec encodes: `subject_digest` (32-byte `khash::uhash` of the kernel image), `builder_id` (from `crates/brand` — the SLSA `builder.id`, the sole SLSA Build-level determiner), `buildType` (enum), a `materials` 32-byte-digest list, the toolchain hash, and the **sovereignty ledger** (`dep + status TEMPORARY|ACCEPTED-PERMANENT|QUARANTINED` per plan principle 1 — the machine-checked carrier the M32 debt token feeds into). **Crucially, `subject_digest` is computed by the very image being attested** — QEMU loads the kernel directly with no external measurer, so a malicious image computes `khash::uhash` *of whatever it chooses to report*. The new token `selfmeasure=UNATTESTED-LOADER` makes this explicit so no reader can infer "the attestation independently measures the image" from `attest-digest=0x…`; M34 (which trusts this as *image identity, assumption-conditional*) inherits the boundary by name, never by marker-grep. It is an **SLSA-CONCEPT-SHAPED fixed-width subset, not a wire-compatible SLSA/in-toto JSON producer** — said so on the token.

## 6. The signed prov head — persisted to M20 (the genuinely-new contract; the multi-sector codec is NEW work)

Today prov heads are **per-boot in-RAM** (`prov.rs` `chain_head`); M33's second deliverable persists a *signed* head across reboot — the two-boot harness, **within a single script invocation against one disk file** (the M20 disk is created fresh per invocation, `run-aarch64.sh:50` / `run-x86_64.sh:63`; "two-boot" means boot-1 then boot-2 against the SAME disk, exactly as M20 already works — NOT persistence across CI runs, §3, §16).

- **The on-disk frame is a NEW multi-sector codec, not a trivial blkfmt reuse (§16 must-fix).** An LMS signature is ~2516 B (W4/H10) — and the landed `blkfmt::record_frame_encode/decode` hard-caps at `MAX_PAYLOAD = SECTOR_SIZE − FRAME_HEADER_LEN = 488` with a `len: u16` header and a **per-512B-sector FNV-1a-32 torn-write detector** (`blkfmt.rs:314,318,376,402-409`). A ~2516-byte record spans **multiple sectors**, so this is **genuinely new codec/state-machine work**, NOT a reuse of the single-sector primitive: it needs (a) a **cross-sector torn-write detector** (today's checksum is per-sector only — a multi-sector record needs a record-spanning checksum so a torn *middle* sector is caught), (b) an **atomic multi-sector commit** (the index-durable-before-signature ordering of §3), and (c) a **fail-closed decode that rejects a partial/torn record** (`None` → treat as unsigned/fresh, never partial-read, never panic — the `record_frame_decode` discipline extended across sectors). The record shape: `{ magic, version, head:[u8;32], I:[u8;4], q:u32 (leaf index used), sig:[u8;~2516], pubroot:[u8;32], record_checksum }`. The signature's *authenticity* is verified separately by `lms_verify`, never trusted from the on-disk checksum (the checksum is torn-write detection only). **This new codec gets its own Kani harnesses (§9): roundtrip-injectivity + fail-closed-on-torn-middle-sector + fail-closed-on-bad-magic/version/length — pure layout, no hashing of symbolic key material, cheap.**
- **The two-boot DoD.** Boot 1: the host signer signs+persists the head (single-signature-per-invocation, §3). Boot 2: the kernel reads it back via `VirtioBlkStore`, runs `lms_verify(pubroot, I, head, sig)` against the **image-embedded root**, and asserts head-survives-reboot + signature-valid + the two regional tamper-rejections (§8). Token `head=SIGNED-PERSISTED-M20`.
- **Composes with #75.** The signed root sits *over* `prov::head_witness`/the 32-byte head — the authenticity layer over the M22 fold. When #75 lands balanced-batch-Merkle, it changes the *head producer* (the fold → a tree root) but **not** the sign/verify contract: the signer signs whatever 32 bytes the head producer emits; `lms_verify`'s Merkle-path leg is already the `verify_inclusion` shape. "Sign an opaque 32-byte root in a separate module so the Merkle successor swaps only the root producer" — exactly the survey's design rule.

## 9. Kani obligations + budget (the #49 / M29-measured per-compression discipline — RE-COST at the MEASURED tamper rate)

**The make-or-break section.** No full-parameter LMS verify is provable (§1.2). The leaf is proven over a **SMALL CONCRETE toy instance at Winternitz `w=1`** — *only `w=1` hits the budget*: a `p=4,h=2` toy at the headline `W4` (max_chain=15) is ≈ 70 worst-case compressions and does NOT fit, so the toy is pinned to **`w=1` (max_chain=1)** at `p≈2, h≈1` ≈ 9–14 compressions (§16 must-fix). **Two corrected cost laws (§16 must-fix):**
- **The 9s/compression figure is the CONCRETE-CONSTANT rate, not the tamper rate.** Measured in the landed suite: `khash_total_deterministic` ≈ 7.5 s/compr and `khash_vectors` ≈ 9.9 s/compr are *fully-concrete, constant-propagated* harnesses; every TAMPER / symbolic harness measured **18–56 s/compr** — `kani_khash_tamper` 2 compr/112.2s ≈ 56 s/compr, `prov_chain_mix_tamper` 2/108.2s ≈ 54 s/compr, `prov_inclusion_sound` 8/147.9s ≈ 18.5 s/compr, M31 `resp_binding`/`domain_sep` 3/70–75s ≈ 23–25 s/compr. - **The SHA-256 per-compression CBMC rate is UNMEASURED** (§4, §16) and MUST be benchmarked on a one-block harness BEFORE this table is trusted and before `EXPECTED_HARNESSES` is bumped; SHA-256's 64-round + message-schedule shape is plausibly 1.2–2× the BLAKE2s rate. **The symbolic-coefficient trap is the #49 trap in a new costume (§16 must-fix) — solved by the M31 PINNED-VECTOR iff.** An LM-OTS verify derives each Winternitz chain's walk length from the signature coefficients, which are derived by HASHING msg‖sig — so a *symbolic-flip-index over the signature* would drive LOOP BOUNDS through a compression with symbolic data, the banned >5-min #49 class (the M30 FrameAccum >120s floor; the M31 "symbolic payload flips killed"). The correct shape is the **M31 `KANI_RESP_BINDING_PIN` iff** (`proofs.rs:4716-4799`): a **fully-symbolic candidate signature** compared for equality against a **host-rederived PINNED genuine signature constant**, yielding **ONE concrete verify execution** — `lms_verify(...) == true ⟺ sym_sig == PINNED_GENUINE_SIG`. This subsumes the flip/restore legs (a flipped sig is on the `!=`/reject side, the genuine is on the `==`/accept side) and KILLS every construction-drift mutant at the Kani level, keeping all compressions concrete. The two REGIONAL controls (OTS-region flip, Merkle-region flip) are concrete instances on the reject side. Without the pin, the tamper harness does not terminate in the lane — so the pin is mandatory, not optional.

## 14. Honest caveats (conceded — encoded as witness tokens where applicable)

- **`M33: prov-lineage OK` proves PUBLIC-VERIFY/PRIVATE-SIGN ASYMMETRY, not operator exclusivity.** The signing key in CI is the CI's (`key=SIMULATED-ENROLLED-CI-CUSTODIED exclusivity=OFF-PLATFORM-ONLY`); the gain over M29 is exclusivity against parties *outside the signing host* + public-verifiability — strictly more than a shared-secret MAC, and **nothing** against a host/CI compromise. §8.7 bans the vocabulary that would overclaim it.
- **The stage-A CI key DELIBERATELY REUSES leaf indices** (`state=SIMULATED-REUSE-OK-NO-SECURITY`) — acceptable only because the compiled-in simulated key has no security value; `HOST-CUSTODIED-NEVER-REUSE` becomes honest only at rung 1+ with the M35 durable counter (§3, §7). The two-level-tree "resolution" relocates the reuse hazard up one level, it does not close it; named, not waved.
- **A signature is not unforgeability-proven.** `sec=ASSUMED-FROM-LITERATURE`: LMS EUF-CMA + hash 2nd-preimage/collision resistance are assumed; no tool in the field proves primitive security, and the *original* SPHINCS+ tight proof was flawed (eprint 2026/192 "Verification Theatre" polices exactly this). The witness stays at the assumed tier and never implies a proven bound.
- **A single signer cannot detect its own fork** — `splitview=UNDETECTED-NO-WITNESS-QUORUM`; a witness/gossip quorum (RFC 6962/Rekor) is a named successor, not an M33 claim.
- **Statefulness is contained, not eliminated.** A single OTS leaf-index reuse destroys all security of that key (SP 800-208 "catastrophic and unrecoverable"); the split keeps the hazard host-side and out of the stateless kernel, but the host counter's crash-durability/atomicity is a new correctness obligation (`assumptions.md`), the crash-ordering window closed at stage A by single-signature-per-invocation (§3). SPHINCS+ would avoid it entirely — at prohibitive kernel-verify cost.
- **The Kani proofs cover a `w=1` toy instance, not the full parameters.** Totality/determinism/tamper-sensitivity (via the pinned-vector iff) are proven on `p≈2,h≈1,w=1`; full W4/H10 correctness rests on the host-test + the in-boot small-parameter RFC 8554 KAT + Miri. This re-scope is structural (the per-compression wall + the symbolic-coefficient #49 trap), named, and is the milestone's central reviewer-negotiation point.
- **The SHA-256 leaf's CBMC per-compression rate is UNMEASURED** and is the second make-or-break number — benchmarked on a one-block harness before any count is pinned; if 1.5–2× BLAKE2s it forces `kani_sha256_kat` host-side and likely confirms the 3-shard split (§4, §9).
- **The persisted-head record is a NEW multi-sector codec**, not a `blkfmt` reuse — cross-sector torn-write detection + atomic multi-sector commit + fail-closed partial-record decode are new surface with their own harness (§6).
- **`measure=SELF-NO-HW-ROOT selfmeasure=UNATTESTED-LOADER`** — the attestation carries only image-digest (self-reported by the attested image, no external measurer) + build-id + the sovereignty ledger; no measured boot, no RTM, no hardware root (`assumptions.md` A4/A6); image-identity ≠ source-identity is not closed (A6). M33 = SLSA L1→L2; L3 is M36.
- **The attestation is DSSE-PAE-SHAPED, not a wire-compatible SLSA/in-toto JSON producer** — `attest=DSSE-PAE-SHAPED`; only byte-level PAE injectivity/roundtrip/fail-closed-decode is proven, never JSON-canonicalization security.
- **`sidechannel=NOT-CLAIMED`** — no constant-time/timing claim for the verify leaf (the kernel-verify path's data-dependent branches are not analyzed; verification is over public data, so the residual is informational, named not hidden).
- **Named-deferred, stated not built:** the #75 batch-Merkle producer, #85 real enrolment, the M35 monitor-custodied persistent leaf-index + cross-run persistent-disk CI lane, witness-quorum split-view detection, the D1 conformance fallback, hardware-token / HSM custody (rungs 3–4).
