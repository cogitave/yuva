## Goal
The L2.0 VMX subtree (vmxon/vmcs/controls/ept/world_switch/exit) is fully implemented and the cumulative chain boots skip-green everywhere, but the genuine VMLAUNCH of the 1-CPUID nested guest has NEVER executed because no local/hosted-CI substrate exposes nested VMX (QEMU TCG filters out CPUID.01H:ECX.vmx[bit5]; stock GitHub runners do not officially support nested virt). This task executes the REAL proof: on a nested-VMX-capable substrate (provided by task 37) the kernel takes the Proven path, the world switch runs a real VMXON->VMLAUNCH->VM-exit round-trip, VMREAD(exit-reason 0x4402) returns basic reason 10 (CPUID), and a NEW strict serial marker prints ONLY on that genuine Proven{exit_reason==10} path. Done-definition: a CI job running on the task-37 substrate boots the unmodified kernel ELF, the serial log contains the strict proof marker `L2.0: vmlaunch PROVEN exit_reason=10`, that job is a REQUIRED gate on the substrate runner (a VMfail or guest-state entry-failure surfaces as RED, never masked), while the existing `L2.0: vmxroot OK` skip-green discipline is untouched on TCG/hosted CI; plus the entry-failure diagnosis path (VMREAD 0x4400 + exit-reason bit-31 detection) is hardened so any real VMCS bug is reported, not silently turned into a wrong-reason exit.

## Approach
1. Step 1 (diagnosis primitive, exit.rs): add `pub(super) unsafe fn raw_exit_reason() -> u32` returning the FULL 32-bit VMCS field 0x4402 (do not mask), and `pub(super) unsafe fn vm_entry_failed() -> bool` = `(raw_exit_reason() >> 31) & 1 == 1`. Keep the existing `exit_reason()` (bits 15:0). This is required because VM-entry failures due to invalid GUEST-state / MSR-loading are delivered as VM EXITS (basic reason 33/34/41) with exit-reason bit 31 SET, NOT as a VMfail — so world_switch::launch() returns true even on those, and without bit-31 detection the proof would mis-report a guest-state failure as a 'caught exit'. Un-gate `exit_qualification()`/`exit_instruction_len()` for use in the diagnostic print.
2. Step 2 (VmxProof enrichment, lib.rs): extend the enum with `EntryFailureExit { basic_reason: u32 }` (the world switch returned an exit but bit 31 was set — basic_reason is 33 invalid-guest-state / 34 MSR-load / 41 machine-check). Keep `EntryFailed { vm_error }` for the VMfail (VMLAUNCH did not enter — error 4/7/8/26 from field 0x4400) and `Proven { exit_reason }` for a real entry+exit. Document each variant with its exact Intel SDM source.

Step 3 (selftest wiring, vmx/mod.rs): after `world_switch::launch()` returns true, branch on `exit::vm_entry_failed()`: if bit 31 set -> `VmxProof::EntryFailureExit { basic_reason: exit::exit_reason() }`; else -> `VmxProof::Proven { exit_reason: exit::exit_reason() }`. On the launch()==false (VMfail) path keep `EntryFailed { vm_error: exit::vm_instruction_error() }`. Leave guest payload as CPUID (0F A2) + HLT pad (F4) — CPUID is the SDM unconditional-exit instruction giving reason 10; no control bit needed.

## Decisions — no open questions (açık karar bırakma)
- NESTED-VMX SUBSTRATE (roadmap §11 open decision, resolved): the real proof runs ONLY on the task-37 substrate, a labeled runner with `kvm_intel nested=1` and a guest vCPU that exposes `vmx` (run QEMU `-accel kvm -cpu host`). Primary = a self-hosted bare-metal Intel runner (label `nested-vmx`) because GitHub docs say hosted nested virt is 'experimental, no guarantees'; documented fallback = a confirmed-nested Azure larger runner. NOT QEMU-TCG-VMX: TCG does not emulate VMX (the vmx CPUID bit is filtered out), so it is rejected outright as a substrate.
- GATING MODEL: two-tier markers. The cumulative-chain marker `L2.0: vmxroot OK` keeps its skip-green allow-skip on TCG/hosted CI (ci.yml/vmm-boot.yml unchanged behaviour). A NEW strict marker `L2.0: vmlaunch PROVEN exit_reason=10` prints ONLY on Proven{10} and is the REQUIRED assertion of the substrate-only proof job (l2-nested-vmx.yml, no longer continue-on-error). This satisfies 'replace the skip-green' without regressing any stock-CI lane.
- GUEST PAYLOAD: keep CPUID (0F A2) followed by a HLT pad (F4), exit_reason 10. CPUID is the SDM 'unconditional VM-exit' instruction and needs ZERO control-bit setup; VMCALL (18) is rejected for the strict gate (the tolerant `||18` acceptance is dropped from the proof assertion, though the kernel still treats 10 or 18 as a valid world-switch for the tolerant marker).
- ENTRY-FAILURE DISAMBIGUATION: a successful world_switch::launch() (returns true) is NOT sufficient evidence of a real entry — VMREAD exit-reason bit 31 must be checked, because invalid-guest-state (33) / MSR-load (34) / machine-check (41) VM-entry failures are delivered AS exits with bit 31 set. New `EntryFailureExit{basic_reason}` variant carries this; only bit-31-clear reason==10 counts as Proven.
- VMfail DIAGNOSIS: on VMLAUNCH VMfail (launch() returns false) always VMREAD VM-instruction-error 0x4400 and surface it (EntryFailed{vm_error}); the kernel prints it as a non-OK diagnostic line so it is RED, not masked. Expected codes: 4 (non-clear VMCS), 7 (invalid control field), 8 (invalid host-state field), 26 (events blocked by MOV SS).
- EPT MAPPING POLICY (roadmap §11, resolved): keep gpa==hpa identity over [0,1 GiB) with 2 MiB RWX/WB leaves (already implemented); the relocated-base alternative is deferred — identity is sufficient and the kernel boot map's PA==VA makes every frame_alloc PA directly usable.
- CODE LOCATION (roadmap §11, resolved): L2.0 stays a kernel-crate self-test over the tb-hal vmx/ subtree (no premature extraction to crates/tb-core); the tb-core split lands at L2.3/L2.4 when the device-model seam forms. No new crate is created by this task.
- MARKER FOR CUMULATIVE CHAIN: the strict PROVEN line is printed BEFORE `L2.0: vmxroot OK` on the Proven{10} path so a single boot of the unmodified ELF emits both — the cumulative grep and the strict proof grep are independent and non-conflicting.
- TIER-1 UNIT TESTS run on host triple via `cargo test -p tb-hal` (no build-std); only pure encoder functions are tested — the asm/global_asm! compiles but is never executed in tests. This is the always-on coverage that does not need the substrate.

## Risks & mitigations
- Substrate unavailability (the headline blocker): GitHub explicitly does not officially support nested virt on hosted runners, and Azure exposure of `vmx` to the runner vCPU is not guaranteed. Mitigation: hard-depend on task 37 to stand up a labeled self-hosted bare-metal Intel runner (kvm_intel nested=1); the proof job runs ONLY there. The stock-CI green chain never depends on it.
- Self-hosted runner security/availability: a self-hosted runner executing untrusted PR code is a supply-chain risk and may be offline. Mitigation: trigger the proof job on push-to-main + workflow_dispatch only (not on fork PRs), ephemeral/locked-down runner, and treat an offline runner as a queued-not-failed state rather than a false red.
- Real-silicon VM-entry failure on first launch despite correct-looking VMCS (e.g. a host-state canonical-base or TR-usability subtlety): Mitigation: the hardened 0x4400 + bit-31/reason-33 diagnosis pinpoints control(7)/host(8)/guest(33) class immediately; the Step-8 audit checklist maps each code to the exact field group to re-check.
- Guest-state entry failure silently mis-reported as a caught exit: without bit-31 detection, reason 33 would look like 'Proven{33}' and only the `==10` check would catch it (printing FAIL, not a clear diagnosis). Mitigation: the explicit EntryFailureExit variant + bit-31 read makes it an unambiguous diagnostic.
- Marker drift between the kernel print, run-x86_64-vmx-proof.sh, and l2-nested-vmx.yml: a typo'd string silently never matches and the gate is vacuously skipped. Mitigation: single exact string `L2.0: vmlaunch PROVEN exit_reason=10` referenced in all three; the proof script greps an anchored fixed string and exits non-zero on miss.
- `cargo test -p tb-hal` host-build friction (no_std crate test harness needs std for the test executable; arch-gated asm must still compile on the host x86_64 triple): Mitigation: only pure functions are under #[cfg(test)]; asm/global_asm! compiles unused; if the no_std test harness fights, fall back to a thin `crates/tb-hal/tests/` integration target that re-exports the pure encoders behind a `testable` cfg.
- Scope creep into L2.1+ (EPT-violation demand handling, full exit set): this task is ONLY the CPUID round-trip proof + diagnosis + gating. Mitigation: explicitly defer reasons 48/49/12/28/etc. to L2.1/L2.2; do not extend the exit demultiplexer here.

## Research notes (with sources)
- CPUID causes an UNCONDITIONAL VM exit in non-root operation -> basic exit reason 10; the guest stub 0F A2 (CPUID) is therefore the minimal payload needing zero control-bit setup. Source: Intel SDM Vol 3C, 25.1.3 'Instructions That Cause VM Exits Unconditionally' + Appendix C basic-exit-reason table (10=CPUID, 18=VMCALL). The existing write_guest_code already emits 0F A2 F4 (CPUID; HLT pad).
- The VMCS exit-reason field 0x4402 packs the basic reason in bits 15:0 and a 'VM-entry failure' flag in bit 31. Source: Intel SDM Vol 3C, 25.9.1 'Basic VM-Exit Information', Table 'Exit Reason'. The current exit::exit_reason() masks 0xFFFF and DROPS bit 31 — the proof must read bit 31 separately.
- VM-entry failures due to INVALID GUEST STATE / MSR LOADING / machine-check are delivered as VM EXITS (not VMfail), with basic exit reasons 33 (invalid guest state), 34 (MSR loading), 41 (machine-check during VM-entry) and exit-reason bit 31 SET. Source: Intel SDM Vol 3C, 26.7 'VM-Entry Failures During or After Loading Guest State' + Appendix C. Consequence: world_switch::launch() returns true on these, so a bit-31 check is mandatory to avoid a false Proven.
- VMfail on VMLAUNCH (CF or ZF set at the resume label) yields a VM-instruction error readable via VMREAD 0x4400: 4=VMLAUNCH with non-clear VMCS, 5=VMRESUME with non-launched VMCS, 7=VM entry with invalid control field(s), 8=VM entry with invalid host-state field(s), 26=VM entry with events blocked by MOV SS. Source: Intel SDM Vol 3C, 31.4 'VM Instruction Error Numbers', Table 31-1. The current EntryFailed{vm_error} path already reads 0x4400.
- GitHub-hosted runners do NOT officially support nested virtualization: 'While nested virtualization is technically possible while using runners, it is not officially supported. Any use of nested VMs is experimental and done at your own risk, we offer no guarantees regarding stability, performance, or compatibility.' Source: GitHub Docs, 'About GitHub-hosted runners' (fetched 2026-06-08). => the real VMLAUNCH proof cannot be a required gate on ubuntu-latest; it needs the task-37 substrate.
- QEMU TCG does not emulate VMX: the vmx CPUID bit is a KVM-passthrough feature (`-cpu host` / `-cpu host,vmx=off` per QEMU 'KVM CPU model configuration' doc), and under TCG QEMU warns 'TCG doesn't support requested feature: CPUID.01H:ECX.vmx [bit 5]' and clears it. Source: QEMU system/i386/cpu.html + this repo's own SOVEREIGNTY-L2-ROADMAP §8a observation (QEMU 6.2 Debian). => only real nested VMX under KVM is a viable substrate.
- Nested VMX requires the L0 host's `kvm_intel nested=1` AND the guest vCPU advertising vmx in CPUID; QEMU must run `-accel kvm -cpu host` (or a named Intel model with +vmx). The existing l2-nested-vmx.yml already probes `/sys/module/kvm_intel/parameters/nested` and `grep vmx /proc/cpuinfo` — the proof script reuses these as a hard precondition.
- The control-MSR ADJUST algorithm final=(desired|allowed0)&allowed1, using the TRUE_* capability MSRs when IA32_VMX_BASIC bit 55 is set, is mandatory and is the #1 cause of silent VM-entry failure if skipped. Source: Intel SDM Vol 3C, 24.6.1 + Appendix A.3/A.4. Already implemented in controls.rs::adjusted(); Tier-1 tests will lock its bit-math.
- Guest non-register-state checks for a successful entry: VMCS-link-pointer must be FFFFFFFF_FFFFFFFF when VMCS shadowing is off (SDM Vol 3C 26.3.1.5), guest TR must reference a USABLE busy 64-bit TSS access-rights type 11 for IA-32e entry (26.3.1.2), and EFER LME|LMA must agree with the IA-32e-mode-guest control (26.3.1.1). The existing program_vmcs already sets link=~0, TR AR=0x8B (type 11, present), GUEST_IA32_EFER=LME|LMA — i.e. the VMCS is already correct-looking; the task is to PROVE it on real silicon and diagnose if not.
- Host-state checks (SDM Vol 3C 26.2): host CS/SS/DS/ES/FS/GS/TR selectors must have RPL=TI=0, TR/CS must be non-null, FS/GS/GDTR/IDTR/TR bases canonical, and with host-address-space-size=1 CR4.PAE=1 + RIP canonical. program_vmcs builds these from the LIVE kernel context (sgdt/sidt/str + masked selectors) — the EntryFailed{8} branch of the Step-8 audit targets exactly this group.
