# L2.1 — EPT-violation demand handling: fault on an unmapped GPA, map a backing frame, resume the guest ("L2.1: ept OK")

## Goal
L2.0 proved the world switch by catching one CPUID exit; the EPT is a static 1 GiB identity map and the guest never resumes. L2.1 proves the R/W/X second-stage triple is THE isolation primitive that replaces /dev/kvm: a long-mode nested guest touches a guest-physical address the EPT does NOT cover, tb-hal catches VM-exit reason 48 (EPT violation), reads the faulting GPA (VMREAD 0x2400) + qualification (0x6400), demand-allocates and installs a 4 KiB EPT leaf for it, issues INVEPT, and VMRESUMEs WITHOUT advancing guest RIP so the faulting store re-executes and completes — then the guest reaches its final clean exit and the host confirms the guest's sentinel store actually landed in the demand-mapped frame. Done = on the nested-VMX lane the kernel prints the cumulative marker "L2.1: ept OK" after exactly one demand fault with the sentinel verified; on no-VMX substrates it prints the same marker tagged as a graceful skip; all new silicon-unsafe stays confined to crates/tb-hal/src/arch/x86_64/vmx and the kernel crate stays unsafe-free.

## Decisions — no open questions (açık karar bırakma)
- EPT hole source: EXTEND guest paging to [0,2 GiB) and keep the EPT at [0,1 GiB); demand GPA = 0x4000_0000 (exactly 1 GiB). Rejected punching a 2 MiB hole inside the [0,1 GiB) identity EPT — it could unmap a live frame_alloc'd code/table frame that happens to fall in that 2 MiB; the 2 GiB approach puts the fault strictly ABOVE all low-RAM live frames with zero collision risk. (Resolves roadmap §11 'EPT mapping policy' for this test.)
- Mapping granularity: demand-map at 4 KiB (a fresh EPT PT + one 4 KiB data frame), not 2 MiB — the M6 allocator yields only 4 KiB frames and 4 KiB is true demand paging. Leaf = hpa|RWX|WB(6), no PS bit at PT level.
- GPA->HPA for the demand page is RELOCATED (GPA 0x4000_0000 -> an arbitrary frame_alloc HPA), not identity; legal because the demand region holds no page-table/code frames that assume GPA==HPA. The L2.0 [0,1 GiB) identity map (which code + page tables DO rely on) is untouched.
- World switch becomes re-entrant with a per-vCPU GuestRegs save area — this is LOAD-BEARING, not gold-plating: the faulting `mov [gpa],rax` re-executes after the demand-map, so guest RAX (=SENTINEL) MUST survive the exit/handle/resume cycle, and the CPU does not save guest GPRs. Unify L2.0's launch() onto the new run() (one implementation) with a regression gate on `^L2.0: vmxroot OK$`.
- RIP-advance: NEVER advance guest RIP on reason 48 (re-execute the faulting store); trapping-instruction RIP-advance is deferred to L2.2. The sentinel check is sensitive to this exact bug (an erroneous advance skips the store -> sentinel_ok=false -> red).
- INVEPT type: prefer single-context (type 1, EPTP[51:12]) when IA32_VMX_EPT_VPID_CAP bit 25 set, else global (type 2) when bit 26 set, else no-op. Honest note: a not-present->present demand transition does not architecturally require a flush, but we issue one per the roadmap mandate and to stay correct for future present->present upgrades.
- Proof shape: BOTH transparent resume (the guest reaches its final clean exit reason 10 after exactly one demand fault) AND host verification that the SENTINEL store landed in the backing frame — upgrading the proof from 'didn't crash' to 'the EPT actually routed the write to our frame'.
- EPT-misconfig (reason 49) and any unexpected exit reason are FATAL — surfaced honestly as a diagnostic line with no `ept OK` substring (red), per the roadmap.
- Facade shape: a SEPARATE `ept_selftest()->EptProof`, not an overload of VmxProof — keeps L2.0's vmx_selftest() byte-unchanged/green and gives L2.1 its own independently-greppable cumulative marker; shared bring-up stays DRY via the refactored internal helpers (enable_and_vmxon, clear_and_load, controls::adjusted, program_vmcs).
- Skip discipline: identical to L2.0 — no-VMX (TCG qemu64) prints `L2.1: ept OK (vmx unavailable, skipped)` (substring still matches the runner grep), aarch64 prints the n/a variant; the real demand-map path fires only on a `-cpu host,+vmx` + kvm_intel nested=1 lane.
- Constants: SENTINEL=0xCAFE_F00D_DEAD_BEEF, DEMAND_GPA=0x4000_0000 (4 KiB-aligned so the store lands at backing-frame offset 0; 1 GiB is the first GPA outside the EPT and inside the 2 GiB guest map). Guest store encoding = absolute `mov [moffs64],RAX` (48 A3 + 8-byte GPA): no extra GPR setup, no stack use, no segment-limit checks in flat long mode.
- Loop-cap: bound the demand loop at 4 EPT violations; exceeding it returns UnexpectedExit (a forgotten INVEPT or a wrong RIP advance would otherwise re-fault forever).

## Risks & mitigations
- Forgetting guest-GPR save/restore -> the re-executed store writes a clobbered RAX and sentinel_ok=false. Mitigation: the GuestRegs save area in run() + the sentinel check, which is sensitive to exactly this bug.
- Advancing guest RIP on the EPT violation -> the store is skipped (wrong value or an infinite loop). Mitigation: the explicit 'no RIP advance on reason 48' rule + the 4-fault loop cap + the sentinel check.
- Missing/incorrect INVEPT -> a stale not-present translation re-faults non-deterministically. Mitigation: single-context-with-global-fallback invept() after every EPT edit; the loop cap catches a re-fault storm.
- EPT-misconfiguration (reason 49) from a malformed entry (reserved bits, memory-type not in {0,1,4,5,6}, PS at the wrong level, or RWX=000 with a non-zero address). Mitigation: the pure encoders (ept_leaf_4k drops PS at PT level; ept_nonleaf=child|RWX) + their unit tests; treat 49 as fatal/red.
- Guest #PF instead of an EPT violation if guest paging does not map the demand linear address. Mitigation: extend guest paging to 2 GiB so linear 0x4000_0000 translates in GUEST paging and the fault is forced into the EPT stage — verified via qualification bit 7 (GLA-valid) + bit 8 (final translation).
- Frame leak on an early-return error path (OOM mid-build). Mitigation: allocate-then-track in DemandFrames and free everything (incl. df) in a single teardown that runs on every exit path, mirroring L2.0's staged frame_free unwind.
- Regressing L2.0 by unifying the world switch. Mitigation: keep launch()'s signature/semantics, assert `^L2.0: vmxroot OK$` on the nested lane and the TCG skip line.
- No nested VMX on hosted CI -> the real demand path is unexercised locally. Accepted + mitigated by the allow-skip discipline; Tier-1 unit tests + the gated nested lane cover it, and the two prerequisites (backlog 2 = nested-VMX substrate, 6 = L2.0 real proof) gate this task.

## Research notes (with sources)
- EPT-violation basic exit reason = 48; EPT-misconfiguration = 49 (Intel SDM Vol 3C, Appendix C 'VM Exit Reasons' / Table C-1; corroborated by the existing crates/tb-hal/src/arch/x86_64/vmx/exit.rs comment).
- The faulting guest-physical address is VMCS field GUEST_PHYSICAL_ADDRESS=0x2400; exit qualification=0x6400; guest-linear address=0x640A (Intel SDM Vol 3C Appendix B read-only data fields; matches the constants already in exit.rs).
- EPT-violation exit qualification bits: 0=data read, 1=data write, 2=instruction fetch, 3=GPA readable (EPT R), 4=GPA writable (EPT W), 5=GPA executable (EPT X), 7=guest-linear-address valid, 8 (when bit7=1)=access to the final GPA translation of a linear address (vs 0=a guest paging-structure access), 12=NMI unblocking due to IRET (Intel SDM Vol 3C §29.3.3.2 'EPT Violations', 'Exit Qualification for EPT Violations').
- EPT paging-structure entries: bit0 R, bit1 W, bit2 X; non-leaf entries reference the next table via bits 51:12 with NO memory type; leaf entries (1 GiB PDPTE / 2 MiB PDE / 4 KiB PTE) carry bits5:3 = EPT memory type (0 UC,1 WC,4 WT,5 WP,6 WB), bit6 ignore-PAT, bit7 = 'maps a page' (PS) for the PDPTE/PDE leaves; the 4 KiB PTE always maps a page (no PS bit). An entry is 'present' iff any of bits2:0 is set; all-zero -> EPT violation (Intel SDM Vol 3C §29.3.2 / Tables 28-1..28-6; consistent with ept.rs `EPT_RWX|(6<<3)|(1<<7)` for the 2 MiB leaf — the new 4 KiB leaf drops bit7).
- EPTP (VMCS 0x201A): bits2:0 = EPT memory type (6=WB), bits5:3 = EPT page-walk-length minus 1 (=3 for 4-level), bit6 = enable accessed/dirty, bits 51:12 = EPT PML4 base (Intel SDM Vol 3C §24.6.11 / §29.3.2; matches `pml4 | 6 | (3<<3)` in ept.rs).
- INVEPT: type 1 = single-context (invalidate mappings associated with EPTP[51:12]); type 2 = global (all EPTPs); supported types reported in IA32_VMX_EPT_VPID_CAP (MSR 0x48C) — INVEPT bit 20, single-context bit 25, global bit 26; descriptor is 128-bit {EPTP in [63:0], [127:64] must be zero} (Intel SDM Vol 3C 'INVEPT', via felixcloutier.com/x86/invept fetched 2026-06-08; matches the existing invept_single_context cap-gating on bits 20 & 25).
- On a VM exit the CPU loads guest RIP/RSP/RFLAGS from the VMCS but does NOT save/restore guest GPRs (RAX..R15), and RFLAGS is set to 0x00000002 on a clean exit (Intel SDM Vol 3C §27.5.1) — so the launch stub must itself save/restore guest GPRs across exit/resume, and CF|ZF at the post-vmlaunch label distinguishes VMfail from a clean exit (matches the world_switch.rs L2.0 design).
- KVM's __vmx_vcpu_run (arch/x86/kvm/vmx/vmenter.S) is the canonical re-entrant world switch: it cmp's the launched flag BEFORE loading guest GPRs, stashes the regs pointer on the host stack, runs VMLAUNCH/VMRESUME, and on exit reloads the regs pointer and saves guest GPRs — the exact pattern adopted for __tabos_vmx_run.
- EPT violations are fault-like: the faulting instruction is restarted after the violation is resolved, so guest RIP must not be advanced (Intel SDM Vol 3C §29.3.3.2; the roadmap's RIP-advance discipline, which exit.rs already documents).
- The TABOS frame allocator hands out arbitrary 4 KiB physical frames (tb_hal::frame_alloc()->Option<u64>, O(1) free-stack pop) and the kernel boot map identity-maps [0,1 GiB) PA==VA, so each frame's PA is directly writable AND usable as the physical address VMX/EPT instructions demand (crates/tb-hal/src/lib.rs pmm facade; ept.rs header).
- QEMU 6.2 TCG refuses CPUID.01H:ECX.VMX even under -cpu max, so the demand-map path is exercised only on a KVM -cpu host,+vmx + kvm_intel nested=1 lane; hosted GitHub runners do not guarantee nested VMX -> the allow-skip discipline (this repo's observed local behaviour, recorded in SOVEREIGNTY-L2-ROADMAP §8a + l2-nested-vmx.yml).
- Current cumulative final marker is 'L2.0: vmxroot OK' across scripts/run-x86_64.sh (MARKER), scripts/run-aarch64.sh (MARKER), and scripts/bench-boot.sh (FINAL_MARKER); ci.yml invokes both run scripts and they substring-grep the marker — so L2.1 must bump all three plus add an L2.1 verdict to l2-nested-vmx.yml.
