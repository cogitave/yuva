## Executive summary
The forward plan splits into four tracks, only one of which is genuinely blocked. The host-provable assurance track (38 M11 caps Kani, 49 encoder Kani, 39 Miri Tier-0), the M14 IPC completion track (50 byte-payload, 51 blocking-recv), and the ARM parity track (48 EL2 nVHE smoke, 47 aarch64 tb-boot producer) are all locally doable today on stock CI with no nested-VMX dependency. Start with 38: it both stands up the Kani CI lane that 49 reuses and machine-proves the capability attenuation invariant that M18's entire frozen-boundary guarantee documentedly reduces to. Land 39 early too, because it creates crates/tb-core for the Miri Tier-0 surface and that crate name collides head-on with the L2.3 orchestrator crate (44) — settling ownership now avoids a painful rebase later. 48 is the standout: it is the only L2.0-class 'we are the hypervisor' proof that actually EXECUTES on a stock hosted runner (QEMU virt,virtualization=on under pure TCG), so ARM reaches real L2 parity while x86 is still waiting on hardware. 49 is the immediate fast-follow once 38's Kani lane exists.

The x86 L2 chain (41 L2.0 VMLAUNCH -> 42 L2.1 EPT -> 43 L2.2 exit-set -> 44 L2.3 devmodel/tb-core split) is the critical path and the honest blocker: every one of its DoD markers is a genuine-proof gate that requires a nested-VMX substrate — the missing task 37 / backlog #2 — which hosted GitHub runners do not provide and QEMU-TCG filters out. The code, the bit-31 entry-failure diagnosis, the Tier-1 host encoder tests, and the TCG skip-green paths for all four can be written and merged now, but the markers cannot turn truly green until a self-hosted bare-metal Intel runner (kvm_intel nested=1, -cpu host) or a confirmed-nested Azure larger runner is stood up. Standing up 37 is therefore the single highest-leverage unblock in the roadmap. The other cross-cutting risk is crate thrash: 38, 39, 44 and 49 each carve a pure forbid(unsafe) leaf out of tb-hal with overlapping modules (caps in 38/39; mmu/paging in 39/49) and a hard tb-core name collision (39's zero-dep leaf vs 44's tb-hal-dependent orchestrator) — these must land under one coordinated extraction design, and the shared vmx/world_switch.rs must evolve strictly 41->42->43->44.

## Cross-cutting concerns
- NESTED-VMX SUBSTRATE (task 37 / backlog #2) is the single external blocker: hosted GitHub CI cannot run nested VMX and QEMU-TCG filters out CPUID.01H:ECX.vmx[bit5], so the REAL DoD markers of 41/42/43/44 cannot turn green until a self-hosted bare-metal Intel runner (kvm_intel nested=1, -cpu host) or a confirmed-nested Azure larger runner exists. Until then those four land only as skip-green markers + Tier-1 host unit tests.
- OVERLAPPING PURE-LEAF CRATE EXTRACTIONS FROM tb-hal need ONE coordinated design: 38 carves out tb-caps-core (Rights+CapTable), 39 carves out tb-core (caps+ipc+mem+infer+mmu+blocks for the Miri surface), 49 carves out tb-encode (VMX/paging/IPC-frame encoders). caps overlaps between 38 and 39; mmu/paging overlaps between 39 and 49. Land a single extraction plan (e.g. tb-core re-exports tb-caps-core and tb-encode) or they will conflict on tb-hal/src/{caps,mmu}.rs.
- crates/tb-core NAME COLLISION: 39 defines tb-core as a ZERO-dependency pure leaf so Miri can host-run it; 44 defines tb-core as the L2 orchestrator that DEPENDS ON tb-hal. A tb-hal-dependent tb-core breaks 39's Miri host build (asm/unsafe enters the interpreter). Resolve by renaming one (e.g. 44's orchestrator -> tb-hyper / tb-vmx-core) or by layering a pure tb-core leaf under a separate orchestrator crate.
- SHARED KANI CI LANE + MARKER NAMESPACE: 38 establishes kani.yml + the per-package (-p, never --workspace, to keep tb-hal asm out of CBMC) invocation + EXPECTED_HARNESSES fail-closed; 49 extends the same lane. V-namespace coordinated: V0 reserved for 38 ('M11: caps-subset PROVEN'), V1 for 49 ('V1: kani-encoders OK').
- SHARED tb-hal/arch/x86_64/vmx/world_switch.rs MUST EVOLVE STRICTLY IN ORDER: 41 (single-shot VMLAUNCH) -> 42 (re-entrant + GuestRegs save area) -> 43 (full GPR file + VMLAUNCH/VMRESUME run loop) -> 44 (consumes GuestRegs via the typed safe facade). One file, four sequential owners; 49 simultaneously re-homes the VMX encoder algebra out of the same subtree.
- CUMULATIVE SERIAL-MARKER CHAIN + RUN SCRIPTS (run-x86_64.sh / run-aarch64.sh / bench-boot.sh / l2-nested-vmx.yml): each L2 rung bumps the final MARKER (L2.0->L2.1->L2.2->L2.3) and adds a verdict block; sequential ownership. The two-tier discipline (skip-green on TCG/hosted vs strict PROVEN on the substrate lane) is shared across 41/42/43/44/48.
- MARKER COLLISION: 50 and 51 BOTH emit the 'M14.1' prefix ('M14.1: payload OK' vs 'M14.1: blocking-recv OK'). Disambiguate (e.g. M14.1 payload / M14.2 blocking-recv) and coordinate their shared edits to ipc.rs/caps.rs/lib.rs/kernel main.rs.
- FRAMEKERNEL INVARIANT (all unsafe/asm confined to tb-hal; #![forbid(unsafe_code)] everywhere else) is enforced by a shared CI grep guard touched by 38/39/44/47/48/50/51 — every new unsafe must be justified and confined.
