# L2.3 — device-model seam + extract crates/tb-core (the trusted-core split)

> **Task #44** · DoD marker: `L2.3: devmodel OK` · Effort: **XL** · Slug: `l2.3-devmodel-tb-core-crate`
> **Depends on:** #8, #7, #6, #2

> Auto-generated from the `tabos-forward-plans` ultracode planning workflow (research-backed, decision-closed). Treat as the working spec for this task; refine in place as implementation reveals detail.

## Goal
Create the new no_std `#![forbid(unsafe_code)]` crate `crates/tb-core` as the safe orchestrator of the from-scratch Type-1 microhypervisor, mirroring how `kernel` stays safe over `tb-hal` today, and land the device-model seam: a guest I/O-exit (Intel basic exit reason 30, COM1 0x3f8) is decoded by tb-core and forwarded across a `DeviceModel` trait boundary to a deprivileged, byte-faithful 16550 backend that prints. This is the extraction point the L2 roadmap reserves: the VMX policy/orchestration that currently lives inline in `tb-hal::vmx_selftest` and `kernel/src/main.rs` moves into tb-core, while ALL silicon-unsafe (VMREAD/VMWRITE/VMLAUNCH/VMRESUME/INVEPT, the world-switch asm, raw EPT/VMCS frame writes) STAYS confined to `crates/tb-hal/src/arch/x86_64/vmx/`. Done = both arches build with the new member and boot green under QEMU-TCG printing the cumulative chain ending in `L2.3: devmodel OK` (graceful device-model-only self-test on the no-VMX path), tb-core's device-model + I/O-decode unit tests pass on the host (`cargo test -p tb-core`), and on the nested-VMX lane a real nested guest's COM1 writes round-trip through the trait to the console before the bare `L2.3: devmodel OK` prints. tb-core contains zero `unsafe`/`asm!` (a compile-enforced framekernel invariant), so the privileged TCB does not grow by one byte — only safe orchestration relocates.

## Approach
1. Scaffold crates/tb-core: new Cargo.toml (publish=false, edition 2021, deps: tb-hal path, tb-boot path; uses crate-wide `alloc`); src/lib.rs with `#![cfg_attr(not(test), no_std)]` + `#![forbid(unsafe_code)]` + `#![deny(missing_docs)]` (exact tb-boot pattern, which gives BOTH the -Zbuild-std no_std kernel build AND host `cargo test`). Add "crates/tb-core" to the root Cargo.toml [workspace] members (it is a root member, NOT a nested workspace — it shares the no_std/build-std/target posture of kernel+tb-hal; only the std tb-vmm is nested).
2. Refactor tb-hal's VMX subtree from a monolithic self-test into a typed SAFE facade in a new crates/tb-hal/src/arch/x86_64/vmx/facade.rs, re-exported from tb-hal/src/lib.rs. The seam carries DATA, never raw pointers: expose `VmxRoot` (enable()->VMXON+CR4.VMXE+CR0/CR4 fixed-MSR clamp; Drop=VMXOFF+restore), `Vmcs` (create(&VmxRoot, frame_pa)->VMCLEAR+VMPTRLD; read(VmcsField)->u64 / write(VmcsField,u64) safe wrappers over the existing unsafe vmread/vmwrite), `EptSpace` (new()/map_2mib(gpa,hpa,EptPerms)/map_4kib/unmap/eptp()/invept() — tb-core decides mappings, tb-hal does the unsafe frame writes), POD `GuestState`/`HostState`/`Controls` (plain `#[repr(C)]` data tb-core fills; tb-hal's `load_guest`/`load_host`/`load_controls` do the VMWRITEs), and `Vmcs::run(&mut GuestRegs)->RawExit` folding world_switch.rs+exit.rs into one safe call that returns `RawExit{reason,qualification,instr_len,gpa}`. All bodies stay unsafe-internally; signatures are safe — identical posture to today's serial_write_byte/vmx_selftest.
3. Extend crates/tb-hal/src/arch/x86_64/vmx/world_switch.rs to marshal the FULL guest GPR file (not just host callee-saved): load guest RAX..R15 from a per-vCPU `GuestRegs` save area before VMLAUNCH/VMRESUME, store them back on exit. This is L2.2's 'hidden GPR save/restore' (the CPU does not preserve RAX..R15 across the switch); it is the prerequisite that lets tb-core read/write guest RAX for I/O IN/OUT. Add a VMRESUME path (re-entry after a serviced exit) alongside the existing first-entry VMLAUNCH, distinguished by a launched flag. Promote exit.rs's currently-`#[allow(dead_code)]` qualification/instr_len/gpa readers into the `RawExit` struct.
4. Define the device-model seam in crates/tb-core/src/devmodel/mod.rs: `pub trait DeviceModel { fn pio_read(&mut self,port:u16,&mut[u8]); fn pio_write(&mut self,port:u16,&[u8]); fn mmio_read(&mut self,addr:u64,&mut[u8]); fn mmio_write(&mut self,addr:u64,&[u8]); }` (mutable-self, matching rust-vmm's MutDevicePio/MutDeviceMmio and tb-vmm's existing BusDevice). Port tb-vmm/src/device.rs into devmodel/bus.rs as a no_std `Bus` (BTreeMap<u64,Entry> from `alloc`, dispatch by largest base<=addr, open-bus 0xFF on unclaimed read, overlap/zero-length rejection) — a near-verbatim std->no_std port keeping its 4 existing unit tests.
5. Port tb-vmm/src/serial.rs into crates/tb-core/src/devmodel/serial.rs as a no_std `Serial16550` implementing `DeviceModel`: keep the exact register map (THR/RBR/IER/IIR-FCR/LCR-DLAB/MCR/LSR-THRE/MSR/SCR) and DLAB/THRE semantics matched to the kernel's polled TX driver, but replace `Box<dyn Write+Send>` with a `pub trait SerialSink { fn emit(&mut self,b:u8); }` (the L2.3 sink is the kernel console via a thin wrapper over tb_hal::serial_write_byte; later it becomes a cross-domain ring) and use `alloc::collections::VecDeque` for the input queue. Carry over its 3 unit tests (LSR-always-THRE, divisor-latch-not-printed, input-DR-drains).
6. Define the tb-core<->device-model exit-upcall + vcpu_run ABI in crates/tb-core/src/exit.rs + vcpu.rs. `enum VmExit { Io{port,write,size}, Mmio{addr,write,size}, Hlt, Cpuid, Hypercall{nr,args}, EptViolation{gpa,read,write,exec}, Fatal{kind}, ... }` classified from RawExit.reason (30=I/O, 48=EPT-violation, 49=EPT-misconfig=Fatal, 2=triple-fault=Fatal, 10=CPUID, 12=HLT, 18=VMCALL, 31/32=RDMSR/WRMSR, 52=preemption-timer). `Vcpu` wraps tb_hal::{Vmcs,EptSpace,GuestRegs}; `Vcpu::run(&mut self, dm:&mut dyn DeviceModel)->VmExit` loops: hal Vmcs::run(&mut regs) -> classify -> for I/O bind guest RAX low `size` bytes and call dm.pio_read/write (write-back RAX on IN) -> advance guest RIP by instr_len (the RIP-advance discipline: advance after trapping-instruction exits, NEVER after EPT-violations/exceptions) -> VMRESUME; terminal/HLT/Fatal/unmapped-MMIO surface as the returned VmExit. hostcall.rs holds the capability/grant-oriented hypercall surface types (vm_create/vcpu_create/vcpu_run/mem_grant/irq_inject/iommu_map) as the L2.9-forward ABI skeleton, with vcpu_run wired now.

## Decisions — no open questions (açık karar bırakma)
- Crate placement: `crates/tb-core` is a ROOT-workspace member (added to the existing [workspace].members alongside crates/tb-boot, crates/tb-hal, kernel), NOT its own nested workspace. Why: it is no_std bare-metal built with the same -Zbuild-std/custom-target machinery as kernel+tb-hal and the kernel links it; nested workspaces exist ONLY to isolate the std tb-vmm's Cargo.lock/build-std. The roadmap §10 left this open ('root-workspace member or its own no_std workspace') — closed to root member.
- Dependency direction (acyclic): kernel -> tb-core -> tb-hal -> tb-boot, with tb-core also -> tb-boot + crate-wide `alloc`. tb-core is the safe orchestrator; tb-hal is the unsafe machine. No reverse edge (tb-hal never names tb-core).
- The tb-core<->tb-hal seam carries DATA, not pointers. tb-hal exposes typed handles (VmxRoot/Vmcs/EptSpace/GuestRegs/RawExit) + POD GuestState/HostState/Controls; tb-core composes policy and fills the PODs. EVERY VMREAD/VMWRITE/VMLAUNCH/VMRESUME/INVEPT, the world-switch global_asm!, and all raw VMCS/EPT frame writes stay inside crates/tb-hal/src/arch/x86_64/vmx/. tb-core does write_volatile to nothing — it cannot (forbid(unsafe_code)).
- DeviceModel trait shape: four mutable-self methods pio_read/pio_write/mmio_read/mmio_write, plus a `Bus` address-range registry and a `Serial16550` device behind a `SerialSink` output trait. This matches rust-vmm vm-device's MutDevicePio/MutDeviceMmio (mutable variant chosen because the UART mutates LCR/DLAB/IER state) and is a direct no_std port of tb-vmm's already-Firecracker-shaped BusDevice/Bus — so the seam is proven, not invented.
- Exit-upcall / vcpu_run ABI: `Vcpu::run(&mut self, dm:&mut dyn DeviceModel) -> VmExit`. tb-core's internal loop services EPT-violation(48 -> map+INVEPT+resume), CPUID(10 -> forge+advance), RDMSR/WRMSR(31/32) itself and forwards only I/O(30) and unmapped-MMIO(48) to the DeviceModel, surfacing HLT/Hypercall/Fatal as the returned VmExit (the 'vcpu_run-returns-exit-reason' contract). Guest RAX low bytes bind the I/O data; RIP advances by instr_len after trapping exits and NEVER after faults.
- What stays vs moves: the std L1 tb-vmm (device.rs/serial.rs/loader.rs/memory.rs/vmm.rs) is UNTOUCHED at L2.3 — it remains the L1 /dev/kvm path. tb-core gets independent no_std twins of device.rs+serial.rs (the Bus + Serial16550). Convergence/demotion of the single tb-vmm device model is L2.9 (backlog #16), explicitly out of scope here; the trait boundary is what makes that later swap a drop-in.
- forbid(unsafe) mapping: tb-core is `#![forbid(unsafe_code)]` exactly like `kernel`, adds ZERO unsafe, and the device model is pure safe register/data emulation. The privileged TCB does not grow — only safe orchestration relocates out of tb-hal/kernel. The invariant is enforced by the crate attribute (compile error on any unsafe) AND a CI grep guard over crates/tb-core.
- Graceful-skip discipline: where VMX is unavailable (QEMU-TCG qemu64 locally and on hosted CI, and on aarch64), tb-core runs a host-side device-model self-test driving the identical Bus+Serial16550 path and prints the SAME `L2.3: devmodel OK` marker substring (tagged), so the required ci.yml TCG lanes stay green on both arches while the new trait/Bus/Serial code is still exercised every boot. Mirrors L2.0's `vmx unavailable, skipped` substring.
- L2.3 guest program: a hand-assembled long-mode COM1-writer stub (poll LSR@0x3fd THRE bit5, OUT 0x3f8) emitting a fixed ASCII probe string then HLT, placed in the EPT/CR3 identity-mapped guest code page — a natural extension of L2.0's single-CPUID write_guest_code, requiring no new platform stack.
- Marker-rendering ownership: tb-core returns an `L2Proof` enum and the KERNEL prints the marker strings (single serial-marker authority), preserving today's VmxProof pattern so the cumulative-boot self-test discipline is unchanged.
- Host-test enablement: tb-core uses `#![cfg_attr(not(test), no_std)]` (the exact tb-boot idiom) so Bus/Serial16550/exit-qualification-decoder get always-on `cargo test -p tb-core` coverage on any runner (Tier-1), independent of the nested-VMX hardware lane.

## Risks & mitigations
- The world-switch must now marshal the full guest GPR file (guest RAX is the I/O datum for IN/OUT) whereas L2.0 only preserved host callee-saved registers around a single CPUID. Mitigation: this guest-GPR save/restore IS L2.2's 'hidden GPR save/restore' built in world_switch.rs before this task; tb-core's run() consumes a `GuestRegs` value. Hard dependency on backlog #8 (L2.2).
- Refactoring tb-hal's monolithic, green `vmx_selftest` into a typed VmxRoot/Vmcs/EptSpace facade risks regressing the L2.0 boot proof. Mitigation: the existing `L2.0: vmxroot OK` marker assertion in ci.yml/run-x86_64.sh is the regression guard; route L2.0 through the SAME tb-core orchestrator so one code path covers both, and keep the refactor mechanical (same VMCS field encodings, same control-MSR adjust algorithm).
- VMREAD/VMWRITE implicitly target the CURRENT (loaded) VMCS — global CPU state — so a safe `Vmcs::read/write` wrapper is only sound while that handle is the loaded one. Mitigation: tb-hal's `Vmcs` is a non-Clone, !Send, !Sync handle whose create() VMPTRLDs it; at L2.3 there is exactly one vCPU so only one is ever current; document the single-current-VMCS invariant and gate it per-pCPU at L2.6 SMP. This soundness boundary is precisely why the unsafe stays in tb-hal, not tb-core.
- no_std port of Serial16550 loses `Box<dyn Write+Send>` and `std::collections::VecDeque`. Mitigation: replace the sink with a `SerialSink` trait (zero-alloc output) and use `alloc::collections::VecDeque` (alloc is online crate-wide since M5); the port is pure safe data with no behavioural change — its 3 ported unit tests prove byte-faithfulness.
- 'Deprivileged' is conceptual at L2.3: the device model is an in-tree call at the same privilege, not yet a separate domain. Mitigation: do not over-claim isolation now — the DeviceModel trait + Bus boundary is the seam that makes the L2.9 cross-domain (shared-ring/VMCALL) move a drop-in; the honest claim at L2.3 is 'the seam exists and is byte-faithful', per the NOVA split where device emulation is deprivileged but the trait shape is fixed first.
- The marker must print on the no-VMX path or the required TCG lanes (and aarch64) go red. Mitigation: the device-model-only self-test path drives the identical Bus+Serial path and prints the same `L2.3: devmodel OK` substring; verified by the existing grep mechanism that already accepts L2.0's tagged skip substring.
- Adding a 4th workspace member + a kernel dep can perturb -Zbuild-std/feature unification or the deny(missing_docs) gate. Mitigation: tb-core mirrors tb-boot's proven Cargo/lib layout (`cfg_attr(not(test), no_std)`, empty-ish deps), and CI builds both arches on every push — a unification break surfaces immediately as a build failure, the arbiter.

## Research notes (with sources)
- rust-vmm vm-device is built on FOUR device traits — DevicePio/MutDevicePio (PIO) and DeviceMmio/MutDeviceMmio (MMIO) — dispatched by an IoManager holding one PIO bus + one MMIO bus that routes a request to the device whose registered address range contains the access; the PioManager/MmioManager traits exist explicitly 'to allow other crates [to] depend on the traits without depending on the actual implementation.' This directly validates tb-core's DeviceModel-trait-as-boundary design and the trait-not-impl dependency edge. (Source: github.com/rust-vmm/vm-device README, fetched 2026-06-08.)
- The repo's own tb-vmm/src/device.rs is ALREADY 'Modelled on the rust-vmm / Firecracker Bus pattern (a sorted map from base address to device, dispatched by the largest base <= addr)' and tb-vmm/src/serial.rs is a complete 16550A matched byte-for-byte to the kernel's polled TX driver (THRE-always-set, DLAB-honoured so the init divisor write is not printed). The L2.3 task is therefore a std->no_std PORT of proven code, not a green-field design. (Source: tb-vmm/src/device.rs, tb-vmm/src/serial.rs.)
- NOVA microhypervisor (the canonical split-VMM / trusted-core precedent): privileged microhypervisor = 9 KLOC, thin user-level environment = 7 KLOC, user-level VMM = 20 KLOC; crucially device emulation AND the instruction emulator live in the DEPRIVILEGED user-level VMM, NOT in the microhypervisor — 'at user level, we trade improved security and lower interface [complexity]'. tb-core mirrors this exactly: privileged VMX-root core owns world-switch/EPT/exit-demux; the device model is the deprivileged component behind the trait. (Source: Steinberg & Kauer, 'NOVA: A Microhypervisor-Based Secure Virtualization Architecture', EuroSys 2010 — nova.txt lines 139-156.)
- The TABOS L2 plan already DECIDES the split-VMM shape and the <~10K-LOC privileged-TCB budget (seL4/NOVA-class), with the explicit rule that exceeding it is 'proof the split is wrong', and reserves any formal-verification budget for tb-core only. The roadmap also pre-decides that tb-core is 'a new no_std mostly-safe orchestrator crate that stays #![forbid(unsafe_code)], exactly as kernel stays safe over tb-hal today', and that L2.0-L2.6 should 'prototype the VMX primitives as a kernel milestone, extract into crates/tb-core at L2.3/L2.4 when the device-model seam forms' — i.e. THIS task is the reserved extraction point. (Source: docs/SOVEREIGNTY-L2-ROADMAP.md §1, §3, §11.)
- Intel SDM basic VM-exit reasons used by the seam: 30 = I/O instruction (the L2.3 COM1 forward), 48 = EPT violation, 49 = EPT misconfiguration (fatal), 2 = triple fault (fatal), 10 = CPUID, 12 = HLT, 18 = VMCALL, 31/32 = RDMSR/WRMSR, 52 = VMX-preemption timer. The I/O exit qualification encodes access size (bits 2:0), direction (bit 3, 0=OUT/1=IN), and port number (bits 31:16); VM_EXIT_INSTRUCTION_LEN drives the RIP advance. (Source: Intel SDM Vol 3C §27 exit reasons + §27.2.1 I/O qualification; already encoded in crates/tb-hal/src/arch/x86_64/vmx/exit.rs and roadmap §4.)
- The current tb-hal VMX subtree is 1312 LOC across 7 files (mod 497, vmxon 310, ept 163, vmcs 103, controls 93, world_switch 77, exit 69); the orchestration to extract (frame alloc/free dance, program_vmcs 60-field sequencing, the launch/catch flow) is the body of tb-hal::vmx_selftest in mod.rs, and exit.rs already has the qualification/instr_len/gpa readers parked under #[allow(dead_code)] 'consumed by the L2.1/L2.2 exit demultiplexer' — confirming the extraction is staged and low-risk. (Source: crates/tb-hal/src/arch/x86_64/vmx/*.rs.)
- The framekernel + cumulative-marker discipline is concrete and reusable: tb-hal is the sole unsafe crate and exposes a safe VmxProof facade the forbid(unsafe) kernel branches on; the kernel prints the markers; run-x86_64.sh greps a single MARKER substring; tb-boot proves the `#![cfg_attr(not(test), no_std)]` idiom that yields BOTH the -Zbuild-std no_std build AND host `cargo test`. tb-core inherits all four patterns verbatim. (Source: crates/tb-hal/src/lib.rs VmxProof, kernel/src/main.rs L2.0 block, scripts/run-x86_64.sh, crates/tb-boot/src/lib.rs.)
- Backlog row 9 (this task) is XL, directly depends on row 8 (L2.2 'exits OK'), which chains row 7 (L2.1 'ept OK') -> row 6 (L2.0 real VMLAUNCH proof) -> row 2 (F1, the nested-VMX validation substrate); the real device-model round-trip can only be PROVEN on that nested-VMX lane (hosted GitHub runners do not guarantee nested VMX), so the always-green path is the TCG device-model-only self-test. (Source: docs/BACKLOG.md rows 2,6,7,8,9; .github/workflows/l2-nested-vmx.yml.)
