# aarch64 tb-boot producer + `_tb_start` EL1 entry: boot the kernel under tb-vmm via an X0=TbBootInfo* register-file handoff (KVM_ARM)

> **Task #47** · DoD marker: `tb-boot: contract v0 OK` · Effort: **L** · Slug: `aarch64-tb-boot-producer`
> **Depends on:** none

> Auto-generated from the `tabos-forward-plans` ultracode planning workflow (research-backed, decision-closed). Treat as the working spec for this task; refine in place as implementation reveals detail.

## Goal
Today the aarch64 kernel has only the QEMU-`virt` boot path (`_start`, x0=FDT) and tb-vmm has no aarch64 backend, so the project cannot boot its own aarch64 kernel under its own VMM — breaking parity with the x86_64 L1 sovereignty rung and gating ARM-first L2/EL2 work. This task adds (1) a kernel-side `_tb_start` EL1 entry + a `.note.TABOS` ELF note (the tb-boot v0 *consumer*), and (2) a tb-vmm aarch64 *producer* that writes a `TbBootInfo` into guest RAM and programs the vCPU register file (KVM_ARM_VCPU_INIT + KVM_SET_ONE_REG: PSTATE=EL1h/DAIF-masked, PC=entry, X0=TbBootInfo*), plus the minimal aarch64 platform tb-vmm must provide (PL011 MMIO console, in-kernel vGIC, DRAM-based guest RAM). Done = on an aarch64 /dev/kvm host, `tb-vmm` boots the unmodified aarch64 kernel ELF (resolving the TABOS note, never `e_entry`) and the guest prints `tb-boot: contract v0 OK` followed by the full cumulative M0..latest serial chain; and on every host (x86_64 CI + WSL2) the producer, loader, serialization, register-file IDs, and PL011 are proven green by `cargo test`. The existing QEMU-`virt` x0=FDT path stays byte-for-byte green.

## Approach
1. KERNEL ENTRY (crates/tb-hal/src/arch/aarch64/boot.rs): in the existing `.text._start` global_asm!, add `.globl _tb_start` immediately above `_start:` so both labels target ONE shared body (set SP=__boot_stack_top; zero .bss [__bss_start,__bss_end) 8 bytes at a time; `msr vbar_el1,__exception_vectors; isb`; `b rust_main`). x0 is preserved end-to-end into AAPCS64 arg0 for BOTH entries. Add a second global_asm! emitting the TABOS note byte-identical to the x86 one: `.section .note.TABOS,"a",%note; .balign 4; .long 6 /*namesz "TABOS\0"*/; .long 8 /*descsz u64*/; .long 0x54420001 /*TB_NOTE_TYPE_ENTRY64*/; .asciz "TABOS"; .balign 4; .quad _tb_start`.
2. LINKER (kernel/linker/aarch64.ld): add `PHDRS { note PT_NOTE FLAGS(4); text PT_LOAD FLAGS(5); data PT_LOAD FLAGS(6); }`; emit `.note.TABOS : ALIGN(4) { KEEP(*(.note.TABOS)) } :text :note` immediately before `.text`; tag `.text`/`.rodata` `:text` and `.data`/`.bss` `:data`; delete `*(.note .note.*)` from /DISCARD/. Keep ENTRY(_start), KEEP(*(.text._start)) (which now also anchors _tb_start), and KERNEL_LMA=0x40080000 — e_entry stays `_start`, the note carries `_tb_start`.
3. BOOT-MAGIC WINDOW (crates/tb-hal/src/lib.rs + arch/{x86_64,aarch64}/mod.rs): replace read_boot_magic's hardcoded x86 [0x1000,1GiB) bounds with `arch::BOOT_INFO_WINDOW: (usize, usize)` = x86 (0x1000, 0x4000_0000), aarch64 (0x4000_0000, 0x8000_0000). The aarch64 window is the identity-mapped RAM gigabyte (L1[1]), present BOTH MMU-off (flat PA before mmu_init) and MMU-on (Normal-WB identity block), so the guarded 8-byte magic read of a TbBootInfo placed in DRAM is always mapped+aligned. The magic gate itself is unchanged (fail-closed).
4. PMM CONSUMER (crates/tb-hal/src/arch/aarch64/pmm.rs): when `read_boot_magic(boot_info)==Some(TB_BOOT_MAGIC)`, parse the `tb_boot::TbBootInfo` + its packed TbMemRegion[] (the single guarded raw read stays in tb-hal; the safe `tb_boot::regions_from_bytes` validator does the rest), pushing RAM kinds clamped to [0x4000_0000,0x8000_0000) and Reserved kinds verbatim; ELSE keep today's hard-coded QEMU-virt map + FDT-totalsize DTB reservation. Shared pmm layer still adds the kernel-image reservation. This makes the producer's mem_regions load-bearing on the tb-boot path while leaving the QEMU-virt path untouched.
5. KERNEL DOC (kernel/src/main.rs): update the rust_main header so the aarch64 bullet lists TWO entries (`_start` x0=FDT/DTB, `_tb_start` x0=guest-physical TbBootInfo*); NO code change — read_boot_magic already gates the `tb-boot: contract v0 OK` line on the magic and the cumulative chain is arch-shared.
6. TB-VMM ARCH SEAM (tb-vmm/src/arch/mod.rs): add `#[cfg(target_arch="aarch64")] pub mod aarch64;`, route `setup()` to `aarch64::boot::setup` under that cfg (drop the Unsupported stub there); add `pub const EXPECTED_ELF_MACHINE: u16` (x86_64=62, aarch64=0xB7) and `pub const GUEST_RAM_BASE: u64` (x86_64=0, aarch64=0x4000_0000) selected per host arch.
7. TB-VMM LOADER (tb-vmm/src/loader.rs): replace the hardcoded `EM_X86_64` machine check with `arch::EXPECTED_ELF_MACHINE` (add EM_AARCH64=0xB7); the PT_NOTE TABOS discovery + MissingTbNote refusal are arch-neutral and stay. Add a unit test building a synthetic ELFCLASS64-LE EM_AARCH64 image with a TABOS note (reuse the existing build_elf harness, parameterized by machine) asserting the note entry is used and a wrong-machine image is rejected.
8. TB-VMM MEMORY (tb-vmm/src/memory.rs): base the single guest region + GuestAddress at `arch::GUEST_RAM_BASE` (region {base:GUEST_RAM_BASE, len, Ram}); register_with_kvm already iterates inner regions, so guest_phys_addr follows the base automatically. On aarch64 the kernel loads at p_paddr 0x4008_0000 which now lies inside [0x4000_0000, base+len).
9. TB-VMM AARCH64 PRODUCER (tb-vmm/src/arch/aarch64/{mod.rs,boot.rs}): (a) write the boot block into the sub-image DTB-staging window — TbBootInfo@0x4000_1000, TbMemRegion[]@0x4000_1040, cmdline@0x4000_1800 (all < IMAGE_LMA 0x4008_0000; bound-checked like the x86 backend), reusing tb-boot serialization and the compile-time size_of asserts. (b) KVM_ARM_VCPU_INIT: `VmFd::get_preferred_target(&mut kvi)` then set `kvi.features[0] |= 1<<KVM_ARM_VCPU_PSCI_0_2` then `VcpuFd::vcpu_init(&kvi)`. (c) KVM_SET_ONE_REG with the verified core-reg IDs (base 0x6030_0000_0010_0000): X0=base|0x0000 ← 0x4000_1000, X1=base|0x0002, X2=base|0x0004, X3=base|0x0006 ← 0, PC=base|0x0040 ← entry, PSTATE=base|0x0042 ← 0x3c5 (EL1h+DAIF). MMU/caches stay at KVM reset (off); the kernel brings the MMU up in mmu_init. (d) call gic::create() before returning. Unit-test serialization, the reg-ID constants, PSTATE=0x3c5, and the <IMAGE_LMA placement bound.
10. TB-VMM vGIC (tb-vmm/src/arch/aarch64/gic.rs): create the in-kernel GICv2 via KVM_CREATE_DEVICE(KVM_DEV_TYPE_ARM_VGIC_V2) and set KVM_DEV_ARM_VGIC_GRP_ADDR for GICD=0x0800_0000 and GICC=0x0801_0000 — the exact bases timer.rs hard-codes — so the guest's M8 GICv2 init + EL1 physical-timer PPI 30 are KVM-emulated and M8/M9 fire (the aarch64 analogue of x86 `create_irq_chip()`). Single vCPU. Called from boot::setup after vcpu_init, before first KVM_RUN.
11. TB-VMM PL011 (tb-vmm/src/pl011.rs): a minimal PL011 BusDevice — write at offset 0x000 (UARTDR) forwards the low byte to the host writer (stdout); read at offset 0x018 (UARTFR) returns 0 (BUSY clear so serial_init's drain-poll passes, TXFF clear so serial_write_byte's poll passes); all other offsets inert. Span [0x0900_0000, +0x1000). Unit-test a write surfaces to a captured writer and FR reads 0.
12. TB-VMM RUN WIRING (tb-vmm/src/vmm.rs + cli.rs): make device+IRQ wiring arch-cfg — x86_64 keeps `create_irq_chip()` + 16550 on pio_bus@0x3f8; aarch64 registers the PL011 on mmio_bus@0x0900_0000 (vGIC is created inside arch::setup). cli.rs default kernel path becomes arch-selected (aarch64 → target/aarch64-tabos-none/<profile>/tabos-kernel). The KVM_RUN exit loop already dispatches MmioRead/MmioWrite, so no loop change is needed.
13. RUNNER (scripts/run-vmm-aarch64.sh): mirror run-vmm-x86_64.sh — /dev/kvm presence gate (SKIP exit 0 if absent/inaccessible), build tb-vmm on the aarch64 host, run it on the aarch64 kernel ELF under `timeout --foreground`, and PASS only if the captured output contains BOTH `tb-boot: contract v0 OK` (proves the X0=TbBootInfo* register-file handoff) and the final cumulative marker; fail-closed otherwise.

CI (.github/workflows/vmm-boot-aarch64.yml): `runs-on: ubuntu-24.04-arm`; install nightly (rust-src, llvm-tools); ALWAYS run `cargo test` in tb-vmm (host-arch-independent producer/loader/serialization/PL011 tests) + `cargo kbuild --target targets/aarch64-tabos-none.json`; the boot step (`bash scripts/run-vmm-aarch64.sh`) is /dev/kvm-gated allow-skip exactly like the x86 vmm-boot KVM_OK discipline. The required ci.yml aarch64 QEMU-virt lane is untouched and remains the regression gate for the x0=FDT path.

DOCS (docs/MILESTONES.md, SOVEREIGNTY-L2-ROADMAP.md §5/GAP, SOVEREIGNTY-ROADMAP.md §7): flip the 'aarch64 has no tb-boot handoff / tb-vmm x86-only' GAP notes to DONE and record the frozen aarch64 tb-boot v0 contract (X0=TbBootInfo*, x1..3=0, PSTATE=0x3c5, MMU off, RAM base 0x4000_0000, boot block <0x4008_0000, vGIC GICD/GICC 0x0800_0000/0x0801_0000, PL011 0x0900_0000).

## Decisions — no open questions (açık karar bırakma)
- `_tb_start` shares ONE body with `_start` (two labels, fall-through) rather than a separate routine — because on aarch64 the boot CPU state is contract-IDENTICAL for QEMU-virt and KVM-direct (EL1h, MMU off, DAIF masked, arg in x0); only the SEMANTICS of x0 differ (FDT vs TbBootInfo*), and the kernel disambiguates at runtime via the magic. (Contrast x86, where `_start` is a 32-bit PVH trampoline that MUST diverge from the 64-bit `_tb_start`.) Zero divergence-bug surface.
- x0 carries the guest-physical TbBootInfo*; x1=x2=x3=0 per the arm64 boot protocol — a deliberate contract-defined override of Linux's x0=dtb, exactly mirroring how tb-boot puts the pointer in x86 rdi instead of Linux's rsi zero-page.
- PSTATE handoff = 0x3c5 (EL1h + D/A/I/F masked); MMU and caches left at KVM reset (OFF). The kernel's mmu_init (M3) brings translation up cold — so the producer needs ZERO MMU/page-table programming (unlike x86, where tb-vmm must pre-build CR3/identity tables). This is why the aarch64 producer is mostly a register-file write.
- Guest RAM base = 0x4000_0000 on aarch64 (vs 0 on x86), via arch::GUEST_RAM_BASE — matches the kernel link address (0x4008_0000), the pmm hard-coded virt map, and the M3 identity RAM gigabyte; avoids wasting ~1 GiB just to reach the link address if based at 0.
- Boot block lives in the 512 KiB sub-image window [0x4000_0000,0x4008_0000) (TbBootInfo@0x4000_1000, regions@+0x40, cmdline@+0x800) — RAM that pmm already reserves as DTB staging, below the image so the kernel's .bss-clear never touches it, and inside the identity gigabyte so it's readable both MMU-off and MMU-on.
- read_boot_magic's safe-read window becomes arch-specific: aarch64 = [0x4000_0000,0x8000_0000) — because on aarch64 there is no RAM below 0x4000_0000 (that gigabyte is Device MMIO), so the boot-info pointer is necessarily in the RAM gigabyte; the x86 [0,1GiB) window would have rejected it.
- .note.TABOS uses the Arm-family-safe `%note` section-type prefix (not `@note`) and is placed in a new PT_NOTE phdr via the linker; e_entry stays `_start`. The current aarch64.ld DISCARDs all notes, so both the linker PHDRS change and un-discarding the note are mandatory for tb-vmm's PT_NOTE-walking loader to find the entry.
- tb-vmm provides an in-kernel GICv2 at GICD=0x0800_0000/GICC=0x0801_0000 (the kernel's hard-coded timer.rs bases) so M8/M9 reach green; the architected EL1 physical timer is then KVM-emulated automatically. This is the aarch64 analogue of the x86 backend's create_irq_chip(); without matching bases M8 would hang.
- tb-vmm registers a minimal PL011 MMIO console at 0x0900_0000 (FR reads 0, DR writes→stdout) for the aarch64 path; the x86 path keeps its 16550 PIO console. Required because the kernel emits every marker through the PL011 — no console, no DoD proof.
- The loader accepts the host arch's machine via arch::EXPECTED_ELF_MACHINE (EM_AARCH64 on aarch64, EM_X86_64 on x86) rather than a runtime flag — tb-vmm only ever boots a same-arch guest under same-arch KVM, and the TABOS-note discovery is machine-agnostic, so this stays correct AND unit-testable on both hosts.
- tb-vmm performs NO host-side cache maintenance: KVM arm64 cleans guest memory to PoC for caches-off guests on stage-2 fault (the documented reason Linux/Firecracker arm64 MMU-off guests boot), so the loaded image + boot block are coherent for the MMU-off `_tb_start` without tb-vmm touching DC CVAC.
- KVM_ARM_VCPU_PSCI_0_2 is set in KVM_ARM_VCPU_INIT even though the guest is single-core — it matches the in-repo [DECISION, verified] roadmap path and keeps WFI / PSCI SYSTEM_OFF well-defined; harmless for one vCPU.
- DoD reuses the existing `tb-boot: contract v0 OK` serial line (fired whenever the magic matches, on ANY tb-boot path) rather than adding an aarch64-specific kernel print — the kernel is arch-shared and a new print would perturb both arches' cumulative output; the new aarch64 tb-vmm run lane is what makes the proof aarch64-specific.
- Delivery is staged for incremental CI-green: (1) handoff + M0..M4 (needs no GIC), then (2) vGIC for M8..M18 — but BOTH land under this one task; every host-arch-independent piece (producer serialization, reg-ID encoding, loader, PL011) is gated by `cargo test` so the build is the arbiter even where aarch64 KVM is unavailable.

## Risks & mitigations
- aarch64 KVM may be unavailable on hosted CI (GitHub arm64 runners are VMs; nested KVM is not guaranteed). Mitigation: KVM-gated allow-skip lane + full `cargo test` coverage of every host-arch-independent piece; the real boot proof runs on an arm64 host with /dev/kvm (Apple Silicon Linux VM with nested virt, Graviton bare-metal-class, or Arda's hardware) — the same skip-but-still-build discipline the x86 vmm-boot job and the L2.0 graceful-skip already use.
- Cache incoherence: tb-vmm writes the image+boot block through a cacheable host mapping, but `_tb_start` runs with the guest MMU/caches OFF (non-cacheable reads). Mitigation: rely on KVM arm64's clean-to-PoC on stage-2 fault for caches-off guests (the mechanism that lets Linux/Firecracker arm64 MMU-off guests boot); if a target host ever lacks it, add a bounded host-side `DC CVAC` loop over only the image + boot block in tb-vmm (which is outside forbid(unsafe)). Documented, not currently needed.
- vGIC/timer base mismatch: the kernel hard-codes GICD/GICC (0x0800_0000/0x0801_0000) and timer PPI 30, so if KVM places the vGIC elsewhere M8 hangs and the marker never prints. Mitigation: explicitly set KVM_DEV_ARM_VGIC_GRP_ADDR to those exact bases; unit-assert the constants; the run lane's missing `M8: timer OK` fails closed loudly.
- Linker PHDRS/note-section change could regress the QEMU-virt boot, which IS on the required ci.yml matrix. Mitigation: run-aarch64.sh is the gate; verify e_entry==_start and the note via llvm-readobj before merge; keep ENTRY(_start) + KEEP(*(.text._start)) intact so `_start`/`_tb_start` are never gc'd.
- `%note` vs `@note` section-type and the AArch64 `//` comment convention can silently mis-assemble the note under the LLVM integrated assembler used by global_asm!. Mitigation: use `%note` (Arm idiom accepted by LLVM MC + GAS) and gate on `llvm-readobj --notes` showing the correct namesz/descsz/type; if rejected, emit the note as raw `.long/.asciz/.balign/.quad` in a plain `"a"` section assigned `:note` by the linker.
- kvm-ioctls 0.24 / kvm-bindings 0.14 aarch64 API drift (get_preferred_target, vcpu_init, set_one_reg byte-slice signature, create_device for the GIC). Mitigation: the pins already match the Firecracker baseline in tb-vmm/Cargo.toml; cross-check the exact call shapes against Firecracker's src/vmm/src/arch/aarch64 at those pinned versions and encode register values with to_le_bytes.
- Widening read_boot_magic's aarch64 window to the whole RAM gigabyte means a stray FDT pointer (also in DRAM on the QEMU path) is now inside the window. Mitigation: the magic gate already fail-closes — an FDT header reads 0xD00DFEED (big-endian) which never equals TB_BOOT_MAGIC; the window only bounds the SAFE read, never the decision.
- Scope: reaching the full M0..M18 chain under a brand-new aarch64 KVM backend is large (L→XL). Mitigation: stage delivery (handoff+M0..M4 needs no GIC; then add vGIC for M8..M18), each sub-step independently greenable, with the `cargo test` matrix gating every host so partial progress never regresses the tree.

## Research notes (with sources)
- arm64 Linux boot protocol, primary CPU register/mode state (the contract tb-boot mirrors): x0 = physical address of the dtb in RAM; x1=x2=x3=0 (reserved); ALL of PSTATE.DAIF masked; CPU in non-secure EL1 (or EL2); MMU OFF; the loaded-image address range cleaned to PoC; dtb 8-byte aligned and <=2 MiB. Source: kernel.org Documentation/arch/arm64/booting.rst, '4. Call the kernel image' (fetched 2026-06-08).
- KVM ARM64 core-register encoding (the producer's KVM_SET_ONE_REG IDs), verbatim from the KVM API doc: base 0x6030_0000_0010_0000 (= KVM_REG_ARM64 | KVM_REG_SIZE_U64 | KVM_REG_ARM_CORE), index = offsetof(kvm_regs)/4: X0=...0000, X1=...0002, X2=...0004, X3=...0006, X30=...003c, SP=...003e, PC=...0040, PSTATE=...0042, SP_EL1=...0044, ELR_EL1=...0046. Source: docs.kernel.org/virt/kvm/api.html (ONE_REG arm64 table).
- PSTATE handoff value = 0x3c5 = PSR_MODE_EL1h(0x5) | PSR_D(0x200) | PSR_A(0x100) | PSR_I(0x80) | PSR_F(0x40). Cross-confirmed by the booting.rst 'all DAIF masked' requirement AND the in-repo aarch64 boot.rs comment 'PSTATE = 0x3c5 = EL1h with D/A/I/F masked' and user.rs SPSR usage.
- KVM_ARM_VCPU_INIT path: VM ioctl KVM_ARM_PREFERRED_TARGET returns a `struct kvm_vcpu_init` (out); set feature bit KVM_ARM_VCPU_PSCI_0_2 (depends on KVM_CAP_ARM_PSCI_0_2), then per-vCPU KVM_ARM_VCPU_INIT. Source: docs.kernel.org/virt/kvm/api.html §4.83 + the VCPU feature list. In-repo this is already a [DECISION, verified] in docs/SOVEREIGNTY-ROADMAP.md §7: get_preferred_target -> vcpu_init(PSCI_0_2) -> KVM_SET_ONE_REG(PSTATE EL1h/DAIF, PC=entry, X0=info), core_reg_base 0x6030_0000_0010_0000, PC at base+2*32.
- QEMU `virt` memory map the kernel hard-codes (so tb-vmm must match): DRAM base 0x4000_0000; PL011 UART0 @ 0x0900_0000 (kernel arch/aarch64/serial.rs); GICv2 GICD @ 0x0800_0000 + GICC @ 0x0801_0000, EL1 physical timer PPI INTID 30 (kernel arch/aarch64/timer.rs); kernel image links at 0x4008_0000 (kernel/linker/aarch64.ld). Sources: in-repo files + QEMU hw/arm/virt.c memmap.
- tb-boot v0 ABI is the single shared contract (crates/tb-boot/src/lib.rs): TbBootInfo 56 bytes #[repr(C)] (magic@0,version@8,flags@12,mem_regions_ptr@16,len@24,cmdline_ptr@32,cmdline_len@40,kernel_entry@48); TbMemRegion 24 bytes; TB_BOOT_MAGIC=0x3056_544F_4F42_5400; TB_NOTE_NAME="TABOS", TB_NOTE_TYPE_ENTRY64=0x5442_0001, namesz 6, descsz 8 (u64 entry). The crate is no_std + forbid(unsafe_code) and arch-neutral, so it is reused on aarch64 unchanged.
- x86_64 reference path to mirror (crates/tb-hal/src/arch/x86_64/boot.rs + tb-vmm/src/arch/x86_64/boot.rs + loader.rs): a distinct `_tb_start` 64-bit entry anchored by a `.note.TABOS` in a PT_NOTE phdr; the producer writes the boot block into low guest RAM, programs the register file (rdi=TbBootInfo*), and the loader resolves the TABOS note ONLY (MissingTbNote, never e_entry). The aarch64 task reproduces this shape with EL1/MMU-off semantics.
- KVM arm64 maintains coherency for caches-off guests by cleaning guest pages to PoC on stage-2 fault (FEAT_S2FWB forces write-back where present), which is why arm64 Linux/Firecracker guests that enter with the MMU OFF boot without the userspace VMM issuing cache maintenance — so tb-vmm needs none for the loaded image + boot block. Source: arm64 KVM stage-2 fault handling + booting.rst PoC requirement.
- ELF machine constants for the loader: EM_X86_64 = 62, EM_AARCH64 = 0xB7 (183); ELFCLASS64=2, ELFDATA2LSB=1; PT_NOTE=4, PT_LOAD=1; note field 4-byte alignment. Source: System V gABI / elf.h (already encoded in tb-vmm/src/loader.rs).
