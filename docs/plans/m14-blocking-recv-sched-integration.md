# M14 blocking-recv + runnable-on-send: wire the empty-channel receiver to deschedule off the M9 run queue and be woken, lost-wakeup-free, by a sender

> **Task #51** · DoD marker: `M14.1: blocking-recv OK` · Effort: **M** · Slug: `m14-blocking-recv-sched-integration`
> **Depends on:** none

> Auto-generated from the `tabos-forward-plans` ultracode planning workflow (research-backed, decision-closed). Treat as the working spec for this task; refine in place as implementation reveals detail.

## Goal
M14 shipped capability-passing channels but the scheduler<->IPC round-trip was explicitly deferred: a receiver on an empty channel gets a synchronous SysStatus::WouldBlock instead of actually blocking, and a send does not wake anyone (ipc.rs/lib.rs/caps.rs, ROADMAP-V2 M14 status line). This task adds the deferred Step-2: a receiver with no message parks OFF the M9 run queue (yields to the scheduler), a sender that delivers makes it RUNNABLE, and the armed-timer scheduler then switches to it — with no lost wakeups. Done = the new cumulative serial marker "M14.1: blocking-recv OK" prints on both arches under QEMU and tb-vmm/KVM, proving a receiver was observed blocked off the run queue and then woken by a send to receive the exact sentinel; the whole M0..M18+L2.0 chain still re-prints (no regression); the only new unsafe is two arch interrupt-mask save/restore primitives in tb-hal.

## Approach
1. 1. Add interrupt-mask save/restore primitives (the lost-wakeup critical-section tool). In crates/tb-hal/src/arch/x86_64/timer.rs add `pub fn local_irq_save() -> u64` (asm `pushfq; pop {f}; cli`, return RFLAGS; IF is bit 9) and `pub fn local_irq_restore(f: u64)` (conditional `sti` iff `f & (1<<9) != 0`). In crates/tb-hal/src/arch/aarch64/timer.rs add `local_irq_save` (asm `mrs {d}, daif` then `msr daifset, #2`; return DAIF, I is bit 7) and `local_irq_restore(d)` (`msr daif, {d}`). Re-export both through crates/tb-hal/src/arch/mod.rs next to timer_rearm/read_cycle_counter. These are the ONLY new unsafe (confined to tb-hal/arch).
2. 2. Add the per-direction wait slot to the channel core. In crates/tb-hal/src/ipc.rs add `waiter: [Cell<u32>; 2]` to `Channel` (sentinel `const NO_WAITER: u32 = u32::MAX`), init both to NO_WAITER in `create()`, and add safe methods `set_waiter(&self, ring: usize, task: u32)`, `take_waiter(&self, ring: usize) -> Option<u32>` (read-and-clear), `peek_waiter`. Keeps `#![forbid(unsafe_code)]` (Cell only). The waiter is keyed by the shared `dir[]`/ring index: a receiver on side s waits on its inbox ring `(1 - s)`; a sender on side s pushes its outbox ring `s` — the SAME physical dir index, so they rendezvous.
3. 3. Wake from the send mechanism so EVERY send path (facade AND the real user syscall) wakes. In crates/tb-hal/src/caps.rs add `pub(crate) fn chan_of(&self, ep: Handle) -> Option<(Rc<crate::ipc::Channel>, u8)>`. In the existing `M_CHAN_SEND` dispatch arm, AFTER the `push_back` borrow is dropped, call `chan.take_waiter(side as usize)`; if `Some(t)`, call the new crate-level `crate::sched_wake_task(t)`. caps.rs stays `#![forbid(unsafe_code)]` (the wake is a safe atomic store; it touches no table/ring, so no RefCell re-borrow).
4. 4. Add the block/wake scheduler state in crates/tb-hal/src/lib.rs (all safe atomics). Add `static TASK_STATE: [AtomicU8; MAX_TASKS]` with `RUNNABLE=0`/`BLOCKED=1`; `pub fn task_is_blocked(t: Task) -> bool`; `pub(crate) fn sched_wake_task(slot: u32)` (store RUNNABLE). Modify `schedule()` (lib.rs:847) to pick the next RUNNABLE successor (skip entries whose TASK_STATE==BLOCKED); when nothing else is runnable it returns (resume current) exactly as today. Add `pub(crate) fn block_current_and_yield()`: store BLOCKED for CURRENT_TASK, scan the run queue round-robin for a RUNNABLE task != self, `yield_to` it; if none exists, `task_api_fatal("deadlock: all tasks blocked on IPC")` (fail-closed; the idle-task/WFI path is the documented forward item).
5. 5. Add the blocking-recv policy facade in lib.rs (scheduler-aware wrapper over the green non-blocking recv). `pub fn agent_chan_recv_blocking(t: Task, ep: caps::Handle) -> Option<(u32,u64,u64)>`: `let g = arch::local_irq_save();` then `loop { let r = <non-blocking recv on t's table via the existing chan_recv path>; if r.0 != WouldBlock { break r } resolve (chan, side) via caps::chan_of; chan.set_waiter((1 - side) as usize, current_task().raw() as u32); block_current_and_yield(); }` then `arch::local_irq_restore(g); Some(r)`. The whole wait runs under masked interrupts so the empty-recheck + waiter-register + deschedule is one indivisible critical section; on resume the loop re-checks under the same mask (spurious-wakeup tolerant).
6. 6. Build the armed-timer self-test in kernel/src/main.rs right AFTER `M14: ipc OK` (main.rs:1753), mirroring the M12 arm/disarm template (main.rs:1255-1296). Add `static STACK_BLK_SENDER: TaskStack<...>`, statics `SEND_AGENT_SLOT: AtomicUsize`, `SEND_EP_RAW: AtomicU64`, `RECV_BLOCKED_OBSERVED/SENT_DONE: AtomicBool`, `const SENTINEL: u64 = 0xB10C_KED5`, and `fn blk_sender()`. Block body: `scheduler_init()` (reset run queue to {boot}); `set_irq_hook(schedule)`; fresh `let (ep_a, ep_b) = agent_channel_connect(agent_c, agent_d, 1)`; store agent_c slot + ep_a.raw() into the statics; `scheduler_spawn(STACK_BLK_SENDER.take(), blk_sender)`; `timer_rearm()`; `let (s,p,_) = agent_chan_recv_blocking(agent_d, ep_b)` (BLOCKS boot off the run queue here); `timer_disarm()`; assert `s==Ok && p==SENTINEL && RECV_BLOCKED_OBSERVED && SENT_DONE && involuntary_switch_count() advanced`; else `m14b_fail(...)`. `blk_sender`: `irq_unmask()`; spin until `task_is_blocked(boot)` then set RECV_BLOCKED_OBSERVED; `agent_chan_send(Task::from_raw(SEND_AGENT_SLOT), Handle::from_raw(SEND_EP_RAW), SENTINEL, 0)`; set SENT_DONE; park in a spin loop. boot is the receiver (slot 0, reused) and the sender takes the only free slot 11 — no MAX_TASKS bump. Print witness `"ipc: receiver parked off the run queue, woken by send"` then `serial_write_str("M14.1: blocking-recv OK
")`.
7. 7. Wire the marker into the arbiter. Keep `MARKER='L2.0: vmxroot OK'` in scripts/run-x86_64.sh and scripts/run-aarch64.sh (it transitively gates the new self-test, which halts on failure), and ADD an explicit second assertion `grep -qF 'M14.1: blocking-recv OK'` to both scripts for direct traceability.
8. 8. Build + boot the arbiter: `cargo kbuild` for targets/x86_64-tabos-none.json and aarch64-tabos-none.json (0 warnings), then `bash scripts/run-x86_64.sh` and `bash scripts/run-aarch64.sh` (M0..M18+L2.0 chain re-prints AND the new marker appears), then the vmm-boot /dev/kvm path.

## Decisions — no open questions (açık karar bırakma)
- DoD marker = "M14.1: blocking-recv OK" (a NEW cumulative marker printed immediately after "M14: ipc OK", before M15). Why: M14 already shipped "M14: ipc OK"; a distinct sub-marker preserves traceability and re-runs every boot. The final-marker grep stays 'L2.0: vmxroot OK' (it transitively requires the new self-test to pass, since a failure halts before L2.0), plus an explicit second grep is added for direct visibility.
- Wait-queue placement = a single per-direction slot `waiter: [Cell<u32>; 2]` inside ipc::Channel, NOT a multi-waiter FIFO. Why: a two-endpoint point-to-point channel has exactly ONE reader per direction (the single owner of the opposite endpoint), so at most one receiver can ever block on a given ring. A general multi-waiter server endpoint (seL4's EPState_Recv TCB queue) would need an intrusive FIFO threaded through a per-task TASK_EP_NEXT link; that is an explicit forward extension, out of scope here.
- Block/wake representation = a parallel `TASK_STATE: [AtomicU8; MAX_TASKS]` (RUNNABLE/BLOCKED) with schedule() SKIPPING blocked entries, NOT run-queue compaction. Why: skipping is O(1) per entry, leaves slot indices stable (no array churn), and is provably backward-compatible — with nothing blocked it is byte-identical M9 round-robin, so M2/M9/M10/M12 stay green.
- Blocking POLICY lives in lib.rs (scheduler-aware); caps.rs/ipc.rs stay MECHANISM-only. Why: caps::dispatch is a single-shot capability primitive with no concept of yielding; keeping the retry/deschedule loop in the scheduler-owning layer preserves the clean M11 chokepoint and keeps caps/ipc forbid(unsafe).
- The WAKE fires INSIDE caps::dispatch's M_CHAN_SEND arm (via crate-level sched_wake_task), not only in the lib.rs send facade. Why: the mature design must wake the blocked receiver for the REAL user-mode send syscall too, not just the kernel self-test facade; doing it in dispatch covers every send path with one site.
- Lost-wakeup-free protocol = mask interrupts (new arch local_irq_save/restore) around the entire {recv-recheck -> register waiter -> mark BLOCKED -> yield} sequence, with a while-loop re-check after each wake. Why: this is the seL4/L4 'kernel runs with interrupts disabled' guarantee — on a single core a sender can only run after the receiver yields, so no send can interleave between the empty-check and the block; the re-check tolerates spurious wakeups.
- Wake semantics = mark-runnable only; the armed M9 timer's next schedule() performs the actual switch to the woken receiver. Why: keeps the integration minimal and identical to the proven M9 preemption path; seL4's possibleSwitchTo synchronous handoff (one less tick of latency) is a documented forward optimization.
- Self-test shape = boot task is the RECEIVER (it blocks off the run queue), and ONE new kernel task in the free slot 11 is the SENDER, over a FRESH agent_c<->agent_d channel; reuse M12's scheduler_init->spawn->timer_rearm->round-trip->timer_disarm template. Why: proves 'off the run queue' directly (the sender observes task_is_blocked(boot)==true before sending) with no MAX_TASKS bump (slot 11 is the only free slot at MAX_TASKS=12; agent tables have capacity 48 so a fresh endpoint fits).
- 'No runnable task at block time' = fail-closed fatal (report + halt 'deadlock: all tasks blocked on IPC'). Why: TABOS has no idle task yet, so all-blocked is a genuine single-core deadlock; halting is honest and CI catches it (no marker). The minimal idle-task/WFI(HLT) path is recorded as the forward item.
- Scope = blocking-RECV + runnable-on-send ONLY. A sender hitting a FULL ring still returns WouldBlock (symmetric blocking-send-on-full deferred), and the copy_to_user byte payload stays deferred. Why: this matches backlog task #21 exactly and keeps the increment auditable; the symmetric send-blocking layer is a noted, additive follow-up.
- New unsafe budget = exactly TWO arch primitives (local_irq_save/local_irq_restore) added under crates/tb-hal/src/arch/* . Why: framekernel confinement — kernel/, caps.rs, ipc.rs, mem.rs stay forbid(unsafe); the lib.rs scheduler additions (TASK_STATE, schedule skip, block_current_and_yield, the facade) are all safe atomics + existing yield_to.

## Risks & mitigations
- Lost wakeup if the empty-recheck and waiter-registration are not inside the interrupt-masked section (the classic sleep/wakeup race). Mitigation: local_irq_save/restore wrap the ENTIRE wait loop; the receiver re-checks the ring under the same mask after every wake (while-loop, spurious-tolerant). Cited mechanism: seL4 runs the kernel with interrupts disabled.
- Arming the timer mid-chain could resurrect the M2/M9/M10/M12/M13 tasks still suspended in their loops and corrupt later milestones. Mitigation: call scheduler_init() to reset the run queue to {boot} before spawning the sender (exact M12 precedent), so ONLY boot + the sender are runnable; disarm the timer before the marker so M15..M18/L2.0 keep running single-threaded.
- The schedule() BLOCKED-skip change could regress M9/M12 preemption. Mitigation: the skip is a no-op when TASK_STATE is all-RUNNABLE (M9/M12 never block), and the M2/M9/M12 self-tests re-run in the same boot — the arbiter fails loudly on any regression.
- Genuine deadlock if a block finds no other runnable task (no idle task exists yet). Mitigation: fail-closed fatal (report + halt), never a silent wrong-state; the self-test guarantees a runnable sender; the idle-task/WFI path is the documented forward item.
- RefCell double-borrow when waking from inside the M_CHAN_SEND arm. Mitigation: the wake reads the waiter Cell AFTER the push_back ring borrow is dropped, and sched_wake_task only does an atomic store (no table/ring borrow); single-core, interrupts masked — the same discipline the green M14 send path already uses.
- Endpoint side/ring confusion (an echo or a deadlock from the wrong dir index). Mitigation: keyed by the shared dir index — recv registers waiter[(1-side)] (its inbox), send wakes waiter[side] (its outbox), the SAME physical ring; the directional sentinel (C->D only) makes an echo bug surface as a hang -> loud failure.
- Layering concern: caps.rs calling a scheduler fn looks like an inversion. Mitigation: it is a crate-internal SAFE call (mirrors the existing dispatch_irq seam) and keeps caps forbid(unsafe); the alternative (wake only in the lib.rs facade) would silently miss the real user-mode send syscall, so wake-in-dispatch is the correct mature choice.
- Boot-as-receiver blocks the marker printer; a wakeup bug would hang the boot forever. Mitigation: the STALL_LIMIT-bounded wakeup wait converts any loss into a QEMU/CI timeout (red), and boot is woken deterministically in the no-bug path; this is acceptable because CI red on hang is the intended fail-closed outcome.

## Research notes (with sources)
- seL4 endpoint is a 3-state rendezvous machine: EPState_Idle / EPState_Send / EPState_Recv. In receiveIPC, when the endpoint has no waiting sender (Idle/Recv) and the recv is blocking, seL4 sets the thread state ThreadState_BlockedOnReceive, records blockingObject = the endpoint, calls scheduleTCB(thread) (removes it from running), and tcbEPAppend(thread, epptr, EPState_Recv) (enqueues the TCB on the endpoint's wait-queue). [seL4 src/object/endpoint.c, receiveIPC]
- seL4 wakes a blocked receiver from the SENDER: when sendIPC finds the endpoint in EPState_Recv it dequeues the head TCB, doIPCTransfer, then setThreadState(dest, ThreadState_Running) + possibleSwitchTo(dest). TABOS mirrors this as 'send marks the waiter RUNNABLE; the armed timer's next schedule() switches to it'. [seL4 src/object/endpoint.c, sendIPC EPState_Recv case]
- seL4's endpoint wait-queue is an INTRUSIVE doubly-linked list threaded through the TCBs themselves (tcbEPNext/tcbEPPrev; the endpoint stores epQueue_head/epQueue_tail) — zero separate allocation. TABOS's two-endpoint channel needs only a single per-direction waiter slot because each direction has exactly one reader; the intrusive N-waiter queue is the forward path for shared server endpoints. [seL4 endpoint.c cancelAllIPC loop on tcbEPNext; tcbEPAppend/tcbEPDequeue]
- The lost-wakeup-free guarantee is structural: seL4 and the L4 microkernel family run the kernel with interrupts (mostly) disabled while in kernel mode, so the empty-check + block + deschedule are one indivisible critical section. On a single core a sender cannot interleave between them. TABOS reproduces this with the new arch local_irq_save/restore around the recv wait loop. [seL4 whitepaper, sel4.txt line 1075 'executes with interrupts disabled while in kernel mode'; sel4_text.txt lines 737-742 'run the kernel with interrupts mostly disabled']
- Blocking removes a thread from the scheduler READY queue and waking re-enqueues it: seL4 uses tcbSchedDequeue/scheduleTCB to deschedule and SCHED_ENQUEUE/tcbSchedEnqueue to re-add, with a per-TCB tcbQueued flag (thread_state_get_tcbQueued) tracking ready-queue membership. TABOS's analog is the per-task TASK_STATE RUNNABLE/BLOCKED flag with schedule() skipping BLOCKED entries (no array compaction). [seL4 endpoint.c cancelAllIPC; sel4_text.txt lines 904, 1006-1021]
- seL4's non-blocking recv on an empty endpoint calls doNBRecvFailedTransfer — the NBRecv 'would not have data' path. This is exactly TABOS's existing SysStatus::WouldBlock (caps.rs:184, reserved 'for M14 channels'), so the blocking layer is a pure wrapper over the already-green non-blocking primitive — zero ABI break, no new method number. [seL4 endpoint.c receiveIPC else-branch; caps.rs SysStatus::WouldBlock]
- seL4 endpoints buffer NOTHING (zero-copy rendezvous), so seL4 blocks a receiver whenever no sender is queued. TABOS channels are BOUNDED ring buffers (ipc.rs Ring{VecDeque, cap}), so a TABOS receiver blocks only when the ring is EMPTY and a sender blocks only when FULL — this task targets the empty/receiver case; symmetric blocking-send-on-full stays WouldBlock and is the noted forward extension. [seL4 endpoint.c rendezvous vs crates/tb-hal/src/ipc.rs Ring]
- Grounding facts in the live TABOS tree confirmed by reading the code: schedule() at lib.rs:847 round-robins RUNQUEUE with no blocked concept; yield_to (lib.rs:301) is reused unchanged and is already called from IRQ context with interrupts masked; the M12 self-test (main.rs:1255-1296) is the proven 'scheduler_init -> spawn -> timer_rearm -> round-trip -> timer_disarm before marker' mid-chain template; MAX_TASKS=12 with slot 11 the only free slot; agent HandleTables have capacity 48 (lib.rs:1293) so a fresh agent_channel_connect fits; the M14 send arm already clones the endpoint Rc out and drops the ring borrow before mutating (caps.rs:1065-1115), the exact discipline the wake-after-push needs. [crates/tb-hal/src/lib.rs, caps.rs; kernel/src/main.rs]
