## Goal
The silicon-unsafe encoders that feed VMX/EPT/MMU assembly, plus the boot- and IPC-protocol (de)serializers, are currently only test-covered (the boot lane). This task elevates Tier-1 of the project's tiered-assurance plan (LANGUAGE-AND-STANDARDS.md: "Kani harnesses on every unsafe block and protocol parser") from documented-intent to a green CI gate. Concretely: extract the PURE bit-level encoder algebra into a new no_std + forbid(unsafe) leaf crate `tb-encode` (mirroring how `tb-boot` already factors out the pure boot ABI), then author Kani #[kani::proof] harnesses proving (a) encode-then-decode is identity for every fixed-size POD frame, (b) decode is total/panic-free and fail-closed on malformed input, and (c) the VMX control-MSR adjust + page-table index/address algebra satisfy their architectural legality invariants for ALL inputs. Done when `cargo kani -p tb-boot` and `cargo kani -p tb-encode` report VERIFICATION:- SUCCESSFUL for every harness, a wrapper script asserts the expected harness count and emits the marker, and a new CI lane greps that marker fail-closed while the existing ci/vmm-boot boot lanes stay green (refactor regression guard).

## Decisions — no open questions (açık karar bırakma)
- WHICH ENCODERS ARE PURE ENOUGH TO VERIFY: (VERIFY) tb-boot TbBootInfo/TbMemRegion/parse_entry64_note/Regions (already pure+host-buildable); the VMX control-MSR adjust algorithm; the CR0/CR4 fixed-bit clamp; EPT leaf/eptp + PTE make_entry/entry_addr/level_index/entry_is_valid; the TSS-descriptor base decoder; the new IPC frame codec + bounded ring. (DO NOT VERIFY, justified) the asm wrappers vmread/vmwrite/vmclear/vmptrld/world_switch global_asm/rdmsr and program_vmcs as a sequence -- CBMC cannot model inline asm, so these stay test-covered by the QEMU/nested-VMX boot lanes, exactly as KERNEL-FOUNDATION-SPEC.md already mandates ('raw asm remains test-covered, not formal').
- CRATE ARCHITECTURE: introduce a dedicated `crates/tb-encode` (no_std + forbid(unsafe) + zero deps + zero asm) as the single source of truth for the pure bit algebra, rather than verifying tb-hal in place. Rationale: cargo kani builds for the host and would compile tb-hal's x86_64 asm siblings (target_arch=x86_64 on the host) which CBMC cannot handle; a pure leaf crate is host-buildable, mirrors the existing tb-boot precedent precisely, and makes the encoders that FEED unsafe asm themselves provably safe -- strengthening the framekernel story instead of bending it.
- IPC SCOPE: the current ipc.rs payload is an inline scalar (the byte payload is deferred to backlog #20/#21), which is not a serialization round-trip. Decision: define the MATURE 16-byte on-wire MessageFrame codec NOW in tb-encode and verify its encode/decode identity + malformed-rejection, so the deferred copy_to/from_user IPC lands on a pre-verified codec. This gives a genuine (de)serialization round-trip to prove and avoids inventing throwaway scope.
- MARKER SEMANTICS: Kani proofs run on the HOST at CI time, not inside the QEMU guest, so this is a CI-log marker, NOT a cumulative guest serial marker. Establish the `V` (verification) marker namespace; this task = `V1: kani-encoders OK`. Reserve `V0` for the sibling M11 rights-subset proof (backlog #3) so it slots in cleanly. The marker is emitted by run-kani.sh after all harnesses verify and greps fail-closed in CI -- preserving 'the build is the arbiter'.
- HARNESS INPUT STRATEGY: build PODs field-by-field from kani::any() inside each harness instead of #[derive(kani::Arbitrary)], to keep zero Kani-attribute pollution in the normal build path; arrays use kani::any::<[u8;N]>() (Kani's built-in Arbitrary for [T;N]).
- BOUNDED vs TOTAL PROOFS (honest scoping): fixed-size POD round-trips (56/24/16-byte frames, NO loops) are TOTAL proofs over the full input space. The three loop-bearing parsers (parse_entry64_note NUL-scan, Regions iterator, BoundedRing) are BOUNDED proofs with explicit small input caps (name<=8, buffer<=72 bytes/<=3 regions, ring ops<=4) and #[kani::unwind(N+1)] so the unwinding-assertion fails closed if the bound is too low. This distinction is documented in the design json and module docs -- no overclaiming.
- CI INVOCATION: run `cargo kani -p tb-boot` and `cargo kani -p tb-encode` as TWO explicit per-package invocations, never `cargo kani --workspace`, because --workspace would attempt to build tb-hal/kernel/tb-vmm and choke CBMC on the asm. Runner OS = ubuntu-22.04 with a pinned kani-verifier version installed via cargo install + cargo kani setup.
- VT-d DMAR builders and the M13 BM25+/ACT-R fixed-point recall math (named in the broader backlog #19) are EXPLICITLY OUT OF SCOPE here: VT-d code does not exist yet (gated on L2.8) and M13 math is a memory-subsystem concern, not a silicon-unsafe encoder. Each gets its own follow-up verification task so this task stays the coherent 'silicon-unsafe encoders + protocol parsers' unit the title names.

## Risks & mitigations
- cargo kani dragging tb-hal asm into CBMC -> mitigated by per-package invocation (-p tb-boot, -p tb-encode) and never --workspace; the pure crates have zero asm and zero non-core deps.
- Loop-bearing parsers (parse_entry64_note, Regions, BoundedRing) causing CBMC non-termination -> mitigated by explicit small input bounds + #[kani::unwind(N+1)] with unwinding-assertions ON, so an under-set bound fails closed rather than hanging; bounds documented as a deliberate bounded-proof.
- Refactor (moving adjust/make_entry/eptp/decode_tss_base out of tb-hal) silently changing emitted encodings -> mitigated by call-through wrappers + pub use (callers untouched), the existing tb-boot/tb-hal unit tests moving with the code, and the ci.yml boot lane + L2.0 skip path as independent arbiters.
- Kani CI cost (GitHub 7GB RAM / 6h limits) -> harnesses are tiny (fixed 16/24/56-byte PODs + <=4-deep bounded loops) verifying in seconds; mitigated further by --output-format=terse and the option to move to a schedule if it ever grows; required-gate status revisited only if runtime becomes an issue.
- Kani version / runner-OS drift (docs historically cite Ubuntu 20.04-only) -> mitigated by pinning a recent kani-verifier release known to support ubuntu-22.04, installing via cargo install --locked + cargo kani setup, and caching the bundled toolchain; the lane is self-contained and does not touch the nightly build-std toolchain.
- Scope creep into VT-d / M13 math -> explicitly fenced out with named follow-up tasks so this task remains the silicon-encoder/parser unit and lands green without further questions.

## Research notes (with sources)
- Kani proof harnesses are functions annotated #[kani::proof]; kani::any::<T>() yields an unconstrained symbolic value of T and kani::assume() narrows preconditions; by default Kani proves absence of panics, failed asserts, arithmetic overflow, OOB indexing and other UB with NO explicit assertions needed -- a successful run is a mathematical proof over ALL inputs in scope (Kani book, tutorial-first-steps + usage).
- Harnesses must be gated `#[cfg(kani)]` (the kani crate is auto-injected only under cargo kani; a normal cargo build/test without it fails to resolve `kani::`), and the crate should add `[lints.rust] unexpected_cfgs = { level="warn", check-cfg=['cfg(kani)'] }` to Cargo.toml so Rust >=1.80's cfg-name checking does not warn (Kani book, usage.html 'Configuration in Cargo.toml' / 'The build process').
- Kani cannot verify unbounded loops; loop-bearing code needs an explicit upper bound on input size PLUS #[kani::unwind(n)] where n must be 'one more than' the iteration count (and sometimes +2/+3 with break/continue). The unwinding-assertion turns an under-set bound into a deterministic FAILURE rather than non-termination; this makes fixed-size POD round-trips (no loops) total proofs and the parser loops explicitly bounded proofs (Kani book, tutorial-loop-unwinding.html).
- Kani ships a GitHub Action model-checking/kani-github-action with working-directory/args/command inputs (args can be --workspace, --tests, --output-format=terse, --sarif for Code Scanning); the per-package form `cargo kani -p <crate>` runs only that crate's harnesses, which is how this plan avoids the asm-bearing tb-hal (Kani book, install-github-ci.html + usage.html 'Usage on a package').
- Kani's -Z concrete-playback --concrete-playback=print emits a runnable Rust unit test with the exact counterexample byte-vectors for a FAILED check (e.g. vec![135,137] decodes to 35207) -- the debugging path for any harness that bites, and the negative-control verification step in this plan (Kani book, reference/experimental/concrete-playback.html).
- Repo precedent for the pure/host-buildable verifiable crate already exists: crates/tb-boot is #![cfg_attr(not(test), no_std)] + #![forbid(unsafe_code)], depends on nothing, parses everything from a borrowed &[u8] via u64::from_le_bytes, and already carries to_bytes/read_from_prefix/validate + a full unit-test suite -- so tb-encode is a faithful clone of an established pattern, not a novel structure (crates/tb-boot/src/lib.rs, crates/tb-boot/Cargo.toml).
- The VMX control-MSR adjust algorithm `final=(desired|allowed0)&allowed1` is documented in controls.rs as 'the #1 cause of silent VM-entry failure' and the project's own KERNEL-FOUNDATION-SPEC.md §formal-scope already names exactly this work: 'Kani/Verus only on safe Rust (page-table arithmetic, address-translation round-trip, capability-derivation monotonicity, bitflag well-formedness); raw asm remains test-covered' -- this task is the direct implementation of that documented scope (crates/tb-hal/src/arch/x86_64/vmx/controls.rs; docs/KERNEL-FOUNDATION-SPEC.md:108).
- The pure page-table algebra already lives isolated and unsafe-free in crates/tb-hal/src/mmu.rs (make_entry/entry_addr/level_index/entry_is_valid are pure const fns with ENTRY_ADDR_MASK=0x0000_FFFF_FFFF_F000), and level_index masks with (ENTRIES-1)=0x1FF -- so the <512 index-bound and address/attr-separation properties are provable with no code change beyond re-homing into tb-encode (crates/tb-hal/src/mmu.rs).
- CI fail-closed convention is fixed in-repo: scripts/run-x86_64.sh sets MARKER, greps serial output, and exits non-zero on a missing marker or timeout; informational lanes (l2-nested-vmx.yml) use continue-on-error while ci/vmm-boot are the required gates -- run-kani.sh + kani.yml follow the required-gate-with-marker-grep form exactly (tabos/scripts/run-x86_64.sh, tabos/.github/workflows/{ci,l2-nested-vmx}.yml).
