# Wire Miri into CI as a mandatory, blocking Tier-0 UB gate over the host-testable pure-logic crates (tb-boot + new tb-core)

## Goal
Make dynamic Undefined-Behavior detection a required, fail-closed CI check over TABOS's host-testable pure logic, fulfilling the project's own resolved decision (OPEN-QUESTIONS §I, 2026-06-07: "Tier0 Miri+coding-guidelines mandatory"; LANGUAGE-AND-STANDARDS §5). Problem today: the encoders/parsers/rights-algebra/ring-buffers are `#![forbid(unsafe_code)]` modules but are TRAPPED inside the no_std silicon crate `tb-hal`, which only builds for the bare-metal `*-tabos-none` targets and cannot run under Miri; and the only host-testable crate (`tb-boot`) is not even exercised by CI (no `cargo test` job exists). Done = a `miri` job in `.github/workflows/ci.yml` runs `cargo miri test` over the host-testable surface, is registered as a required status check, is green on `main`, and the kernel boot lanes (M0..M18 + L2.0 markers) still pass unchanged. The Tier-0 surface is made host-buildable by extracting the pure logic into a new `#![forbid(unsafe_code)]` leaf crate `tb-core`, keeping ALL unsafe in `tb-hal` (framekernel rule preserved/strengthened).

## Approach
1. Create the leaf crate `crates/tb-core` with header `#![cfg_attr(not(test), no_std)] #![forbid(unsafe_code)] extern crate alloc;` and `[dependencies]` EMPTY (core+alloc only, zero arch/silicon deps) -- the exact pattern tb-boot already uses. Add it to the root `[workspace] members`.
2. Move the six pure-logic modules verbatim from `crates/tb-hal/src/` into `crates/tb-core/src/`: caps.rs (rights algebra / Object model), ipc.rs (bounded ordered FIFO = the ring buffer), mem.rs (tiered store), infer.rs (model registry), mmu.rs (PageTable512/Frame4K entry math), blocks.rs (shared-block registry/CAS). They already carry inner `#![forbid(unsafe_code)]` and only use `alloc` (Vec/Rc/RefCell/Cell/VecDeque/BTreeMap/Box) -- which links against std's allocator under the host `cfg(test)`/Miri build, and against tb-hal's #[global_allocator] in the bare-metal build. Declare them in tb-core/src/lib.rs (caps/infer `pub`, the rest `pub` so tb-hal facades reach them).
3. Break blocks.rs's silicon coupling by dependency inversion: define `pub trait FrameIo { fn frame_alloc(&self) -> Option<u64>; fn frame_zero_word(&self, pa: u64) -> Result<(),()>; fn frame_free(&self, pa: u64); }` in tb-core::blocks and change the frame-acquiring constructor to take `&dyn FrameIo` instead of calling the free functions `crate::frame_alloc`/`crate::addr_store_load`/`crate::frame_free` (the only three silicon calls any pure module makes -- grep-verified at blocks.rs L79/L87/L96). The pure registry/RECORD-CAS/Rc-sharing logic is untouched.
4. Rewire tb-hal: add `tb-core = { path = "../tb-core" }` to tb-hal/Cargo.toml; in tb-hal/src/lib.rs delete the six `mod` lines and replace with `pub use tb_core::{caps, infer};` plus `use tb_core::{mem, ipc, blocks, mmu};` (kernel call sites `tb_hal::caps::*` / `tb_hal::infer::*` keep resolving unchanged); implement `tb_core::blocks::FrameIo` for a tb-hal `SiliconFrameIo` zero-sized type whose methods call the real `frame_alloc`/`addr_store_load`/`frame_free`, and pass `&SiliconFrameIo` at the block-creation facade so M15 behavior is byte-identical. Repoint `crate::mmu` -> `tb_core::mmu` in arch/x86_64/mmu.rs and arch/aarch64/mmu.rs.
5. Add a per-module `#[cfg(test)] mod miri_tests` (std-harness, no `#[should_panic]`, asserting via assert_eq!/Result like tb-boot) where coverage is thin -- caps rights-subset/dispatch, ipc FIFO push/pop/full/cap-move, mem tier promote/recall, mmu entry encode/decode round-trips, blocks CAS under a `#[cfg(test)] struct MockFrameIo` (Vec-backed) so blocks runs under Miri with no silicon. tb-boot's existing tests already qualify.
6. Add a `miri` job to .github/workflows/ci.yml (single workflow, shares the existing concurrency-cancel): checkout; install the pinned nightly with components `miri, rust-src` via dtolnay/rust-toolchain; cache `~/.cargo/{registry,git}`, `target`, and the Miri sysroot keyed on hashFiles(rust-toolchain.toml, **/Cargo.lock); `cargo miri setup` (builds sysroot, keeps later output clean); fast native pre-filter `cargo test -p tb-boot -p tb-core`; then `MIRIFLAGS="-Zmiri-strict-provenance -Zmiri-symbolic-alignment-check" cargo miri test -p tb-boot -p tb-core` (host triple, default); then the big-endian ABI proof `cargo miri test -p tb-boot --target s390x-unknown-linux-gnu`; final `echo "T0: miri OK"`.
7. Pin rust-toolchain.toml `channel` from bare `nightly` to a dated `nightly-YYYY-MM-DD` known to ship both `miri` and `rust-src` (verified against rust-lang.github.io/rustup-components-history) so the blocking gate is deterministic and never flakes on a miri-less nightly; keep `components = [rust-src, llvm-tools]` (do NOT add miri here -- only the miri job installs it, so the boot lanes stay lean).
8. Register `ci / miri` as a required status check in the repo branch-protection rules for `main` (one-time admin step) so the gate is genuinely blocking for merges, and record the Tier-0 surface + invocation in LANGUAGE-AND-STANDARDS §5 next to the existing tiers decision.
9. Run the existing build-and-boot matrix to confirm the extraction preserved every milestone marker; iterate on any first-run Miri finding (most likely an Rc back-edge cycle flagged by the leak checker -- fix with Weak).

## Decisions — no open questions (açık karar bırakma)
- Tier-0 surface = exactly two crates: `tb-boot` (encoders/parsers + ELF-note validator) and the NEW `tb-core` (caps=rights algebra, ipc=ring buffer, mem, infer, mmu entry-math, blocks). Rationale: the task's named surface (encoders/parsers/rights-algebra/ring-buffers) maps onto these, and caps cannot be isolated alone because its Object body embeds mem::MemSubstrate/ipc::Channel/blocks::Block/infer::ModelSession as fields (grep-verified) -- they are one cohesive pure cluster.
- Module split = full extraction into a leaf crate, NOT cfg-gating tb-hal to compile on the host. Rationale: tb-hal/src/lib.rs is ~2000 lines of silicon facade over `mod arch` (global_asm! that cannot assemble for the host) + unsafe heap/pmm; gating all of it behind target_os would create two crates inside one. A clean `tb-core` is the mature design and STRENGTHENS the framekernel rule (LANGUAGE-AND-STANDARDS line 45: forbid(unsafe) in all upper layers) by moving pure logic OUT of the sole unsafe crate.
- blocks.rs silicon coupling resolved by a `FrameIo` trait (dependency inversion), not by leaving blocks in tb-hal. Rationale: leaving blocks out would drag caps (which holds Rc<Block>) out of Tier-0 too, gutting the surface; inverting the three frame calls is surgical and lets Miri exercise the CAS/registry logic against a Vec-backed mock.
- Host-buildability mechanism = `#![cfg_attr(not(test), no_std)]` on tb-core (identical to tb-boot), so under `cargo miri test` the cfg(test) build links std (allocator + libtest), and no `miri_start`/MIRI_NO_STD is needed (those are only for no_std BINARIES run via `cargo miri run`).
- Cargo invocation = `cargo miri setup` then `cargo miri test -p tb-boot -p tb-core` on the default HOST triple (x86_64-unknown-linux-gnu, Miri's best-supported OS), plus a second `cargo miri test -p tb-boot --target s390x-unknown-linux-gnu` big-endian lane to PROVE the tb-boot LE wire format (from_le_bytes/to_le_bytes) is endian-stable. No `-Zbuild-std` is triggered because the workspace .cargo/config.toml confines build-std to the `kbuild` alias only.
- Aliasing model = DEFAULT Stacked Borrows; do NOT enable `-Zmiri-tree-borrows` in the blocking gate. Rationale: the Miri README warns Tree Borrows is more experimental and code it accepts today 'might be declared UB in the future' -- unacceptable instability for a merge-blocking check. A non-blocking Tree-Borrows lane can be added later.
- MIRIFLAGS = `-Zmiri-strict-provenance -Zmiri-symbolic-alignment-check`; leak-checking and isolation left at their secure defaults. Strict-provenance is free insurance (forbid(unsafe) crates cannot do int-to-ptr casts anyway) and the leak checker turns Rc-cycle bugs into hard failures.
- panic strategy is a non-issue and requires NO override: the Cargo Book confirms the test harness ignores `panic` and forces unwind on tests + all their deps, so the workspace `[profile.dev] panic = "abort"` does not break `cargo miri test`.
- Gate placement = a job inside the existing ci.yml (named `miri`), not a separate miri.yml. Rationale: single workflow, shares concurrency-cancel, surfaces as one `ci / miri` check; the DoD explicitly permits 'job in ci.yml'.
- Determinism = pin rust-toolchain.toml to a dated nightly that ships miri+rust-src (consistent with the file's own 'bump the date freely' comment), eliminating the 'latest nightly lacks miri' flake class for a blocking gate.
- Effort is L, not the backlog's M: the M estimate assumed a tb-boot-only gate; the mature, decision-closed Tier-0 surface requires the tb-core extraction + FrameIo inversion. Honest re-sizing per the 'mature not minimal' mandate.
- A fast native `cargo test -p tb-boot -p tb-core` runs BEFORE Miri in the same job (Miri is ~10-50x slower), giving quick logic feedback and incidentally making tb-boot's currently-uncovered tests run in CI for the first time.

## Risks & mitigations
- First Miri run trips the leak checker on an Rc back-edge cycle in ipc/blocks/caps (e.g. a bidirectional Channel endpoint holding Rc both ways). Mitigation: this is a REAL latent leak bug -- convert the back-edge to `Weak`; it is exactly the class of bug the gate exists to catch.
- The FrameIo inversion subtly changes block frame-zeroing order and regresses the `M15: blocks OK` marker. Mitigation: keep tb-hal's SiliconFrameIo methods byte-identical to the old inline `frame_alloc`/`addr_store_load`/`frame_free` calls; the boot lane fails loudly if it diverges.
- The chosen nightly lacks the miri or rust-src component, flaking the blocking gate. Mitigation: pin a dated nightly verified against rust-lang.github.io/rustup-components-history; the boot lanes already require rust-src so the date is jointly constrained.
- Miri wall-clock grows as Tier-0 expands and slows every PR. Mitigation: surface is small today (two leaf crates, single-threaded Rc/RefCell logic); if runtime exceeds ~5 min, switch to `cargo miri nextest run -jN` (restores parallelism per the Miri README) -- no test changes required.
- Doc intra-links like `[crate::caps::ObjKind]` inside the moved modules break after relocation. Mitigation: they remain valid because the modules stay in the SAME crate (tb-core); the few tb-hal->tb-core references become `tb_core::` paths, validated by `cargo doc`.
- Extraction accidentally pulls a silicon symbol into tb-core, creating a tb-core->tb-hal cycle. Mitigation: grep already proved only blocks.rs touches silicon (3 calls), all inverted via FrameIo; `cargo build` fails closed on any residual cycle.
- Miri reports an UNSUPPORTED-operation (not UB) for something in the pure logic. Mitigation: none expected (pure core+alloc, the best-supported Miri surface on Linux); if it occurs, gate the specific test with `#[cfg_attr(miri, ignore)]` and file it -- never disable the whole job.

## Research notes (with sources)
- Miri runs the cargo test suite under a MIR interpreter and detects OOB/use-after-free, uninitialized reads, invalid bool/enum values, misaligned accesses, and data races; install with `rustup toolchain install nightly --component miri` then `cargo miri test`; the documented GitHub Actions job uses `cargo miri setup` then `cargo miri test` (source: rust-lang/miri README, 'Running Miri on CI').
- `cfg(miri)` is automatically set for code interpreted under Miri, and `#[cfg_attr(miri, ignore)]` skips tests Miri cannot run -- the standard escape hatch for unsupported ops (source: rust-lang/miri README, 'Using Miri').
- For no_std: MIRI_NO_STD and the `miri_start` entry point are ONLY for no_std BINARIES executed via `cargo miri run`; library crates are tested via `cargo miri test` and link std under the cfg(test) build, so TABOS's `#![cfg_attr(not(test), no_std)]` leaf crates (tb-boot, new tb-core) need no miri_start (source: rust-lang/miri README, 'Entry point for no-std binaries' + 'Using Miri').
- Cargo Book (Profiles, §panic): 'Tests, benchmarks, build scripts, and proc macros ignore the panic setting. The rustc test harness currently requires unwind behavior... when using the abort strategy and building a test, all of the dependencies will also be forced to build with the unwind strategy.' => the workspace `[profile.dev] panic = "abort"` does NOT break `cargo miri test`; no override needed.
- Cross-interpretation: `cargo miri test --target s390x-unknown-linux-gnu` runs the suite on Miri's big-endian 'target of choice', useful for endian-sensitive code; all Rust Tier-1 targets are supported and Linux targets have the best Miri support (source: rust-lang/miri README, 'Cross-interpretation' + 'Supported targets').
- `-Zmiri-tree-borrows` is even more experimental than Stacked Borrows and code it accepts today 'might be declared UB in the future' -- so the default Stacked Borrows model is the correct choice for a merge-blocking gate (source: rust-lang/miri README, '-Zmiri-tree-borrows').
- `-Zmiri-strict-provenance` halts on int-to-ptr provenance laundering; Miri's leak checker (on by default) errors when allocations remain reachable-only-via-cycles at exit -- catches Rc cycles (source: rust-lang/miri README, flags section + intro).
- Repo target specs targets/x86_64-tabos-none.json and targets/aarch64-tabos-none.json both set `"os":"none"` and `"panic-strategy":"abort"`, so `target_os = "none"` cleanly discriminates bare-metal from the Miri host build; crates/tb-boot/src/lib.rs already proves the `#![cfg_attr(not(test), no_std)]` + std-harness unit-test pattern (12 passing tests today, not yet run by CI).
- Grep-verified coupling: caps.rs embeds mem::MemSubstrate/ipc::Channel/blocks::Block/infer::ModelSession in its Object body (must co-locate in tb-core); blocks.rs is the ONLY pure module calling silicon facades in real code -- `crate::frame_alloc` (L79), `crate::addr_store_load` (L87), `crate::frame_free` (L96) -- requiring the FrameIo inversion; mmu.rs has zero cross-crate coupling.
- Project decisions to satisfy: LANGUAGE-AND-STANDARDS §5 lines 81-84 ('Tier 0 (~free): safe Rust + Miri (in CI, the dynamic UB detector the Rust Project itself uses)'), OPEN-QUESTIONS §I line 99 ('Tier0 Miri+coding-guidelines mandatory', Resolved 2026-06-07), and the framekernel rule line 45 ('unsafe only in the kernel-foundation crate; #![forbid(unsafe_code)] in all upper layers').
- Determinism source: per-nightly availability of the `miri` and `rust-src` components is published at rust-lang.github.io/rustup-components-history -- used to pick the pinned `nightly-YYYY-MM-DD` so the blocking gate never lands on a miri-less nightly.
- Existing CI baseline: .github/workflows/ci.yml currently has only a build-and-boot matrix (cargo kbuild + QEMU serial-marker scrape) and NO `cargo test` job; the workspace .cargo/config.toml confines `-Zbuild-std` to the `kbuild` alias, so a plain `cargo (miri) test -p ...` builds for the host with normal std and no build-std.
