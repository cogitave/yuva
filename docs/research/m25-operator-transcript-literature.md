# M25 literature survey — the verified operator transcript & the exogenous-oracle channel

Companion to [`docs/proposals/M25-operator-transcript.md`](../proposals/M25-operator-transcript.md). Synthesis of a 4-arm arxiv/standards research pass: **(A)** attestation & instance identity, **(B)** tamper-evident transparency logs & secure audit, **(C)** the human/external exogenous oracle for a self-graded loop, **(D)** the OS↔operator channel & human-factors of alerting. Every M25 mechanism traces to a primary source below; where M25 goes *beyond* a source it is flagged **[BEYOND]** with the reason (per the standing directive: nothing un-grounded; surpass the literature only where we see better, and say so).

## Arm A — attestation & "which instance am I" (the INTRO frame)

- **RFC 9334 — Remote ATtestation procedureS (RATS) Architecture** (IETF, 2023). Attester → Evidence → Verifier → Relying Party. **§3.2 layered attestation**: each measured layer is the attesting environment that measures the next — exactly M25's INTRO frame folding the M22 provenance head into the transcript genesis (the transcript is a new layer attesting *on top of* the M22 chain). **§10.2 freshness** (nonce/epoch/timestamp): a verifier-supplied nonce is the gold standard for replay defense — M25 has **no RX**, so it can offer only an epoch-ID-style self-assertion (**§10.3**). M25 **concedes** this: no liveness/freshness this milestone (**[caveat]**).
- **TCG DICE / open-dice** (Trusted Computing Group, *Device Identifier Composition Engine*). `CDI = OWF(UDS, measurement)`; a derived identity per measured layer. M25's fold `head' = chain_mix(head, prov_hash(entry))` is **structurally DICE minus the per-device secret (UDS)** — yielding a *measurement-derived identity* with **no key**, hence a public, recomputable structural ID, not a sealed DeviceID. **[BEYOND / honest-downgrade]**: M25 deliberately ships the keyless structural form (`keyed=0`) and names the keyed successor (M26+), rather than overclaiming a DICE DeviceID it cannot root without a UDS.
- **RFC 9711 — the Entity Attestation Token (EAT)** (IETF, 2024). UEID (Universal Entity ID) = a stable per-entity identifier (like a serial number), distinct from a key-certified identity. M25's folded head is a **structural UEID** — an instance fingerprint, explicitly *not* a certified key. The kind/severity field vocabulary is kept EAT/claim-like for forward-compat with a real EAT in M26.

## Arm B — tamper-evident transparency logs & secure audit (the fold + seq + truncation guard)

- **Schneier & Kelsey — "Cryptographic Support for Secure Logs on Untrusted Machines"** (USENIX Security 1998; *Secure audit logs to support computer forensics*, ACM TISSEC 1999). The forward-integrity hash-chain: `H_i = hash(H_{i-1} || M_i)`; an attacker who compromises the machine at time *t* cannot undetectably alter entries *< t*. This is M22's `chain_mix` fold, reused verbatim. **Known gap**: the bare chain is blind to **truncation of the tail** (lopping off the most recent entries leaves a valid prefix).
- **Ma & Tsudik — "Forward-Secure Sequential Aggregate Authentication" (FssAgg)** (IACR ePrint 2007/052; NDSS 2009). Closes the truncation gap with an aggregated authenticator + an explicit **sequence number / closing commit** so the verifier detects a missing tail. M25 adopts the **structural** form: a **strictly-monotone `seq` folded INTO `canon()`** + a **closing GATE_VERDICT frame that commits the final seq + running head** → a reader detects tail-truncation, reorder, gap, and duplication. (`seq` *must* be inside the hashed bytes — a side-channel seq is renumberable; the **Terrapin attack**, Bäumer-Brinkmann-Schwenk, arXiv:2312.12422 / CVE-2023-48795, is the cautionary tale: unauthenticated sequence/handshake prefixes are silently malleable.)
- **RFC 6962 / RFC 9162 — Certificate Transparency (v1 / v2)**; **Crosby & Wallach — "Efficient Data Structures for Tamper-Evident Logging"** (USENIX Security 2009). Append-only Merkle history tree; **Signed Tree Head (STH)** + consistency proofs prove append-only-ness and detect **equivocation/split-view** (showing different readers different logs). M25 ships only the linear fold now; the **signed-STH + consistency-proof successor** (and the balanced-Merkle batch inclusion of task **#75**) is M26+. **Concedes** the split-view first-mile gap: a keyless head cannot prove two readers saw the same transcript without gossip or a signature.
- **OpenTelemetry Logs Data Model (v1.53)** + **CBOR deterministic encoding** (draft-bormann-cbor-det / RFC 8949 §4.2). A typed LogRecord = EventName + integer **SeverityNumber** (1..24) + structured body; deterministic encoding = one canonical byte string per value (injectivity). M25's frame `kind` + integer `sev` mirror the OTel model (a recognized standard, no float), and `canon()` is a deterministic injective encoding (Kani-proven), the CBOR-det discipline applied in-kernel.

## Arm C — the exogenous human/external oracle (why M25 exists at all)

- **Christiano, Leike, Brown, Martic, Legg, Amodei — "Deep RL from Human Preferences"** (arXiv:1706.03741, NeurIPS 2017). A scarce human supplies preference labels over trajectory segments; the agent learns a reward model from them. The human is the **exogenous ground-truth oracle** a self-graded agent lacks — *precisely* the role M25's transcript sets up (surface decisions → human eventually labels → that label is the M24 gate input M24 currently has no honest source for).
- **Settles — "Active Learning Literature Survey"** (Univ. Wisconsin TR-1648, 2009). **Uncertainty sampling / margin sampling / query-by-committee**: query the human on the *most informative* (most uncertain / most-disagreed) instances to minimize the label budget. M25's EXPERIENCE_DIGEST surfaces exactly the **BORDERLINE** decisions — ranked by `|kan_score_shadow − heuristic margin|` (margin sampling) and forget-vs-counterfactual disagreement (a query-by-committee proxy) — so a scarce human labels the records that most un-confound the gate.
- **Thomas, da Silva, Barto, Giguere, Brun, Brunskill — "Preventing undesirable behavior of intelligent machines"** (the **Seldonian** framework, *Science* 364:6451, 2019). A safety constraint is checked on a **sealed held-out** dataset that the candidate-selection process never touches; with **Dwork, Feldman, Hardt, Pitassi, Reingold, Roth — "The reusable holdout"** (*Science* 349:6248, 2015), adaptive reuse of a holdout overfits it unless mediated. **M25's leakage guard**: every frame carries `partition_id`; `canon()` **fail-closed asserts `partition != SAFETY_HELD_OUT`** so the transcript can NEVER surface a held-out record/label/statistic — the human's labels stay a one-shot pre-registered gate input. **[BEYOND]**: M25 encodes the Seldonian no-snoop requirement as a **Kani-proven canon-time invariant in the encoder**, not as operational discipline.

## Arm D — the OS↔operator channel & human-factors of alerting

- **EEMUA-191 / ISA-18.2 / IEC 62682** — alarm management standards. A sustained operator alarm rate above ~1 per 10 min (and bursts past ~10 in 10 min = an "upset") causes **alarm flooding** and operator desensitization (the documented contributor to Three Mile Island & the Texas City review findings). M25's EXPERIENCE_DIGEST is therefore **aggregated + rate-limited**, never per-decision — a per-decision stream would blow past the human ceiling in milliseconds, making the oracle useless (the alert-fatigue failure mode).
- **Backpressure / bounded-buffer fail-closed** (classic real-time systems discipline; cf. Lamport's bounded-buffer, and the kernel-logging "drop, don't block" rule). M25 emits over the existing `tb-encode::BoundedRing`: an overrun **drops with a counted overflow marker** rather than growing unboundedly or blocking the kernel — the surfacing channel can never become a liveness hazard for the OS it observes.
- **Serial console as the canonical operator transport** (16550 UART / ARM PL011) — the lowest-TCB, always-present out-of-band channel, used by every hypervisor/BMC. M25 is **TX-only** over it this milestone; **RX + an enrolled credential** (so a human can *command* gate activation) is the M26 inbound half (it fails the no-human CI gate today, hence deferred).

## What M25 deliberately does NOT claim (the honesty boundary, encoded as witness tokens)
| Property | Claimed by M25? | Token | Successor |
|---|---|---|---|
| Tamper-**evidence** vs *non-adversarial* corruption (drops, glitches, benign flips, reorder) | **YES** (fold + strict-monotone seq) | `fold-verified=1 tamper-caught=1 seq_monotone=1` | — |
| Truncation/reorder/replay-within-transcript detection | **YES** (seq-in-canon + closing commit) | part of `tamper-caught=1` | — |
| Instance binding ("which boot am I") | **YES** (INTRO ↔ live M22 head) | `intro_bound=1` | EAT/DICE keyed (M26) |
| Cryptographic authenticity / non-repudiation / forgery-resistance | **NO** (keyless FNV is publicly recomputable) | `keyed=0` | keyed forward-secure aggregate MAC (#74, M26) |
| Freshness / liveness (verifier nonce) | **NO** (TX-only, no RX, no trusted clock) | (implied by `keyed=0`) | RX + nonce challenge (M26, RFC 9334 §10.2) |
| Equivocation / split-view resistance | **NO** (no signature/gossip) | — | signed-STH + consistency proof (#75, M26) |
| A human actually read/believed the transcript | **NO** (self-test is self-graded plumbing) | `oracle=HUMAN-DEFERRED-M26` | RX + enrolled operator credential (M26) |

The simulated operator-verifier in the boot self-test proves the **channel** (emit + recompute + reject-on-tamper), never the **oracle** — the marker `M25: operator OK` means "the verified transcript channel works", explicitly *not* "a policy was validated" or "the M24 gate cleared".
