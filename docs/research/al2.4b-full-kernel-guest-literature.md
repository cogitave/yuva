## 1. The requirements anchor — the AArch64 Linux/PE boot protocol

The contract a VMM must reproduce to hand control to a real arm64 kernel is the **AArch64 Linux boot protocol** ([kernel.org `arch/arm64/booting.html`](https://docs.kernel.org/arch/arm64/booting.html)). Every production aarch64 VMM (kvmtool, Firecracker, QEMU, Cloud-Hypervisor) implements exactly this, and it is a **near-exact superset of the nVHE EL1 drop TABOS already performs** in `boot.rs` (the same `SPSR_EL2 = 0x3C5` PSTATE, the same x0 handoff convention) — only pointed at a real `Image`/`_tb_start` instead of an in-tree stub. The primary-CPU initial state:

- **`x0` = physical/IPA base of the boot blob** (DTB for Linux; for TABOS, `TbBootInfo*`). `x1=x2=x3=0` (reserved). This is the **only** channel by which the guest learns its memory map and devices.
- **`pc` = the kernel entry** (= the Image base; the 64-byte Arm64 Image header is itself executable and branches over its own data — for TABOS the entry is `_tb_start`, see §3).
- **`PSTATE` = EL1h with D,A,I,F masked**. kvmtool sets `PSR_D_BIT|PSR_A_BIT|PSR_I_BIT|PSR_F_BIT|PSR_MODE_EL1h == 0x3C5` ([kvmtool `arm/aarch64/kvm-cpu.c` `reset_vcpu_aarch64`](https://git.kernel.org/pub/scm/linux/kernel/git/will/kvmtool.git/tree/arm/aarch64/kvm-cpu.c)) — **byte-identical** to the `SPSR_EL2 = 0x3C5` value TABOS `boot.rs` already uses for its EL1 drop and every L2.x guest `eret` (`INIT_PSTATE_EL1`, const-asserted as `SPSR_EL1H_DAIF`).
- **CPU state at entry**: MMU OFF; all interrupts masked in `PSTATE.DAIF`; non-secure; D-cache range of the Image clean to PoC; **I-cache "may be on or off, and must not hold any stale entries"** (the image-copy path must `IC` invalidate); little-endian; Image at a 2 MiB-aligned base + `text_offset`. The kernel accepts entry at EL1 (a guest under TABOS is handed EL1, believing it is the top).
- **Timer state at entry (the protocol's own timer clauses — load-bearing for §4):** `CNTFRQ` programmed with the timer frequency; `CNTVOFF` "programmed with a consistent value on all CPUs" (an inconsistent/changing CNTVOFF is the time-goes-backwards pitfall, §7b); and **for EL1 entry specifically, `CNTHCTL_EL2.EL1PCTEN (bit 0) must be set where available`** — i.e. the EL2 monitor MUST grant the guest physical-counter access or a protocol-conformant guest is entitled to hang. TABOS's own M8/M9 path uses `CNTP_*_EL0` directly, so the monitor must leave `EL1PCEN+EL1PCTEN=1` (passthrough) or trap-and-emulate them.
- **The 64-byte `Image` header** (what a real loader validates before copying): `code0/code1` (8B, executable), `text_offset` (8B LE), `image_size` (8B LE, "at least image_size bytes from the start of the image must be free"), `flags` (8B LE — bit 0 endianness, bits 1–2 page size, **bit 3 "physical placement anywhere in the 48-bit range"**), `res2..res4`, **`magic = 0x644d5241` ("ARM\x64") at offset 0x38**, `res5` (PE/COFF offset). The aL2.4b guest-RAM loader's validation set is exactly: magic match, `text_offset`/`image_size` honored, base 2 MiB-aligned.
- **DTB constraints** (for a Linux-class guest; `TbBootInfo` is exempt but should mirror the spirit): 8-byte aligned, ≤ 2 MiB, mapped cacheable in 2 MiB blocks so it "must not be placed within any 2M region which must be mapped with specific attributes" (i.e. never adjacent to MMIO); pre-v4.2 kernels additionally required it within the 512 MiB below/around the image.

**The aL2.4b delta vs aL2.4-v1**: v1 `eret`s into a `#[unsafe(naked)]` stub with no DTB/boot-block and a magic in x0; aL2.4b must (a) **land a kernel image inside a stage-2-confined guest-RAM carve**, (b) **place a generated `TbBootInfo` block and put its IPA in x0** (not a magic), and (c) `eret` to the image entry. The PSTATE, the `eret` mechanics, and the per-VMID `VTTBR_EL2` are **already proven** (aL2.0..aL2.6, M27).

## 2. The closest pedagogical and production analogs

- **`ashwio/arm64-hypervisor-tutorial`** ([github](https://github.com/ashwio/arm64-hypervisor-tutorial), [ashw.io blog](https://ashw.io/blog/arm64-hypervisor-tutorial/1)) — **THE closest pedagogical analog**: boots a guest EL2→EL1 and prints via an **emulated PL011** in pure A64 asm, then layers stage-2, DTB spoofing, CPU-feature/sysreg trap-emulation, device emulation, and virtual interrupts step by step. The direct template for the aL2.4b increment, and notably it stages the **PL011-print-only** first cut before the vGIC — exactly the "green floor then upgrade" shape aL2.4b adopts (the M27a→M27b pattern).
- **kvmtool** ([git.kernel.org `will/kvmtool`](https://git.kernel.org/pub/scm/linux/kernel/git/will/kvmtool.git/tree/arm/aarch64/kvm-cpu.c)) — the canonical small reference for **boot register state + FDT**: `reset_vcpu_aarch64` is the verbatim "x0=DTB, pc=entry, PSTATE=0x3C5, x1..x3=0" recipe; `arm/fdt.c` generates the minimal GIC/timer/PL011/PSCI/memory DTB. The cleanest small reference; what TABOS reproduces with `build_handoff` instead of an FDT. FDT placement datapoint: kernel at `RAM_base + text_offset` (read from the Image header since [Zyngier's 2020 patch](https://patchwork.kernel.org/project/linux-arm-kernel/patch/20200605104907.1307967-1-maz@kernel.org/) — hardcoding 0x80000 broke on Linux 5.8 when text_offset changed); FDT at **end of RAM − 2 MiB**.
- **Firecracker** ([firecracker-microvm `src/vmm/src/arch/aarch64/{layout.rs,mod.rs,fdt.rs}`](https://github.com/firecracker-microvm/firecracker)) — the **"tiny device set + virtio-mmio" minimalism** reference: DRAM at `0x8000_0000`, kernel `Image` at `SYSTEM_MEM_START+SYSTEM_MEM_SIZE` (2 MiB-aligned), **FDT placed at the END of DRAM** (`dram.last_addr() - FDT_MAX_SIZE(2 MiB) + 1`, initrd just below it), GICv3-only FDT, virtio-mmio + PSCI, sets `x0=fdt`. **KVM-backed**, so it shows the *configuration*, not the EL2 emulation — useful as the device-set minimalism target, not as the emulation template.
- **crosvm** ([`aarch64/src/lib.rs` + `fdt.rs`](https://github.com/google/crosvm/blob/main/aarch64/src/lib.rs); [memory-layout appendix](https://crosvm.dev/book/appendix/memory_layout.html)) — the third production datapoint: same protocol, RAM at `0x8000_0000`, **FDT position configurable (`--fdt-position`, default *end* of RAM when `--kernel`)**, GIC + timer + PSCI + virtio in the generated FDT, no ACPI on aarch64. Confirms the three-VMM consensus: *RAM base + Image at text_offset + FDT high in RAM + x0=FDT + PSTATE=EL1h-DAIF-masked* is THE contract; only addresses differ (QEMU `virt` uses RAM@`0x4000_0000`, kvmtool/Firecracker/crosvm RAM@`0x8000_0000`).
- **Dall & Nieh, "KVM/ARM: The Design and Implementation of the Linux ARM Hypervisor," [ASPLOS 2014](https://dl.acm.org/doi/10.1145/2541940.2541946)** — the canonical trap-and-emulate burden analysis for exactly aL2.4b's device triple: GICv2 **distributor fully emulated** (trap-and-emulate), GICv2 **CPU interface NOT emulated** (the hardware virtual interface GICV is stage-2-mapped at the guest's GICC address; the hypervisor programs GICH list registers), timers via the virtual timer + EL2 trap of the physical timer. The architecture aL2.5 already mirrors at small scale.
- **Bao** ([bao-project](https://github.com/bao-project/bao-hypervisor); [Martins et al. 2020](https://www.bao-project.org/) + [arXiv:2303.11186](https://arxiv.org/pdf/2303.11186)) — the **architectural twin of the TABOS nVHE monitor** (already the M27 anchor): a clean-slate Armv8 EL2 static-partitioning hypervisor that boots real guest OSes under stage-2 with a **deliberately thin EL2 layer** (mostly passthrough + a vGIC, no scheduler). The "how thin can EL2 be" reference, and the proof that a real guest kernel under a thin EL2 monitor is well-trodden.
- **rust-vmm: `vm-fdt` + `vmm-reference`** ([vm-fdt](https://github.com/rust-vmm/vm-fdt), [vmm-reference vcpu init](https://github.com/rust-vmm/vmm-reference)) — dependency-light, `no_std`-friendly building blocks for the boot blob + vCPU register init; closest in spirit to TABOS's verified-leaf, low-dep style.
- **QEMU `hw/arm/virt.c`** ([virt machine](https://www.qemu.org/docs/master/system/arm/virt.html)) — the de-facto **device-address/DTB contract** every guest expects (RAM @ `0x4000_0000` guaranteed; everything else discovered via DTB). The reference for which addresses a guest may hardcode vs must read from the boot blob, and the same machine the TABOS CI lane already runs.

## 3. The loader gap — what is DONE vs MISSING between v1 and a full-kernel guest (in-tree audit, 2026-06-11)

**DONE (verified in-tree, zero new work).**
- **The boot-block producer** `tb_boot::aarch64::build_handoff` ([`crates/tb-boot/src/aarch64.rs`]) serializes `TbBootInfo` + `TbMemRegion[]` + cmdline at the frozen placement (`info@0x4000_1000`, `regions@0x4000_1040`, `cmdline@0x4000_1800`, all below the image LMA `0x4008_0000` = `AARCH64_IMAGE_LMA`) and returns the six `RegWrites` (`X0=TbBootInfo*`, `X1..X3=0`, `PC=entry`, `PSTATE=0x3C5`). The host test `handoff_is_the_block_the_el2_monitor_splices_for_al2_4` (`tb-boot/src/aarch64.rs:333`) **explicitly freezes this as the aL2.4b splice contract** (`gpr[0]=TbBootInfo*`, `ELR_EL2=PC`, `SPSR_EL2=0x3C5`). Crucially, `build_handoff` is `no_std` and **already compiles into the kernel itself** (deliberate — `aarch64.rs:47-51`), so the HOST kernel can produce the boot block over the guest carve with no new plumbing; the six `RegWrites` become a frame splice instead of `KVM_SET_ONE_REG`.
- **The `_tb_start` EL1 entry EXISTS** ([`crates/tb-hal/src/arch/aarch64/boot.rs:96`], anchored by the `.note.TABOS` ELF note carrying the entry address): it consumes `x0=TbBootInfo*` at EL1h and sets `x20=0`, so in-guest `BOOTED_AT_EL2=0` and `BOOT_ENTRY_EL` stays `0xFF` — the kernel prints `boot: entry-el=0xff el2=0x0`, a **free in-guest discriminator line**, and `tb-boot: contract v0 OK` when the magic validates (a positive in-guest assertion). *An earlier draft of this survey listed `_tb_start` as MISSING — corrected: the standing "aarch64 tb-boot HANDOFF" open decision is already resolved in-tree; what is NOT resolved is the pmm consumption below.*
- The **v1 two-stage-MMU primitive** + the monitor's HVC `#8`/`#9` arms with the identity-alias sentinel readback (`FAIL_NG_READBACK` — the check the guest cannot fake), and **M27's machinery**: two-VMID `VTTBR_EL2` switching + `tlbi vmalls12e1is`, the CNTHP EL2 deadline (`0x480` slot, re-arm-before-EOI, `EOI_HARD_CAP`), monitor-counted per-VMID MMIO progress cells (`el2mmio.rs`).
- The **`BOOTED_AT_EL2` green-skip gate** on every EL2 selftest — the whole L2 block self-skips in-guest for free (§5).

**MISSING #1 — guest RAM + the FIRST non-identity stage-2 in the codebase.** No guest RAM exists; v1/M27 stage-2s identity-map **all** host RAM. Needed: a contiguous PA window carved out of the 128 MiB (e.g. the top 32 MiB, `0x4600_0000..0x4800_0000`) reserved from the host pmm at boot, and a stage-2 mapping **IPA `0x4000_0000`+off → carve PA** whose GiB1 maps ONLY the carve. `tb_encode::stage2::s2_leaf_2mib/s2_leaf_4k` already take an arbitrary `pa` — the builder generalizes; the **isolation flip** is the confinement, not the descriptor math. This *resolves* the standing "EPT MAPPING POLICY: identity vs relocated base" decision: the guest's **IPA space reproduces the canonical link address space** (the image lands at IPA `0x4008_0000` = its link LMA, so `_tb_start`'s link address IS its IPA — **no relocation**); what moves is the host PA underneath. Every stage-2 edit carries the `TLBI VMALLS12E1`+DSB obligation under the right VMID (§7b-8).

**MISSING #2 — the image producer (a logged external-dependency decision).** A second copy of the kernel must land at IPA `0x4008_0000` inside the carve. Two candidates: **(a)** run-script `-device loader,file=<objcopy flat bin>,addr=<carve_pa+0x8_0000>` — simplest, host-side, token `loader=QEMU-DEVICE-LOADER`, an honest external-dependency ledger entry; **(b)** self-copy of the kernel's own `PT_LOAD` segments with a boot-time pristine-`.data` stash (`.bss` re-derived from linker symbols) — the self-hosting flavor, harder, the M36 direction. v1 takes (a) and logs the decision.

**MISSING #3 — the pmm regions port (the gap that makes a confined guest fault).** [`crates/tb-hal/src/arch/aarch64/pmm.rs`] hard-codes the QEMU `virt` 128 MiB window and parses only the FDT *header*; unlike `x86_64/pmm.rs` it **never consumes `TbBootInfo.mem_regions`**. An unfixed guest would believe it owns 128 MiB and `frame_alloc` straight into unmapped stage-2 → spurious aborts. The x86 `BootMemMap` tb-boot branch must be ported to the aarch64 pmm BEFORE the guest can boot confined (and the host FDT path must not regress — both-arch boot gates catch it).

**MISSING #4 — the exit protocol (the semihosting kill).** The kernel ends its chain in a semihosting QEMU-exit; issued from a TCG EL1 guest that **kills the WHOLE VM, host included** (TCG semihosting is not cleanly EL2-trappable). Needed: a `TbBootInfo` flags bit (`flags` is passed 0 today) — **IN-GUEST / no-semihosting-exit**: the kernel parks in `wfi` instead. Completion evidence for the monitor must be **NON-TEXT**: the final WFI trapped under `HCR_EL2.TWI` + a **doorbell** MMIO store-count to a monitor-watched unmapped IPA (the M27 forward-progress pattern) carrying a **monitor-chosen per-boot nonce** — the M34 candidate-trust discipline starts here.

**MISSING #5 — launch vocabulary.** New HVC imms `#16` (arm/launch) and `#17` (doorbell/done); `#15` is the last used in the EL2 dispatch ([`el2/mod.rs`], HVC arms ~1067–1153).

**IRQ routing while the guest runs.** The host is parked inside the launch HVC (its B0 frame resident). v1 passes physical IRQs through: `HCR_EL2.IMO=0` → IRQs delivered at EL1 = the running guest; `CNTHCTL_EL2=0x3` already un-traps CNTP/CNTPCT — so M8/M9/M12 work with **zero** timer/GIC emulation. v2 (M34-prep) flips `IMO=1` + the monitor CNTHP deadline (the tpsched machinery) and injects the guest's CNTP PPI 30 via GICH LRs (the L2.5 single-LR window generalized to multi-LR + the maintenance IRQ).

## 4. The device-emulate paths — what a real kernel guest touches (corrected against the live guards)

The full chain drives four device families plus two absences. **Correction vs the earlier draft:** running the guest's UART as identity pass-through is NOT an acceptable green route — raw guest text on the marker channel collides with the host lane's unanchored guards and is forgeable (§5); and identity-mapping virtio at the guest is actively dangerous. The corrected per-device verdicts:

- **PL011 UART @ `0x0900_0000` — TRAPPED-AND-EMULATED, REQUIRED for the green landing.** Punch a stage-2 hole over the 2 MiB block containing `0x0900_0000`; route the resulting Data Aborts through the existing `el2mmio.rs` `device_mmio` seam keyed by IPA. Emulation is tiny: `DR` write → buffer to the host/monitor; `FR` read → TXFF=0/RXFE=1; IBRD/FBRD/LCR_H/CR accepted-and-ignored — sufficient for the TABOS guest because `serial.rs` uses single-register volatile `str`/`ldr` (**ISV=1 decodable**, the same constraint the aL2.3 stub proved). Guest bytes are then re-emitted ONLY as framed, hex-or-escape-encoded `guestlog:` lines (§5) — never raw. *Linux-guest delta (B2's driver-VM, not aL2.4b):* the `amba-pl011` driver probes the **PrimeCell periph/cell ID registers `0xFE0–0xFFC`** ([PL011 TRM DDI 0183](https://documentation-service.arm.com/static/5e8e36c2fd977155116a90b5); [QEMU `pl011.c`](https://github.com/qemu/qemu/blob/master/hw/char/pl011.c) `pl011_id_arm`) and needs RX flags + IRQ.
- **GICv2 — GICD @ `0x0800_0000` + GICC @ `0x0801_0000`.** **v1: identity pass-through of both** — honest for a SOLE guest with the host parked and `IMO=0`, token `gic=PASSTHROUGH-SOLE-GUEST`. **v2: the canonical KVM/ARM split** (Dall & Nieh ASPLOS'14; [KVM vGICv2 docs](https://docs.kernel.org/virt/kvm/devices/arm-vgic.html)): trap-and-emulate **the distributor only** (GICD_CTLR/TYPER/ISENABLER/IPRIORITYR/ITARGETSR + reset values); the CPU interface is NEVER emulated — stage-2-map the hardware **GICV** virtual CPU interface (PA `0x0804_0000`) at the guest's GICC IPA `0x0801_0000` (needs an L3 4 KiB table: GICD/GICC share one 2 MiB block) and inject via GICH list registers + `GICH_HCR.En` (the aL2.5 `gich_lr_encode` machinery generalized). GICv2 over GICv3 (no per-vCPU 64 KiB redistributor pages; matches M8's hard-coded addresses).
- **ARM generic timer (CNTP, PPI 30) — NO MMIO emulation at all.** It is a sysreg device: **v1 passthrough** via `CNTHCTL_EL2=0x3` (`EL1PCEN|EL1PCTEN`), `CNTVOFF_EL2` pinned 0 — and the grant must **survive M27's IMO window flips** (per-world, §7b-5). Two literature-anchored routes: **(a) TABOS-as-guest (aL2.4b)** — the guest drives the real CNTP, PPI 30 delivered at EL1 under `IMO=0`; zero emulation. **(b) Linux-as-guest (B2)** — Linux at EL1 defaults to the VIRTUAL timer (CNTV, PPI 27; the KVM convention — [LWN](https://lwn.net/Articles/713562/)), so a Linux guest needs `CNTVOFF_EL2` per-guest + a trapped CNTP. v2 under `IMO=1` re-injects the guest's CNTP PPI virtually; **CNTHP stays the monitor's own preemption clock** (M27b) — the two timers coexist by design, separated by EL.
- **virtio-mmio @ `0x0A00_0000` (stride `0x200`) — NEVER identity-map; emulate OPEN-BUS.** Identity pass-through would (a) put **two live drivers on one host device** (the parked host kernel already owns the rng/blk transports) and (b) hand a stage-2-confined guest a DMA device it programs with **IPA-based addresses that are wrong on the host bus** — silent corruption. v1 leaves the region UNMAPPED at stage-2 and emulates **open-bus RAZ/WI** reads, so the guest probe finds no device → M19 prints `(no device, skipped)` / M20 `(no disk, skipped)` — legitimate GREEN skips in-guest (§5). Real virtio-rng/blk device models are v2+ (and arguably belong to the M30 tb-vmm host-backend arc, not at EL2).
- **SMMUv3 @ `0x0905_0000` — open-bus.** `IDR0` reads 0 → the L2.6 selftest takes its existing green `Unavailable` skip. **Semihosting** — neutered via the `TbBootInfo` IN-GUEST flag rather than trapped (§3-MISSING#4).

The honesty boundary: "booted a real kernel to a witnessed doorbell" is **NOT** "a Firecracker/KVM-class VMM" — no SMP, no virtio rootfs, no migration, no device hotplug, TCG non-cycle-accurate. Mark accordingly (proposal §5).

## 5. The in-guest acceptance profile — the guard collision, the channel split, and marker custody

This is the subtle, load-bearing piece of aL2.4b, and the part the **current anti-hollow guards get wrong in BOTH directions**. The same `rust_main` chain boots as the guest. Three verified facts:

1. **The guest has NO EL2 of its own** (one level of nesting is the claim, not recursive EL2). Every EL2-gated selftest already has a machine-emitted green-skip path **today**: the `BOOTED_AT_EL2.load(Acquire) != 1` guard makes the whole block return `Unavailable` — `L2.0: el2 OK (no EL2, skipped)` … `M27: sched OK (no EL2, skipped)`; L2.6 skips via its `IDR0.S2P` gate; M26 has no EL2 exits to observe. The `_tb_start` path guarantees `BOOTED_AT_EL2=0` (§3), and `boot: entry-el=0xff el2=0x0` is the free discriminator. **Zero kernel changes needed for the skips.**
2. **The guest-side partition (the sovereignty plan's M0..M28 widening):**
   - **MUST be PROVEN in-guest (any `(skipped)` rejected):** M0..M18 (+M18.1/M18.2) **and the EL2-independent upper chain M21..M25 + M28** — all in-RAM/deviceless pure-compute + serial paths; M28's self-test plays its OWN simulated enrolled verifier in-kernel (host-independent), and its `kan_active=0 oracle=SIMULATED-ENROLLED-KEY` tokens must survive the guestlog framing byte-faithfully. M8/M9/M10/M12 need only the v1 GIC/CNTP pass-through. `tb-boot: contract v0 OK` is additionally REQUIRED (the positive handoff assertion).
   - **MUST legitimately SKIP green in-guest:** L2.0–L2.6, M26, M27 — and the profile **requires** the `(no EL2, skipped)`/`(no stage-2 SMMU, skipped)` *form*: a guest that prints the REAL `L2.0: el2 OK` is **rejected** (it would mean it was not actually deprivileged — the inverted-suspicion rule).
   - **CONDITIONAL:** M19 `(no device, skipped)` / M20 `(no disk, skipped)` — required in that form under v1's open-bus absence. **n/a:** the x86-only vmxroot line.
3. **The guard collision (verified in `scripts/run-aarch64.sh`):** the host guards are **UNANCHORED substring greps over the WHOLE serial stream** (`grep -qF`/`-qE`). A forwarded guest line `M20: persist OK (no disk, skipped)` trips the host lane's explicit M20-skip REJECT (`run-aarch64.sh:148` — ironically itself an anti-hollow guard); guest `(no EL2, skipped)` variants fire other reject/`::warning` paths; and **raw guest text could forge ANY host marker or witness line** — the exact non-line-anchored hole the sovereignty plan's M31 section names for model bytes. **Mere line-prefixing does NOT fix substring greps** — the rejected substring survives inside a prefixed line.

**The channel split (the fix).** All guest serial leaves ONLY via the trapped-PL011 emulate path (§4) and is re-emitted as framed, hex-or-escape-encoded **`guestlog:`** lines — a new Kani-proven `tb-encode` leaf (bounded-length frames, total encode/decode, **no marker-alphabet bytes pass through raw**). Guest bytes are thereby **regex-inert to every existing guard — injection-proof by construction** — and the encoder is the same leaf M31 needs for untrusted model bytes.

**The script structure (strip-then-assert; the host profile is NOT weakened).** `run-aarch64.sh` gains a pre-filter stage, not modified guards: (1) the RAW stream is captured; (2) **GUEST** = the decoded `guestlog:` frames, **HOST** = the raw stream with `guestlog:` lines stripped; (3) **every EXISTING guard runs byte-identical over HOST** — the diff to the guard block is zero, machine-checkable; (4) a **NEW in-guest guard set** runs over GUEST: positively require the in-guest chain with the partition above (including requiring the skip *forms*), require the discriminator + `tb-boot: contract v0 OK`, reject the same overclaim tokens; (5) **anti-hollow canaries**: zero `guestlog:` residue in HOST, every host marker found ONLY in HOST, and the `L2.4b` marker present iff the guest section ran.

**Marker custody (the M34 forward-pointer, binding).** `L2.4b: el1-kernel-guest OK` is emitted by the **HOST** from **monitor-witnessed evidence only**: doorbell MMIO count > threshold + the monitor-chosen per-boot nonce echoed through the doorbell store + the final trapped WFI — **never from guest text**. Pre-M34, guest text is *corroborating* DoD evidence (it is OUR trusted image); at the **M34 promotion boundary the same bytes are UNTRUSTED** ("a malicious candidate prints the whole chain for free; marker-grep carries zero evidential weight") and drop to zero weight — **same plumbing, no rebuild**. Building the witnessed channel now is what makes aL2.4b's harness reusable as M34's acceptance substrate.

## 6. The EL2 emulation burden — making the lower-EL view self-consistent

The "guest is at the top" illusion is mostly free, because a guest **cannot read EL2 registers from EL1** (they are UNDEFINED at EL1). The work is making the lower-EL view self-consistent ([HCR_EL2 semantics](https://arm.jonpalmisc.com/latest_sysreg/AArch64-hcr_el2)):

- **HCR_EL2**: `RW=1` (EL1 is AArch64), `VM=1` (stage-2 ON — the spatial sandbox), `IMO/FMO/AMO=1` **only** while the monitor wants to own physical IRQ/FIQ/SError (M27's proven IMO-window discipline) else 0 so the guest fields its own; **`TGE=0`** (TGE=1 would route the guest's own EL1 exceptions to EL2 and break it). Spoof `VPIDR_EL2 ← MIDR` and `VMPIDR_EL2 ← MPIDR` (TABOS already does this in `boot.rs`) so the guest reads sane CPU IDs.
- **Stage-2 + MMIO trap-and-emulate** ([Arm ARM DDI 0487](https://developer.arm.com/documentation/ddi0487/latest), stage-2): a guest access to an unmapped IPA faults to EL2 as a Data Abort (`ESR_EL2.EC=0x24`, lower-EL); the handler decodes ISS (ISV/SAS/SRT/WnR/FAR/HPFAR) for access size/direction/register/IPA, services the device, advances `ELR_EL2`, `eret` — **exactly the aL2.3 `el2_mmio_emulate` path today**; every emulated device is one more case in that switch.
- **Optional sysreg traps** (NOT needed for a cooperative demo, add per guest demand): `TVM/TRVM` (track guest VM-control-register writes), `TID1/2/3/4` (spoof CPU/cache ID registers — CCSIDR/CLIDR/CTR), `TWI/TWE` (turn guest WFI/WFE into a yield — **TWI is load-bearing for aL2.4b's final-WFI completion witness**, §3-MISSING#4), `TSC` (trap SMC-conduit PSCI). Each is one bit + one handler case.
- **Hiding EL2 from the guest — what is free vs what needs a trap** ([HCR_EL2](https://developer.arm.com/documentation/ddi0601/2020-12/AArch64-Registers/HCR-EL2--Hypervisor-Configuration-Register)): (i) **`CurrentEL` is free** — at EL1 it architecturally reads EL1. (ii) **EL2 sysregs are free** — `*_EL2` accesses from EL1 are UNDEFINED → the guest's own EL1 vector takes them, never ours. (iii) **ID registers are the one honest leak**: `ID_AA64PFR0_EL1.EL2` reads "EL2 implemented" — harmless for TABOS-as-guest (the `BOOTED_AT_EL2` gate keys off the boot EL, not the ID field); a *strict* illusion sets `HCR_EL2.TID3` and serves a spoofed ID-group-3 file (the KVM/NV approach — [LWN](https://lwn.net/Articles/919851/)). (iv) **`VPIDR_EL2/VMPIDR_EL2`** (already set) cover MIDR/MPIDR without traps.
- **The ISV=0 emulation hole (design constraint, not an option):** stage-2 MMIO emulation can only decode aborts with `ESR_EL2.ISV=1` (single general-purpose-register LDR/STR, no writeback). **LDP/STP, pre/post-indexed forms, and DC ZVA to an emulated region fault with ISV=0 — no syndrome, nothing to emulate**; KVM's answer is "kill the guest or punt to userspace" ([KVM_CAP_ARM_NISV_TO_USER](https://patchwork.kernel.org/project/qemu-devel/patch/20200323113227.3169-2-beata.michalska@linaro.org/); [kvmarm thread](https://lists.cs.columbia.edu/pipermail/kvmarm/2019-September/037115.html)). TABOS controls its own guest (`serial.rs` is single-GPR by construction; TABOS has no instruction decoder and will not grow one) — the rule is **audit every driver touching trapped regions for ldp/stp/writeback forms, fail-closed = guest dead**, and the monitor must render an ISV=0 abort on an emulated IPA as a **loud named red** (`el2: nisv-abort ipa=<hex>`), never a silent re-enter.
- **PSCI conduit** ([PSCI binding](https://www.kernel.org/doc/Documentation/devicetree/bindings/arm/psci.txt); [SMCCC](https://developer.arm.com/Architectures/Secure%20Monitor%20Calling%20Convention)): trap HVC (already routed to EL2) / SMC (`HCR_EL2.TSC`), dispatch on the SMCCC function ID. Minimal for a UP guest: `PSCI_VERSION` + `SYSTEM_OFF`/`SYSTEM_RESET`. TABOS-as-guest uses the doorbell/WFI protocol instead (§3); PSCI matters for a future foreign guest.

**Discipline cost (TABOS-specific):** all of the above is unsafe/asm and MUST live in `tb-hal` (the kernel stays `#![forbid(unsafe_code)]`); each emulated-device decision is a candidate for the M22/M27 tamper-evident fold; the honesty boundary is sharp (§5).

## 7. Effort, sequencing, and the staged landing (the M27a→M27b smoke-first lesson)

**XL overall — but front-loadable.** Stages 1+2 below are each ~one standard milestone landing (M-sized each) and together deliver the green marker M34 needs; Stage 3 is the genuinely-XL half and blocks only M34's preemptive-monitoring posture, not the marker.

- **Stage 0 — docs-only (this survey + the proposal; parallel-safe now):** freeze the `guestlog:` frame codec shape, the carve geometry, the HVC `#16`/`#17` vocabulary, the `TbBootInfo` IN-GUEST flag bit, and the strip-then-assert script contract. No code.
- **Stage 1 — "confined boot" (one PR-loop landing):** the aarch64 pmm `TbBootInfo`-regions port (x86 parity) + the pmm carve reservation + the non-identity stage-2 builder (+ extended stage-2 Kani harnesses) + the run-script `-device loader` second image + the `build_handoff` write + HVC `#16` splice/`eret` + `flags=IN-GUEST` no-exit + doorbell/final-WFI completion + the monitor-emitted marker. Guest UART may be identity pass-through TEMPORARILY, **but the lane is NOT merged green** (or merges behind a non-default profile flag) — the §5 guard collision exists until Stage 2. Measure boot wall-time here (host chain + guest chain under single-thread TCG vs the 90 s ceiling).
- **Stage 2 — "guestlog + in-guest profile" (one PR-loop landing; the FIRST green `L2.4b: el1-kernel-guest OK`):** the PL011 stage-2 hole + `el2mmio`-seam forward + the `guestlog` tb-encode leaf (pinned `EXPECTED_HARNESSES` bump) + the run-aarch64.sh split (HOST guards byte-identical over the stripped stream; the new GUEST guard set with the §5 partition) + virtio open-bus RAZ/WI + the adversarial DoD cases: a guest store to a host-RAM IPA → witnessed stage-2 fault; forged host-marker bytes from the guest → appear only hex-framed; a guest semihosting attempt absent.
- **Stage 3 — "monitor-preemptible guest" (v2, M34-prep, its own milestone):** `IMO=1` window + the CNTHP monitor deadline over the guest (tpsched reuse — also M35's revert-deadline mechanism) + GICD emulate + GICC→GICV stage-2 remap (4 KiB L3) + multi-LR injection of the guest's CNTP PPI; optionally real virtio models or the M30 tb-vmm channel as the guest's device backend.

**The boot-hang lesson (the L2 chain's hard-won discipline, multiplied).** Every L2 rung that mis-set an EL2 register, mis-routed an IRQ, or skipped a teardown surfaced as a **silent boot hang**, not a clean red. aL2.4b multiplies that surface; reuse the chain's mitigations: the no-EL2-allocation rule in abort handlers; teardown-first on the done path; the hard EOI/exit cap (M27); the independent monitor-side checks the guest cannot fake (the doorbell nonce, the identity-alias readback, the trapped WFI); and the **meta-mitigation** §7b argues for — a monitor-side **CNTHP-armed boot deadline** + storm cap so every §7b class renders as a *fast named* CI red (`L2.4b: HANG class=<storm|stall> last-trap=<EC/IPA>`), never a silent job timeout. **Multi-iteration is expected** — the smoke-first M27a→M27b shape, deliberately.

**Parallelism note:** aarch64-only; touches `tb-hal/el2` + `pmm.rs` + `run-aarch64.sh` + `tb-encode`. CONFLICTS with the in-flight M29 work only on the tb-encode harness count + the run-script edits — sequence those two files' changes after M29's stages A/B land; everything else is disjoint.

## 7b. The known hang/storm pitfall corpus — what breaks when a real kernel enables its own S1 MMU + timers under stage-2

The web survey's hardest-won material: every one of these is a **documented production bug class** in KVM/arm64 or Xen/arm, each manifests as a silent hang/storm (the L2 chain's known failure shape), and each has a named mitigation aL2.4b can adopt up front. Ranked by likelihood-of-hitting-us × debugging cost:

1. **Stale-cache boot corruption (the guest-enables-S1-MMU pitfall).** With `SCTLR_EL1.M=0` a guest's data accesses are **non-cacheable**; the loader/monitor wrote the image + boot block through cacheable mappings. Without a **clean-to-PoC of every written guest page** the guest reads stale pre-image bytes — and corruption surfaces *later*, after the guest enables its S1 MMU, as an un-debuggable wandering fault. KVM's mitigation lineage: clean+invalidate at stage-2 fault/map time, and on pre-S2FWB hardware additionally **trap VM-control registers (`HCR_EL2.TVM`) to detect `SCTLR_EL1.M` 0→1 and flush** (`stage2_flush_vm()`); FEAT_S2FWB (Armv8.4) removes the dance ([Dall S2FWB patch](https://patchwork.kernel.org/project/linux-arm-kernel/patch/20191025135144.8805-1-christoffer.dall@arm.com/); [KVM `mmu.c`](https://github.com/torvalds/linux/blob/master/arch/arm64/kvm/mmu.c); booting.rst's "clean to PoC" clause is the same fact as a loader obligation). **aL2.4b adoption:** the image/boot-block write path does `DC CVAC` per written line + `IC IALLU` before `eret` — and **NOTE: TCG is cache-coherent, so this entire class is INVISIBLE on the QEMU CI lane and lights up only on real silicon** — an honesty token, `cachemodel=TCG-COHERENT-UNTESTED`.
2. **Set/way cache ops from the guest (`HCR_EL2.TSW`).** Kernels issue `DC ISW/CSW/CISW` set/way ops, which are **un-virtualizable** (they act on the physical cache the guest doesn't own). KVM traps them (`TSW`) and over-approximates with a full clean ([set/way series](https://www.mail-archive.com/kvm@vger.kernel.org/msg99655.html)). TABOS-as-guest issues none — **declare the non-dependence instead of building the trap**, and set `TSW` anyway so a future foreign guest faults loudly instead of corrupting.
3. **Level-triggered timer-PPI screaming.** The arch-timer PPIs are **level-triggered**: if the guest's timer fires while the monitor holds the physical IRQ (`IMO=1`) and the line is neither masked, deactivated, nor the virtual interrupt EOI'd, the same IRQ re-fires at EL2 forever — a livelock storm. Documented verbatim in KVM ("a screaming HW interrupt"); the fix is **unconditionally mask/disable the hardware timer on guest exit and re-arm on entry** ([Zyngier](https://patchwork.kernel.org/project/kvm/patch/1442419087-24648-3-git-send-email-marc.zyngier@arm.com/)). M27's re-arm-before-EOI + storm-cap discipline is the in-tree twin — **extend the storm cap to the guest's CNTP PPI** (a Stage-3 obligation; v1's `IMO=0` route sidesteps it).
4. **`CNTVOFF` inconsistency / time going backwards.** booting.rst makes "CNTVOFF consistent" a boot-protocol requirement; a drifting virtual offset gives the guest a non-monotonic clock — soft-lockup detectors, RCU storms ([counter-offset series](https://www.spinics.net/lists/kvm-arm/msg48565.html)). **aL2.4b:** pin `CNTVOFF_EL2=0` and assert it in the entry path.
5. **Missing `CNTHCTL_EL2.EL1PCTEN/EL1PCEN`.** The inverse timer pitfall: a monitor that leaves the physical counter/timer trapped hangs a CNTP-using guest at its **first counter/calibration read** — and TABOS's M8 uses CNTP, so this is OUR first-boot hang candidate ([CNTHCTL_EL2](https://arm.jonpalmisc.com/latest_sysreg/AArch64-cnthctl_el2); booting.rst's EL1-entry clause). One register write, but **per-world**: the guest's grant must survive M27's monitor-owned IMO window flips.
6. **vGIC bring-up holes** (Stage 3): `GICH_HCR.En` left clear (injected vIRQ never delivered → eternal WFI), distributor-model reset values dropping the `GICD_CTLR` enable (all IRQs dead), and `EOImode` mismatches leaving the physical line active → storm. The aL2.5 single-vINTID path already encodes the En+LR discipline; the generalization risk is the **distributor reset values** ([KVM vGICv2](https://docs.kernel.org/virt/kvm/devices/arm-vgic.html); Dall & Nieh §GIC).
7. **ISV=0 aborts on emulated MMIO** (§6) — silent if mishandled; must be the named red.
8. **Stage-2 TLB staleness.** Any stage-2 edit without `TLBI VMALLS12E1` + DSB under the right VMID leaves the guest executing through stale IPA→PA translations — S1-under-S2 TLB entries are **VMID-tagged combined translations**, so the staleness multiplies once the guest's own S1 MMU is on ([pKVM stage-2 series](https://patchwork.kernel.org/project/linux-arm-kernel/patch/20221017115209.2099-20-will@kernel.org/)). The v1 identity world never edits a live map; the carve loader **must** add the invalidate to its arm sequence.
9. **Boot-blob placement attribute trap** (Linux-class guests): the DTB must be mapped cacheable in 2 MiB blocks — placing it in a 2 MiB region shared with device MMIO is a spec violation that boots-then-hangs (booting.rst §DTB). `TbBootInfo@IPA 0x4000_1000` sits in plain carve RAM below the image — conformant; keep it that way.

**The meta-mitigation** the corpus argues for: a **monitor-side CNTHP-armed boot deadline + storm cap** (the M27b machinery, the same mechanism the sovereignty plan reuses for M35's revert deadline) so every class above renders as a fast, named CI red instead of a silent job timeout.

## 8. Kani obligations for any new codec leaf

aL2.4b is mostly silicon glue (loader + splice + stage-2 arm) living in `tb-hal` **outside** the proof boundary — but every **new pure encoder leaf** carries Kani harnesses in lockstep with `EXPECTED_HARNESSES` + `kani.yml`, each with a genuine negative control, and under the house **mutation-test rule** (a harness counts only if a documented seeded mutation of the leaf makes it FAIL — a harness that survives its mutation is vacuous and rejected):

- **The `guestlog:` frame codec (MANDATORY — the one new leaf, ~3–5 harnesses):** bounded-length frames; `encode`/`decode` total + round-trip; injective; and the load-bearing safety property — **no byte of the marker/guard alphabet passes through raw** (the output is regex-inert to every host guard). *Neg:* an identity/passthrough encoder leaks a forged `M20: persist OK` substring — the harness must catch exactly that.
- **The non-identity stage-2 carve map (~2 harness extensions):** the guest-IPA→carve-PA map is **injective + range-bounded** (no two guest IPAs alias one host PA; every mapped PA falls inside the carve; no mapping reaches host RAM outside it). *Neg:* an off-by-one in the carve base aliases the guest's first page onto a host frame outside its slice.
- **The `TbBootInfo` handoff codec** — only if `build_handoff`'s layout grows (the IN-GUEST flag bit reuses the existing `flags` field; no layout change expected): round-trip totality + injectivity over `mem_regions`. *Neg:* dropping a region-length field lets two layouts alias.
- **REUSE, do NOT duplicate:** `stage2.rs`/`el2_trap.rs`/`paging` already cover the descriptor algebra, the DABT/SYS64 ISS decode, and `make_entry`/`level_index` (the v1 guest reuses `paging::make_entry` byte-for-byte). Add a harness only for genuinely new math, bounded by a documented `assume` envelope (the #49 over-quantification lesson).

The boot lane exercises only the `msr`/`eret`/`write_volatile`/image-copy glue (OUTSIDE the proof boundary); the residual silicon obligations (cache maintenance on the EL1-written tables + the image copy under `SCTLR_EL2.M=0`, plus §7b-1's TCG invisibility) are declared in [`assumptions.md`](../assumptions.md), the same shared-state caveat L2.0/L2.1 carry.

## [BEYOND] the literature
- **A full unmodified guest kernel whose acceptance is the conjunction of (a) monitor-witnessed non-text evidence (doorbell nonce + trapped final WFI + confined stage-2 faults) and (b) a host-side in-guest acceptance profile that machine-distinguishes "must-prove" from "must-skip" over an injection-proof framed log** is not in the prior art: kvmtool/Firecracker/Bao boot a guest kernel but do not run the *guest's own* cumulative verified-marker chain as a (corroborating) acceptance oracle, do not encode an EL-aware skip profile as an anti-hollow guard, and do not frame the guest console as untrusted bytes by construction. The marker-nesting + skip-profile + witnessed-custody triple is the novel honesty mechanism; the boot protocol itself is well-trodden — claim the novelty narrowly.
- **The same `build_handoff` block + `0x3C5` splice serving both the L1 boot and the L2-guest boot** (one frozen contract, host-tested, that the EL2 monitor splices instead of `KVM_SET_ONE_REG`) — the verified-leaf/frozen-contract discipline applied to the guest-boot handoff.

## [NARROWED] vs a production VMM
- **UP only** (one vCPU; no PSCI `CPU_ON`). **No virtio rootfs** (M19/M20 skip green via open-bus absence; no re-hosted block backend). **No migration, no device hotplug.** **TCG, non-cycle-accurate** (a liveness/witness verdict, not a timing measurement; `realtime=NOT-CLAIMED`). **Sole-guest GIC pass-through first** (`gic=PASSTHROUGH-SOLE-GUEST`; the trap-and-emulate multi-guest device model is Stage 3). **Confinement is claimed only to the adversarial-case depth actually landed** — reject `isolated`/`verified-guest`/`sandboxed` near the marker until the Stage-2 adversarial cases (and beyond) exist. Honesty tokens must also reject `KVM-class`, `Firecracker-replacement`, `SMP`, `cycle-accurate`.
