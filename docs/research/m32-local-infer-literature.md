## 2. Strand 1 — engine and integration shapes

- **The `-sys`-crate engine worker (CHOSEN).** Vendored llama.cpp linked behind a `-sys` crate in a keyless, env-cleared worker process — the plan-§M32 wording verbatim; whether our own vendored `-sys` or the `llama-cpp-2` ecosystem crate is pinned at implementation (the DEBT-LOCKSTEP Rule 1 grep patterns move in lockstep — proposal §17.20).
- **Runner-up (documented, not chosen): supervised `llama-server` over a UNIX socket.** Zero `-sys` churn and a native HTTP API — but upstream's own `SECURITY.md` says, verbatim: *"Do not use the RPC backend, rpc-server and llama-server functionality"* **[VERIFIED]** — and it also advises: *"Always execute untrusted models within a secure, isolated environment such as a sandbox (e.g., containers, virtual machines)."* **[VERIFIED]** Putting upstream's HTTP/template C++ surface in the loop against upstream's own advisory, plus losing pre-parse self-sandboxing (we do not control its `main()`), demotes it to runner-up. **Rejected: `llama-cli` stdin/stdout pipes** — banners/perf text are not a machine protocol, and raw model bytes interleaved with logs fight the hex-encode discipline.
- **CI artifacts: `ggml-org/models` tinyllamas.** `tinyllamas/stories260K.gguf` = **1.19 MB** and `tinyllamas/stories15M-q8_0.gguf` = **26.7 MB**, exactly as listed on the Hugging Face tree **[VERIFIED]**. The 260K model is what llama.cpp's own server test harness uses — `--hf-repo ggml-org/models --hf-file tinyllamas/stories260K.gguf` appears verbatim in `tools/server/tests/README.md` **[VERIFIED]** — so the lane's smoke artifact rides upstream's own CI precedent. The 1.19 MB artifact is committed in-repo (network-independent smoke); the 26.7 MB Q8_0 exercises the quantized-kernel path the lane actually claims. Named operator/nightly tier: SmolLM2-135M-Instruct Q4_K_M (~105 MB) / Qwen2.5-0.5B-Instruct GGUF (~0.4 GB) **[UNVERIFIED — workflow-sourced sizes]** — Q4_K_M matters because B3 pins it as the load-bearing custody format.
- **Quantization context.** GGUF spec (ggml-org docs) and the k-quants landing (llama.cpp PR #1684) define the format family; GPTQ (arXiv:2210.17323) and AWQ (arXiv:2306.00978) are the canonical PTQ baselines. *"Which Quantization Should I Use? A Unified Evaluation of llama.cpp Quantization on Llama-3.1-8B-Instruct"* (arXiv:2601.14277) **[VERIFIED title]** covers 3–8-bit K-quant and legacy formats on one modern model — the coverage argument for standardizing on Q8_0/Q4_K_M rather than an exotic quant that would strand the B3 cutover.
- **Process split precedent.** The daemon/worker key-custody split mirrors the landed house pattern: the safe-Rust key-holding peer (`tools/xport-harness` — serve loop on landed main at `src/main.rs:239-418`; double-gated activation per `src/live.rs:12-16`) versus a confined untrusted-input parser. The new dep-clean `tools/xport-core` crate exists because Cargo records package-level deps in every consumer's lockfile — the stage-C `ureq`/`serde_json` on the harness package would trip the daemon's zero-network grep on day one (proposal §17.3).

## 3. Strand 2 — the determinism envelope

- **The upstream-stated bit-exact envelope (THE load-bearing quotes).** llama.cpp PR #16016 (an opt-in CUDA deterministic mode proposed by an outside contributor) drew the maintainer position the lane contract adopts. JohannesGaessler, verbatim: *"I don't want to maintain guarantees for bit-for-bit identical results as the batch size is varied."* and *"you can already get bit-for-bit identical results by disabling prompt caching and using only a single slot for the HTTP server."* **[VERIFIED — both sentences re-fetched from the PR thread]** The envelope = single slot, no prompt cache, pinned build — adopted verbatim; batch-size invariance is REFUSED upstream and therefore never claimed (the lane pins the batch instead).
- **Multi-slot nondeterminism is measured, not theoretical.** Issue #7052, title verbatim: *"Non-deterministic output of the llama.cpp server when using multiple slots"* **[VERIFIED]** — temp 0, fixed seed, 8 slots → 5–8 unique completions; single slot → consistent. The single-sequence pin is load-bearing.
- **Threading escapes `--threads 1` via BLAS/OpenMP.** Issue #22956, title verbatim: *"Nondeterministic CPU inference with `--image` despite `--threads 1` due to OpenMP in OpenBLAS"* **[VERIFIED]** — OpenBLAS compiled with OpenMP silently overrides the thread setting; `OMP_NUM_THREADS=1` confirmed as the working mitigation. Hence the proposal's compile-time `GGML_OPENMP=OFF` + `GGML_BLAS=OFF` with the runtime env pin kept as belt-and-braces (§17.20).
- **Greedy/temp-0 does NOT imply reproducibility.** *"Understanding and Mitigating Numerical Sources of Nondeterminism in LLM Inference"* (arXiv:2506.09501) **[VERIFIED title]**: *"changing system configuration, such as evaluation batch size, GPU count, and GPU version, can introduce significant differences in the generated responses"* — up to 9% accuracy variation and ~9,000-token response-length drift under BF16 greedy decoding. *"Non-Determinism of 'Deterministic' LLM Settings"* (arXiv:2408.04667) **[VERIFIED title]**: up to 15% accuracy variation across naturally occurring runs under settings expected to be deterministic. The mechanical cause is floating-point non-associativity under varying reduction order: *"Impacts of floating-point non-associativity on reproducibility for HPC and deep learning applications"* (arXiv:2408.05148) **[VERIFIED title]**. Flash-Attention numeric deviation (arXiv:2405.02803) **[UNVERIFIED — workflow-sourced]**.
- **The mitigation literature is server-class and NOT adopted.** Thinking Machines, "Defeating Nondeterminism in LLM Inference" (2025) — batch-invariant kernels — and SGLang's deterministic mode solve concurrent-serving determinism; *"LLM-42: Enabling Determinism in LLM Inference with Verified Speculation"* (arXiv:2601.17768) **[VERIFIED title]** enforces determinism via a verify-rollback loop over a nondeterministic fast path. M32's lane is stop-and-wait single-`req_id` by construction — it pins the envelope rather than engineering batch invariance, and the SGLang-class concurrent claim is in the proposal's NEVER-claimed list.
- **Quantization is not determinism.** Only integer-ONLY pipelines are exact by construction (I-BERT, arXiv:2101.01321; I-LLM, arXiv:2405.17849 — canonical); llama.cpp k-quants are int8-dot + **FP32 scale/accumulate + FP norms/softmax**, so "determinism-from-quantization" would be false and is never claimed.
- **The fleet is heterogeneous — build from source, pin the ISA.** actions/runner-images#3389, verbatim: *"Some of the computers running Github actions have avx512 enables. others do not."* **[VERIFIED]** Release binaries do runtime CPU-feature dispatch → different SIMD paths → different reduction orders → different bits. Hence: built from the pinned source, `GGML_NATIVE=OFF`, `GGML_CPU_ALL_VARIANTS=OFF`, ISA pinned x86-64-v3, dispatch static, and the `llama_print_system_info()` tripwire. GPU nondeterminism (issue #10197) and the general determinism discussion (issue #3014) **[UNVERIFIED — workflow-sourced]** are moot on the GPU-less runners but named for completeness.
- **[BEYOND].** The sources defend serving stacks and benchmarks; none of them aim a pinned-envelope determinism claim at a CI marker-grep adjudicator with **repo-pinned golden digests adjudicated workflow-side, a frozen second-prompt negative control measured-distinct at pin time, and a per-run nonce-freshness leg** (proposal §4) — the M36 reproducibility-ladder pattern applied early, hardened against the stub-daemon forgery the adversarial review named (§17.1).

## 4. Strand 3 — the GGUF/CVE threat surface

**The GGUF parser has a sustained, multi-year exploit record — it is the plan-named untrusted-input C attack surface, with receipts:**

- **The RPC backend is its own disaster class:** CVE-2024-42479 — write-what-where in `rpc_server::set_tensor`, CVSS **9.8**, fixed in b3561 **[VERIFIED]** — plus the SECURITY.md "do not use" sentence (§2 above). Compiled OUT of the vendored tree, asserted absent by grep (proposal §5.3).
- **Model-metadata-as-template is a demonstrated RCE class:** CVE-2024-34359 — Jinja2 SSTI in **llama-cpp-python** (the bindings, not llama.cpp's own template path — the review's §17.16 accuracy fix), CVSS **9.7**: the GGUF chat template from model metadata rendered in an unsandboxed `jinja2.Environment` **[VERIFIED]**. M32 never renders GGUF-embedded templates regardless of layer — raw-completion only.
- **Artifact custody:** CVE-2024-37032 ("Probllama" — Ollama path traversal via model digest, Wiz) is the anti-pattern M32 inverts: SHA256 pins in-repo, the safe-Rust daemon re-verifies before the worker fd handoff, the C parser never touches an unverified byte **[UNVERIFIED — workflow-sourced detail; the CVE itself is well-documented]**. The hub-poisoning base rate is measured (MalHug, arXiv:2409.09368 **[UNVERIFIED — workflow-sourced]**) — the pin guarantees identity, not safety: `weights=UNTRUSTED-INPUT-NAMED` survives the pin. sigstore model-transparency / OMS signing is the named successor.
- **Sandboxing precedent and the cautionary tale.** llamafile added pledge()/SECCOMP sandboxing to llama.cpp — and the 0.10.0 rebuild **dropped it** (the sandboxing features are absent from the rebuilt core, acknowledged as not yet carried over) **[VERIFIED]**: a silently-vanishing safety property remembered in prose, not machine-checked — exactly the failure class the proposal's §5.2 anti-hollow sandbox probes and the DEBT-LOCKSTEP guard exist to make structurally loud. M32's worker self-sandbox: `seccompiler` (the Firecracker lineage) + `rust-landlock` (Ubuntu 24.04's ~6.8 kernel = Landlock ABI v4), filters installed BEFORE GGUF parsing — inverting llamafile's post-load order. NOT built on user namespaces: `apparmor_restrict_unprivileged_userns=1` is the 24.04 default and Qualys published bypasses of the mechanism in 2025 **[UNVERIFIED — workflow-sourced detail]**. Docker's default seccomp profile has historically not allowlisted the `landlock_*` syscalls **[UNVERIFIED — workflow-sourced]** — one reason the daemon+worker run ON the runner while only the QEMU boot stays in the aarch64 container (proposal §10/§17.7).
- **Cache custody:** GitHub `actions/cache` evicts at 7-days-unused / 10 GB repo pressure (documented platform policy) — routine eviction must never be a standing red: the hash-pinned setup-step re-fetch (proposal §17.5) confines network to workflow setup, and hard-FAIL is reserved for hash MISMATCH.
- **Injection at the serial/log boundary:** Spotlighting (arXiv:2403.14720 — encoding the untrusted text drops attack success >50% → <2%) and the indirect-prompt-injection threat frame (arXiv:2302.12173) are the M31 survey's verified basis, inherited verbatim — M32 moves the same hex-encode discipline one ring inward, onto the daemon's own grepped stdout.

## 5. Strand 4 — the debt-token discipline

- **The problem class.** TEMPORARY dependencies silently age into unpatched permanent ones; prose carriers (TODO comments, a DEBT.md) rot ungrepped; and the V-01..V-06 no-CVE set (§4) proves CVE-feed scanners cannot police this dependency at all. The plan's principle 1 therefore demands the debt be **machine-checked, never prose-remembered** — a sovereignty ledger emitted inside the M33/M36 attestations.
- **House precedent (the token regime is landed practice, not invention).** Machine-emitted honesty tokens with guard-enforced strip-then-reject: M21's `active=0` dormant witness, M24's `gate-not-met` honest refusal, M29's `sec=ASSUMED-FROM-LITERATURE` prove/assume boundary, M31's closed `backend=` set; cross-file lockstep greps (the bench.yml BRAND/NOTE_TYPE_HEX guard; the `EXPECTED_HARNESSES_TOTAL` lockstep across `kani-shards.sh`/`verify-encode.sh`/`kani.yml`); and the M30 §5.6 two-stream cross-process adjudication. M32's DEBT-LOCKSTEP composes these into a **bidirectional token⇔dependency coupling**: dep-present∧token-absent FAILS, dep-absent∧token-present FAILS, closure-claimed∧dep-present FAILS, and the pre-closure CC-poison probe (worker build MUST FAIL under `CC=/bin/false` against a pristine target dir) proves the C is genuinely in the build graph — a debt token over secretly C-free code is also a lie.
- **External precedent class.** `cargo-deny` (license/source/advisory bans as CI gates), `cargo-audit` (RUSTSEC feed), `cargo-geiger` (unsafe counting) are the Rust ecosystem's machine-checked dependency police — the proof that grep/manifest-level dependency policy CAN be a hard CI gate. **[BEYOND]:** none of the surveyed tools couples a *witness vocabulary* (the lane's emitted tokens) to a *build-graph state* in BOTH directions with a fail-closed poison probe; DEBT-LOCKSTEP is that extension, and the closed token grammar (`engine=` state-valued never brand-valued, `debt=` retired-not-renamed, `memsafety=` paired 1:1) exists so the B3 closure flip is a one-landing, zero-grammar-churn event — pre-registered down to the reserved peer byte (`peer::INFER_DAEMON_PURE = 0x04`).
- **The closure authority.** `docs/research/b3-pure-rust-engine.md` §5 defines the five gates (CC-poison cchain, byte-exact tokenizer parity vs the `llama.cpp tokenize` oracle, the ppl-delta quality floor, the tps-ratio perf floor, the one-landing flip) and §6 the four coordinated grep guards; the proposal's §12 freezes its constants. The ladder TEMPORARY → CLOSURE-DEFINED → CLOSED instantiates the `LANGUAGE-AND-STANDARDS.md` §6 row shape (`:96-97`), with the M33 attestation codec as the machine-readable carrier.
