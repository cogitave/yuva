# M33 literature survey — provenance lineage (the signature-scheme choice, the signed-root/transparency-log shape, the SLSA/in-toto/DSSE attestation codec, and the stateful-signature custody hazards)

Companion to [`docs/proposals/M33-prov-lineage.md`](../proposals/M33-prov-lineage.md) — **co-landed in the same PR** (the proposal's §13 step 0; the M29/M30/M31/M32 research-first precedent is binding, and the proposal's body cites this file as a load-bearing dependency in its `Depends on:` line). This survey discharges the research half of **#91 / plan C1** (`docs/plans/sovereignty-plan.md:64-65` §M33) in the four strands the proposal synthesizes: (a) the signature-scheme choice (why a hash-based scheme, why LMS, why curve crypto is CBMC-infeasible), (b) the signed-root / transparency-log shape (CT signed tree heads, the split-view fork, witness cosigning), (c) the SLSA/in-toto/DSSE-PAE attestation codec, and (d) the key-custody + stateful-signature hazards (leaf-index reuse, forward security). **Every load-bearing citation the proposal relies on was independently verified against the primary source at landing (2026-06-22)** — marked **[VERIFIED]**; anything carried from the research workflow without an independent re-fetch is tagged **[UNVERIFIED — workflow-sourced]**, never silently included. Where M33 reaches beyond what a source supports it is flagged **[BEYOND]**.

The single thread tying the four strands: M33 buys **public-verify / private-sign asymmetry** over M29's symmetric MAC — and the literature is mined to choose the one scheme family that is simultaneously (1) Kani/CBMC-tractable in-house, (2) standard enough to claim conformance, and (3) honest about the residual custody/state hazards the asymmetry does not close. The proposal's own §16 adversarial round corrected an RFC 8554 parameter error (W4=p=67, not p=34) and a per-compression cost law; both corrections are confirmed against the primary sources below (§1, §6).

## 1. Strand A — the signature-scheme choice

**The forced family: hash-based, because curve crypto is not bounded-model-checkable in-house.**

- **Ed25519 / P-256 are CBMC-infeasible and the literature on *how* curve crypto gets verified is the dispositive evidence.** Every production-grade formal verification of curve25519-class field arithmetic uses **deductive / synthesis** tooling, never bounded model checking:
  - **fiat-crypto** (`mit-plv/fiat-crypto`) is "a framework for **correct-by-construction** finite field arithmetic in Coq" — the field-arithmetic *backend is extracted from Coq proofs of arithmetic correctness*, and `curve25519-dalek-fiat` exposes exactly this Coq-synthesized backend. **[VERIFIED]**
  - **HACL\* / EverCrypt** (Project Everest) is "a formally verified cryptographic library written in **F\***," with Curve25519, Ed25519, and SHA-2 each "verified for memory safety, mitigations against timing side-channels, and functional correctness" in F\* and compiled to C; deployed in the Linux kernel, Firefox NSS, mbedTLS, WireGuard. **[VERIFIED]**
  - The TweetNaCl X25519 correctness result (Schwabe et al.) is a **Coq** proof; the Jasmin "Last Mile" line is **EasyCrypt**. **[VERIFIED — title/tooling]**
  - **Synthesis (`[BEYOND]` the survey, asserted by the proposal):** none of these uses CBMC/Kani-style bounded model checking, because `2^255-19` field multiplication is hundreds of 256-bit modular multiplications, each a bignum carry chain — the documented CBMC SAT-explosion case (`m29-crypto-mac-literature.md §5`, the in-repo precedent the proposal cites). An in-house curve25519 *proven by Kani* is therefore infeasible; the proposal's §1.1 "DISQUALIFIED" verdict is sound against the unanimous tooling record above. P-256 is strictly worse and both forfeit post-quantum resistance.

- **LMS (RFC 8554) is the chosen kernel-verify primitive.** RFC 8554 = **"Leighton-Micali Hash-Based Signatures"** **[VERIFIED]**; it defines **LM-OTS** (the one-time signature) and **LMS** (an N-time scheme over a Merkle tree of LM-OTS keys). Verification "complet[es] Winternitz chains through iterative hash applications, then reconstruct[s] the Merkle root using the authentication path values" — the exact verify shape the proposal's `lms_verify` walks (§2, §6 of the proposal). The hash is pinned to SHA-256: "Here SHA256 denotes the SHA-256 hash function defined in NIST standard [FIPS180]." **[VERIFIED]** LMS is a pure hash chain — implementable in-house from the RFC exactly as M29 implemented BLAKE2s from RFC 7693, adding no field math.

- **The LM-OTS parameter table (the budget-arithmetic foundation; the §16 parameter-error correction confirmed).** RFC 8554 Table 1 / the LMOTS_SHA256_N32_W{1,2,4,8} sets give the number of n-byte chain elements `p`:

  | Winternitz `w` | `p` (chain elements) | per-chain max walk `2^w − 1` |
  |---|---|---|
  | w=1 | **265** | 1 |
  | w=2 | **133** | 3 |
  | w=4 | **67** | 15 |
  | w=8 | **34** | 255 |

  **[VERIFIED — re-fetched from RFC 8554].** This **confirms the proposal's §16 must-fix #6**: the W4 chain count is **p=67**, not p=34 (p=34 is the *W8* value); the draft's earlier "W4 p=34" was the error, now corrected throughout the proposal. The worst-case W4 verify ≈ `67 × 15` chain-steps + the H10 Merkle path ≈ **~1062 SHA-256 compressions** is the proposal's headline "no real-parameter HBS verify fits the lane" figure, and it is directly derivable from this table (each LM-OTS / Merkle step is one sub-64-byte block = exactly one SHA-256 compression). W8 (p=34, walk 255) ≈ 8670 compressions — why the smaller-signature W8 is rejected by name.

- **XMSS (RFC 8391) — rejected as the kernel-verify primitive (more codec surface).** RFC 8391 = **"XMSS: eXtended Merkle Signature Scheme"** **[VERIFIED]**, specifying XMSS, XMSS^MT, and WOTS+. It carries more verifier surface than LMS (an L-tree construction and per-node prefix-keyed PRF addressing), so the proposal rejects it for kernel-verify while still leaning on its statefulness warning (strand D). XMSS verify ≈ 130–150 compressions **[UNVERIFIED — workflow-sourced estimate; the order-of-magnitude is consistent with WOTS+ `len≈67` × small chains but the exact count is not re-derived here]**.

- **SLH-DSA / SPHINCS+ (FIPS 205) — rejected as kernel-verify primitive (cost), named only as the stateless fallback.** FIPS 205 = **"Stateless Hash-Based Digital Signature Standard"**, standardizing **SLH-DSA** "based on SPHINCS+." **[VERIFIED]** Its statelessness would eliminate the strand-D leaf-index hazard entirely — but verify is thousands of hash calls (≈ 100× khash), the largest verifier LOC, so its statelessness "does not pay for kernel-verify cost" (proposal §1). A genuine, named tradeoff, not a free lunch.

- **The "no published CBMC/Kani HBS verifier exists" novelty risk — confirmed, and it is the dominant practical risk.** The state of the art in *verified* HBS is **eprint 2026/134, "Completing the Chain: Verified Implementations of Hash-Based Signatures and Their Security"** (Barbosa/Hülsing lineage; SandboxAQ; **to appear TCHES 2026**) — "the first formally verified implementation of a hash-based signature scheme … linked to a machine-checked proof of security," a **Jasmin** XMSS/XMSS^MT refining RFC 8391, with the security reduction in **EasyCrypt** (the CRYPTO'23 Barbosa-Dupressoir-Grégoire-Hülsing-Meijers-Strub proof). **[VERIFIED]** No bounded-model-checked HBS verifier is in the literature — so the M33 LMS leaf is *novel packaging with no reference harness to copy*, and implementation-correctness bugs (not security-proof gaps) are the dominant risk, caught by the official-vector KAT + Miri net (proposal §9). The original SPHINCS+ *tight* proof being **flawed-then-fixed** (Kudinov-Kiktenko-Fedorov found the WOTS-proof flaw in 2020; Hülsing-Kudinov 2022 "Recovering the Tight Security Proof of SPHINCS+," later EasyCrypt-formalized — **[VERIFIED]**) is the proposal's cited warning against any in-house *security* claim: the witness stays at `sec=ASSUMED-FROM-LITERATURE`.

## 2. Strand B — the signed-root / transparency-log shape

- **A signature proves authenticity-of-extension, not chain honesty or fork-freedom — the split-view attack is the named gap.** RFC 6962 = **"Certificate Transparency"** **[VERIFIED]**, defining an append-only Merkle log and **Signed Tree Heads (STH)** (§3.5). It explicitly models the **split-view / fork attack**: a log "violating its append-only property by presenting two different, conflicting views of the Merkle Tree at different times and/or to different parties." **[VERIFIED]** Detection is *external*: "Violation of the append-only property is detected by global gossiping, i.e., everyone auditing logs comparing their versions of the latest Signed Tree Heads. As soon as two conflicting Signed Tree Heads for the same log are detected, this is cryptographic proof of that log's misbehavior." **[VERIFIED]** This is the literal basis for the proposal's `splitview=UNDETECTED-NO-WITNESS-QUORUM` token: a single signer signing two divergent heads is undetectable without an external witness/gossip quorum (§1.4, §8, §14).
- **The signed prov head IS a single-signer STH analogue.** The M33 signed root is exactly a Signed Tree Head over the prov head — but with one signer and no gossip cohort, so it inherits CT's *signing* guarantee (authenticity of what was signed) without CT's *consistency* guarantee (no fork). The proposal is honest that closing the fork gap requires a witness quorum, named as a successor (RFC 6962 monitors/auditors §5.3–5.4; the Rekor transparency-log lineage), **not** an M33 claim. **[BEYOND]** — the proposal applies the CT split-view frame to a *kernel-resident* signed memory-provenance head, a target CT was not written for; the inheritance is by analogy and is tokened, not asserted as conformance.
- **Checkpoint / witness cosigning (the named successor).** The modern "checkpoint" + cosigning-witness pattern (the Go-checksum-DB / sigsum / C2SP checkpoint lineage; cosigned tree heads that a quorum of independent witnesses attest) is the concrete mechanism a future rung would adopt to gain split-view detection. **[UNVERIFIED — workflow-sourced; named as a direction, no load-bearing DoD token depends on it]**

## 3. Strand C — the SLSA / in-toto / DSSE-PAE attestation codec

- **DSSE PAE is the exact signed-bytes encoding, verified byte-for-byte.** The DSSE (Dead Simple Signing Envelope) protocol defines **PAE** (Pre-Authentication Encoding) as, verbatim: `PAE(type, body) = "DSSEv1" + SP + LEN(type) + SP + type + SP + LEN(body) + SP + body`, where SP is a single ASCII space and LEN is the ASCII decimal byte length "without leading zeros," and "the signature is computed over the PAE bytes rather than the payload directly." **[VERIFIED]** This length-prefixed structure "ensures injectivity — each component's boundaries are unambiguously defined by its preceding length value." This is precisely the `attest=DSSE-PAE-SHAPED` leaf the proposal models (§5): an ASCII length-prefixed injective encoding, trivially `no_std`/zero-dep, reusing the house length-prefixed-injective canon (the `opframe_rx` MAC-bytes split). The proposal correctly scopes the proof to **byte-level PAE injectivity + roundtrip + fail-closed decode** and explicitly does NOT claim JSON-canonicalization security.
- **The in-toto Statement v1 is the carried predicate shape.** The in-toto attestation framework v1 Statement layer defines `_type` (`https://in-toto.io/Statement/v1`), `subject` (an array of ResourceDescriptors, each with `name` + `digest`), `predicateType` (a URI), and `predicate`. "Subject artifacts are matched purely by digest." **[VERIFIED]** The M33 `attest.rs` codec encodes a *fixed-width subset* of this Statement (`subject_digest`, `builder_id`, `buildType`, a materials digest list, the toolchain hash, the sovereignty ledger) — "an SLSA-CONCEPT-SHAPED fixed-width subset, **not** a wire-compatible in-toto JSON producer," said so on the token (proposal §5).
- **SLSA v1.0 build levels L1/L2/L3 — the claim ceiling.** SLSA v1.0 defines three build levels **[VERIFIED]**: **L1** = "Package has provenance showing how it was built" (exists, unsigned); **L2** = builds on a "hosted build platform" where "the provenance is tied to that infrastructure through a digital signature" (prevents post-build tampering); **L3** = "Hardened Builds" with "strong controls to prevent runs from influencing one another" and signing "secret material … inaccessible to user-defined build steps." This validates the proposal's claim ladder exactly: M33 reaches **L1 → toward L2** (an artifact exists / is signed), and **L3 is M36's job** (`repro=BIT-IDENTICAL-2BUILD`, non-falsifiable provenance + isolated builds). The SLSA-uses-in-toto link (provenance carried as an in-toto attestation) is the framework's standard binding. **[VERIFIED — SLSA defines L1/L2/L3 as stated; the in-toto-format binding is documented in SLSA's attestation-model section but was not re-fetched verbatim here — [UNVERIFIED — workflow-sourced] on the exact "uses in-toto" wording, though it is the universally-deployed shape].**
- **The self-reported-digest honesty boundary (the `selfmeasure=UNATTESTED-LOADER` token).** No source closes the gap that, with no measured boot / RTM / external measurer (the QEMU image is loaded directly), the `subject_digest` is computed by the very image being attested — "a malicious image computes `khash::uhash` of whatever it chooses to report." The proposal names this with a dedicated token so M34 cannot infer independent measurement; the survey corroborates only that SLSA L1/L2 do **not** require a hardware root (that is L3+ / measured-boot territory), so the proposal's `measure=SELF-NO-HW-ROOT` is consistent with an L1→L2 claim. **[BEYOND]** — the honesty token is a project invention with no external source to verify; it is correct *by construction*, not by citation.
- **The "Verification Theatre" overclaim-policing reference.** eprint **2026/192, "Verification Theatre: False Assurance in Formally Verified Cryptographic Libraries"** (published 2026-03-06) — a case study of libcrux/hpke-rs finding thirteen vulnerabilities that "escaped formal verification" because "the verification boundary … is not communicated clearly," including *false serialization proofs* and *axiomatized-but-unsound* specs. **[VERIFIED]** This is the proposal's cited justification (§5, §14) for refusing a JSON-canonicalization-security harness and for keeping the proven tier (totality / roundtrip / fail-closed) strictly disjoint from the assumed tier — over-scoping a green harness that asserts less than it appears to is exactly the failure this paper documents.

## 4. Strand D — key-custody + stateful-signature hazards

- **Leaf-index reuse is catastrophic and unrecoverable — the load-bearing custody warning.** NIST SP 800-208 = **"Recommendation for Stateful Hash-Based Signature Schemes"** **[VERIFIED]**, approving LMS, HSS, XMSS, XMSS_MT, with parameter sets using "either SHA-256 or SHAKE256 with 192- or 256-bit outputs." **[VERIFIED]** On the reuse hazard, verbatim from the recommendation's framing: "These algorithms require persistent, crash-consistent counters; **key reuse caused by counter reset is catastrophic and unrecoverable**." **[VERIFIED]** RFC 8391 §1.1 states it independently: "If a secret key state is used twice, no cryptographic security guarantees remain. In consequence, it becomes feasible to forge a signature on a new message." **[VERIFIED]** This is the entire basis for the proposal's VERIFY/SIGN SPLIT (§2): the secret key and the never-decrement counter are the two things that must never enter the stateless, verified kernel.
- **Hardware non-exportability is an SP 800-208 profile requirement — confirms the rung-3 ladder.** SP 800-208 "requires that key and signature generation be performed in hardware cryptographic modules that do not allow secret keying material to be exported." **[VERIFIED]** This is the literal definition of the proposal's §7 rung-3 (`key=HW-NONEXPORTABLE`, the SP 800-208 HSM profile) — the survey confirms the rung is a real standardized custody tier, not an invented one.
- **The state-reuse vectors SP 800-208 enumerates — counter reset, restore-from-backup, failover — map to the proposal's stage-A hazards.** The recommendation's "persistent, crash-consistent counter" requirement is exactly why the proposal (§3) closes the crash-ordering window at stage A by *single-signature-per-invocation-by-construction* and defers the durable never-decrement TOP-level counter to the M35 monitor. The proposal is honest that the stage-A compiled-in simulated key **deliberately reuses leaf 0 every run** — acceptable only because that key has no security value — and emits `state=SIMULATED-REUSE-OK-NO-SECURITY` (NOT `NEVER-REUSE`) on that lane. The BSI laser-fault-injection result (state reuse is *practically* inducible, not merely theoretical) is **[UNVERIFIED — workflow-sourced]** and not load-bearing for any DoD token.
- **Forward security (FssAgg / Bellare-Yee) — the M28 precedent and its limit.** The M28 `key_evolve` PRF-chain (`key_{i+1} = khash(key_i, "YUVA-KEY-EVOLVE-V1")`, `opframe_rx.rs:57,380`, tokened `keyevolve=PRF-DOMSEP`) upgrades to the **Bellare-Yee (CT-RSA 2003) forward-security reduction shape**, conditional on the PRF assumption + domain separation + old-key erasure. **[VERIFIED — in-repo; the Bellare-Yee citation is carried from the landed M28/M29 honesty note, not independently re-fetched here → [UNVERIFIED — workflow-sourced] on the external paper, [VERIFIED — in-repo] on the token].** The load-bearing custody point the proposal makes (§3, §7): the **LMS asymmetric signing key's monotone leaf-index is a SEPARATE state machine** from M28's symmetric PRF-chain — an implementer must NEVER advance the LMS leaf-index via `key_evolve`; the two custody domains are disjoint by construction. Forward security gives no post-compromise healing (a stolen signing seed yields all future signatures) — the M28 `oldkey-zeroized` caveat, re-stated.
