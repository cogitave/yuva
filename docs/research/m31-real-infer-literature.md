# M31 literature survey — the real Anthropic adapter (byte framing, injection-proof serial, the secret-gated live lane)

Companion to [`docs/proposals/M31-real-infer.md`](../proposals/M31-real-infer.md). This survey discharges the research half of **#89 / A2 real-Anthropic-adapter** (`docs/plans/sovereignty-plan.md` §M31): the byte-prompt/byte-response framing onto the landed M30 channel (the BULK, per the plan amendment), the Anthropic Messages API shape the host bridge must speak, the injection-neutralization discipline for untrusted model bytes crossing guest serial, the GitHub-Actions secret-lane patterns for the optional live lane, and the challenge-nonce liveness design that makes `M31: real-infer OK` non-replayable. Where M31 goes beyond a source it is flagged **[BEYOND]**.

---

## 1. The decision frame (from the project's own constraints)

Five hard fixtures the literature must be fitted to, not the other way round:

- **The M30 channel is landed and poll-only.** `tb-encode::inferwire` (magic `0x5444`, ver 1, canonical LE, reserved-zero fail-closed, truncated-khash tag, `FrameAccum` magic-scan resync) rides a single-port virtio-console byte stream under `POLL_CAP` with **no IRQ path** (#71 watch). The kind set is CLOSED and fail-closed (`kind_known`); `ERR=3` is reserved precisely "so a future adapter never overloads ECHO_*". `INFER_PAYLOAD_CAP=1024` transitively pins `ECHO_MSG_CAP`, `INFER_ACCUM_CAP`, the kernel's fixed stack buffers, and every Kani bound (#49) — raising it is expensive and still a hard bound.
- **The guards are NOT line-anchored.** The lane filters are `grep -E '(^|[^[:alnum:]])(M30:|xport:)'` (`scripts/run-x86_64.sh:476,488,527`, mirrored aarch64) — token-boundary, not `^`-anchored. Untrusted model text containing `M31: real-infer OK backend=ANTHROPIC-LIVE` WOULD satisfy a naive M31 grep. This single fact drives the whole §4 encoding discipline.
- **Key-custody precedent is M30 §4** (HOST-custodied per-run K, never in the guest image/cmdline, the §5.7 leak negative) — but the API key is STRICTER: M30's K was deliberately revealed on-channel (ECHO design); the API key is never revealed anywhere, to anyone, on any channel.
- **`InferBackend` is a u64 scalar toy** (`crates/tb-hal/src/infer.rs:40-45,69-74`) with the variable-length path explicitly deferred ("rides a future M14 channel / M15 block", `infer.rs:22-27`). M31 is the named heir of that deferral.
- **The plan amendments are binding** (sovereignty-plan §M31 + task #89): dual markers, hex-encoded serial, the TLS-outside-kernel decision WRITTEN into `LANGUAGE-AND-STANDARDS.md`, `key=CAPREF-HOST-CUSTODIED host=RESIDUAL-TCB`, in-GUEST-only zero-ambient-authority, NOT fit for unattended runs.

## 2. The Anthropic Messages API — the shape the host bridge must speak

- **Endpoint.** `POST https://api.anthropic.com/v1/messages` — one endpoint for everything; JSON body; required headers `x-api-key` (the secret — host-bridge-only, never crosses the virtio channel), `anthropic-version: 2023-06-01`, `content-type: application/json`.
- **Request minimum.** `{model, max_tokens, messages:[{role, content}]}`; optional `system`/`stream`/`tools`/`stop_sequences`. For M31 the bridge needs only `model` + `max_tokens` + ONE user-string message — everything else is host-side policy, not wire format.
- **Non-streaming response.** `{id:"msg_…", type:"message", role:"assistant", content:[{type:"text", text}…], model, stop_reason, stop_details?, usage:{input_tokens, output_tokens, …}}`. The `stop_reason` taxonomy: `end_turn | max_tokens | stop_sequence | tool_use | pause_turn | refusal` — **`refusal` may carry EMPTY content**, so the bridge must branch on `stop_reason` BEFORE reading `content[0]`. The HTTP response carries a server-assigned **request-id header (`req_…`)** — the per-request, third-party-correlatable evidence to fold into the M25 transcript. The closed `StopReason` enum in `infer.rs` already covers this union (the M16 design anticipated it).
- **Streaming is a HOST-BRIDGE-ONLY concern.** `stream:true` returns SSE (`message_start → content_block_start → content_block_delta → content_block_stop → message_delta → message_stop`; mid-stream errors as `error` events); SDKs require it for large `max_tokens` to dodge HTTP timeouts. Recommendation: the bridge accumulates the full text and delivers ONE reassembled chunk sequence over the guest channel — **SSE never crosses the virtio boundary**, so the guest codec stays a fixed frame grammar.
- **Token limits → byte sizing.** Limits are token counts, not bytes (e.g. claude-haiku-4-5: 200K context / 64K output; non-streaming practical output ceiling ~16K tokens before SDK timeouts). UTF-8 averages ~3–4 bytes/token, worst case more — the bridge must clamp `max_tokens` so worst-case response BYTES fit the channel's body cap (or chunk), and clamp prompt bytes against the cap on the request leg. `usage.{input,output}_tokens` give exact post-hoc counts for the transcript.
- **Error taxonomy.** HTTP status + `{type:"error", error:{type, message}, request_id}`: 400 invalid_request, 401 authentication, 403 permission, 404 not_found, **413 request_too_large** (the host-side mirror of the channel's own cap-reject), **429 rate_limit (retry-after header)**, 500 api_error, **529 overloaded**. Retryable = 429/500/529 (+408/409 connection-class); other 4xx are permanent. Design point: the bridge maps this to a **small closed enum in an `ERR` frame** (code + retryable flag) — raw error JSON is still untrusted text and never passes to the guest or serial unencoded.

## 3. Framing the byte bodies — what the field uses

**The landed baseline is already the right family.** inferwire's out-of-band length + magic-scan resync is **octet-counting framing** — RFC 6587's preferred syslog/TCP mode, chosen there precisely because non-transparent (trailer/delimiter) framing lets unescaped in-band terminators split/forge messages. Content-transparent by construction: untrusted payload bytes can never terminate or open a frame.

| Option | Mechanism | Precedent | Verdict |
|---|---|---|---|
| **A. One large frame** | raise `INFER_PAYLOAD_CAP`, whole body per frame | — | rejected as the only mode: the cap repin is transitive (accum, stack buffers, every Kani bound) and becomes a silent product constraint |
| **B. Chunked sequence** | `INFER_REQ`/`INFER_RESP` chunks: seq + MORE flag + whole-body digest, correlation-id binds the sequence, chunk metadata bound INSIDE the MAC | **CoAP block-wise RFC 7959** (SZX blocks 16..1024, lockstep request/response per block, Size1/Size2 totals — the constrained-device gold standard, a natural fit for a polled channel); gRPC's 5-byte length-prefix; SSH Binary Packet Protocol (length+MAC per packet) | **RECOMMENDED** |
| C. Byte stuffing | COBS: delimiter framing made content-transparent, ≤1/254 overhead, resync at every delimiter | Cheshire & Baker SIGCOMM '97 | redundant — length-prefix + magic-scan + MAC already gives bounded resync; a second codec discipline for nothing. Delimiter framing WITHOUT stuffing rejected outright (the RFC 6587 trailer problem) |

- **Max-size negotiation precedents:** MQTT 5.0 CONNECT `Maximum Packet Size` (receiver declares, sender must not exceed); TLS `record_size_limit` RFC 8449 (both peers declare); CoAP `Size1`/`Size2` + SZX. **M31's minimal analog: compile-time shared consts** — `tb-encode` builds for the host triple, so BOTH ends compile the SAME leaf: compile-time agreement is the cheapest honest negotiation, no handshake to get wrong. Reject-on-overflow, fail-closed, **never silently truncate** (the API's own 413 is the host-side mirror).
- **Backpressure on a polled channel:** stop-and-wait, ONE outstanding request, correlation-id rejects stale/mismatched responses — this IS the backpressure (CoAP block-wise is exactly lockstep). **The new hazard is real-API latency (seconds) vs the poll budget:** never let "API is slow" and "peer is dead" (hard `FAIL xport-timeout` per M30) be the same observation — either size the budget for network RTT or add a bridge-emitted PENDING/heartbeat frame so the guest's poll loop sees verified progress. If concurrency ever matters: virtio-vsock's credit scheme (`credit = peer_buf_alloc − (tx_cnt − peer_fwd_cnt)`, piggybacked per header) and SSH channel windows are the precedents — **do not claim flow control the design doesn't have**.

## 4. Injection neutralization — untrusted model bytes vs a trusting text channel

**Threat model.** Model output is UNTRUSTED BYTES that will transit guest serial, host stdout, CI logs, and grep guards that are NOT line-anchored (§1). Two forgery surfaces: (1) **marker forgery** — response text containing `M31: real-infer OK backend=ANTHROPIC-LIVE` or any honesty token would satisfy the greps; (2) **terminal/CI-renderer attacks** — ANSI escapes in model output rendered by terminals viewing logs (hide text, spoof output, clickable exfil links).

- **CWE-117** (Improper Output Neutralization for Logs): attacker-controlled CR/LF/control bytes forge log entries and poison downstream parsers; the canonical mitigation is **ENCODE-BEFORE-WRITE**, not blocklist filtering.
- **SMTP smuggling** (SEC Consult / Longin, 2023): parser differentials over the in-band end-of-data marker (`<LF>.<LF>` vs `<CR><LF>.<CR><LF>`) let attackers forge entire spoofed messages past SPF/DMARC — the definitive modern case that **untrusted bytes sharing the marker channel's alphabet eventually get parsed as markers**. Same lesson as RFC 6587 deprecating trailer framing: out-of-band length or a marker-free alphabet, never escaping discipline alone.
- **LLM-specific:** Terminal DiLLMa (Embrace The Red, 2024) and Trail of Bits' MCP ANSI-deception work — LLM output carrying raw/escaped ANSI hijacks the terminals rendering it: exactly the "untrusted model bytes hit a trusting text channel" class. **Microsoft Spotlighting** (Hines et al., arXiv 2403.14720): of {delimiting, datamarking, encoding}, ENCODING the untrusted text (base64/ROT13) is the strongest mode — attack success >50% with plain delimiters → <2% encoded. Independent confirmation that **transform-to-an-inert-alphabet beats delimiter discipline**. M31 applies the same mechanism at the SERIAL boundary instead of the prompt boundary. **[BEYOND]** — the sources defend the prompt/terminal; none of them aim the encoding at a CI marker-grep adjudicator.
- **Chosen mechanism (binding, task #89):** hex-encode ALL model-derived bytes before serial. Lowercase hex `[0-9a-f]` is **regex-inert by construction**: it cannot contain `:`, space, newline, uppercase `M`, `=`, or ESC `0x1b` — so it can neither forge an `M31:`-prefixed marker/token nor carry an ANSI sequence; cost is 2× length. Emit as `resp-hex=` lines WITH an out-of-band `resp-len=` field (truncation-evident), chunked across fixed-cap lines each carrying `seq=`, plus a digest commitment (`resp-digest=`) so the transcript folds a fixed-size commitment rather than trusting the dump.
- **Defense-in-depth (none sufficient alone):** (a) token-boundary/line-anchor the M31 greps anyway — cheap but NOT sufficient (any raw path leaking one newline re-opens the hole, hence encode-first); (b) markers printed ONLY by kernel code paths after verification — decoded model text is never echoed raw on any lane; (c) the strip-then-reject overclaim guards stay: the mock lane actively rejects `real|ANTHROPIC-LIVE` near its marker so a forged live claim cannot enter the cumulative chain.

## 5. The secret lane — GitHub Actions patterns for an optional live job

- **Platform behavior.** Actions deliberately withholds secrets from fork-triggered `pull_request` runs (anti-exfiltration); `pull_request_target` restores them but is the classic **pwn-request footgun** (untrusted code with secret access — GitHub Security Lab) — **never** use it for the M31 live lane. Same-repo branches and `push`/`schedule`/`workflow_dispatch` events DO see secrets.
- **Missing-secret gating.** Canonical pattern: map the secret to env in a probe step, test emptiness, export `has_secret` as a step output and gate the live steps on it; the terser job-level `if: ${{ secrets.ANTHROPIC_API_KEY != '' }}` also works on current runners. The live lane is OPTIONAL and never a required check; the CI-required cumulative chain carries only `M31: infer-e2e OK backend=MOCK-DETERMINISTIC`.
- **Loud skip.** On a missing secret the lane must be VISIBLY skipped, never silently green: emit `::warning::` (annotations surface in the PR UI) plus a distinct named log line the run scripts can assert on — mirroring M30's "present-then-silent is hard FAIL, legitimately-absent is loud skip" discipline. **A skip that prints nothing is indistinguishable from a hollow pass.**
- **Key hygiene (`key=CAPREF-HOST-CUSTODIED`).** The key lives only in the host-bridge process env, sourced from the Actions secret; the guest image/cmdline never carry it (the M30 custody precedent — and stricter: M30's K was revealed on-channel by design; the API key is NEVER revealed anywhere). GitHub masks registered secret values in logs **but masking fails on transformed values** — never print the key, its prefix, or its length; fold only the request-id + response digest into the transcript, never the raw authenticated request.
- **Cost and flake containment.** The live lane pins ONE call: cheapest adequate model, small `max_tokens`, no retries beyond SDK defaults; 429 (read retry-after) and 529 map to a distinct FAIL-RETRYABLE outcome so a rate-limited run is never reported as a liveness failure — **and never as a pass**. Consistent with the plan: M31 is NOT fit for unattended overnight runs (network + secrets).

## 6. Liveness — proving the round-trip is fresh, not canned

**Goal:** prove the secret-gated run made a FRESH round-trip to the live Anthropic API — not a canned fixture, not a replayed prior transcript, not the mock backend wearing the live token.

- **The pattern: challenge-nonce-in-prompt → required-transform-in-response.** Precedents: TPM remote attestation (verifier-chosen random nonce signed inside the quote = freshness + anti-replay; RFC 9683/9684 make a fresh high-entropy nonce a MUST), ACME RFC 8555 (server-issued single-use `Replay-Nonce` in every signed request; challenge token + key authorization as proof-of-control), HTTP Digest RFC 7616 nonce/cnonce, WebAuthn challenges. The shared invariant: **the verifier picks the random value AFTER the session starts, and the prover's answer must be a function of it that cannot be precomputed.**
- **LLM adaptation.** The host verifier generates per-run nonce N from host RNG at run start; the prompt embeds `hex(N)` plus an instruction to apply a **trivial-but-non-identity deterministic transform** (reverse the hex string — kept trivial so model compliance is near-certain). Acceptance: (a) `transform(N)` present in the response, (b) the transform is non-identity so a dumb echo/reflector of the prompt cannot pass, (c) HTTP 200 with a request-id header present. This defeats: recorded fixtures (stale N), the mock backend (doesn't know N or the rule — and its lane token differs), and reflectors. **[BEYOND]** — nonce-freshness protocols exist everywhere; aiming one at "is this text really from a live LLM endpoint" with a model-compliant transform is the adaptation.
- **Non-replayability — four binds:** (1) fresh per-run nonce entropy (TPM/ACME discipline — single-use, verifier-held); (2) the server-assigned request-id (`req_…`) + a digest of the full response body folded into the M25 transcript — third-party-correlatable, per-request evidence; (3) **M30's two-leg adjudication reused verbatim**: the kernel-witnessed serial line and the bridge's independently-printed line must string-match on nonce/transform/digest — a guest-side forgery fails the host leg, a host-side fixture fails the kernel leg; (4) the live marker emitted only after both legs agree, and the mock lane's strip-then-reject guard keeps it out of the cumulative chain.
- **Honest residuals (for `assumptions.md`).** The proof establishes "**a process holding the key obtained a fresh nonce-dependent transform from an endpoint behaving as the live API over TLS**". It does NOT prove absence of a host-level MITM beyond the rustls/OS trust roots (`host=RESIDUAL-TCB`), does not bound model behavior, and the transform check is probabilistic — a compliant-but-wrong model answer must be reported as TRANSFORM-MISS (distinct from LIVENESS-FAIL: request-id present + 200 but transform absent), never silently retried into a pass. And because the bridge is trusted ground, the liveness claim is bridge-honesty-conditional — host exclusivity awaits the M33 signature primitive.

## 7. The verified-leaf fit — what is proven vs assumed

M31 inherits the M22..M30 discipline: the transport is plumbing, the codec is the leaf, khash is reused never re-implemented. The extension lives INSIDE `inferwire` (same magic, same ver): new closed kinds, an in-payload chunk sub-header (the reserved `flags` byte stays zero — canon forces it, decode rejects nonzero, so chunk metadata cannot ride the header without a ver bump), a sibling `verify_infer_resp` with a NEW domain separator (the `ECHO_DOMAIN`/`KDF_DOMAIN` precedent), chunk index bound INSIDE the MAC (the M28/Terrapin lesson), and a Kani-proven reassembly accumulator with a compile-time total-size bound.

**PROVEN (Kani, concrete inputs / short-symbolic, #49):** kind-extension round-trip + fail-closed unknown-kind reject; sub-header round-trip + reserved-bits fail-closed; assembler in-order/total-bound/never-overflow + gap/duplicate/mismatch rejects; response-binding iff-theorem incl. chunk seq; tamper rejection (flip-then-flip-back); echo-vs-infer domain separation.

**ASSUMED / NOT-PROVEN (tokened, never prose-claimed):** khash PRF strength (`sec=ASSUMED-FROM-LITERATURE`); bridge honesty + TLS stack + trust roots (`host=RESIDUAL-TCB`); model behavior (unbounded, untrusted bytes); inference SEMANTICS on the mock lane (`backend=MOCK-DETERMINISTIC` is a deterministic transform, not a model).

**[BEYOND] the literature.** The composition — **a Kani-proven no_std chunked-byte codec over a stock virtio channel, carrying live-LLM traffic whose serial representation is encoding-neutralized against the project's own CI marker-grep adjudicator, with a nonce-transform liveness proof folded as request-id + digest into an in-kernel provenance transcript, and the prove/assume boundary machine-emitted as strip-then-reject witness tokens** — appears in no single source. Spotlighting encodes at the prompt boundary; CWE-117 names the log boundary; CoAP chunks constrained transports; TPM/ACME prove freshness; none of them defend a marker-based anti-hollow CI regime against the model's own output.
